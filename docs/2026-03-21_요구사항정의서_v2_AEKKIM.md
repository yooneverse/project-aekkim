# 애낌 (AEKKIM) 요구사항정의서

> 문서 버전: v2.6
> 기준 일자: 2026-03-29
> 기준 범위: 현재 구현되어 있는 Android 앱 화면 및 FE 라우트 기준 현행화

## 1. 문서 목적

이 문서는 애낌 Android 앱의 현재 구현 범위를 기준으로 요구사항을 다시 정리한 현행 문서다.
기획 초안이나 확장 아이디어가 아니라, 실제 앱에서 동작 중인 기능과 화면 흐름을 우선 기준으로 삼는다.

## 2. 제품 개요

애낌은 사용자의 구독 현황, 결제 내역 기반 분석, 프로모션 추천, 체크인 알림을 통해
불필요한 구독 비용을 줄이도록 돕는 Android 앱이다.

현재 MVP 기준 주요 목표는 다음과 같다.

1. 사용자가 소셜 로그인 후 권한과 약관 동의를 완료하고 앱을 시작할 수 있어야 한다.
2. 결제 내역 기반으로 구독 후보를 확인하고 실제 구독 목록을 관리할 수 있어야 한다.
3. 대시보드, 인사이트, 추천, 알림, 마이페이지 흐름이 끊기지 않고 연결되어야 한다.
4. 체크인과 해지 가이드를 통해 저사용 구독 관리 행동으로 이어져야 한다.

## 3. 현재 기준 범위

### 3.1 포함 범위

- Android 앱
- Google, Kakao 로그인
- 약관 동의 및 권한 플로우
- 구독 분석 로딩 및 구독 확인
- 대시보드 기반 구독 관리
- 인사이트 화면
- 프로모션 추천 목록 및 상세
- 알림 목록 및 읽음/삭제
- 마이페이지, 알림 설정, 로그아웃, 회원탈퇴

### 3.2 제외 또는 부분 지원 범위

- Naver 실제 로그인: 미구현, 안내 토스트만 제공
- 외부 웹 기반 약관 상세 페이지: 미사용, 앱 내부 화면으로 제공
- 별도 문의 페이지/고객센터 연결 화면: 현행 UI에서 노출하지 않음
- 고도화된 인사이트 시각화: 현재는 요약 카드, TOP 3, 목록 중심

## 4. 사용자 및 진입 조건

### 4.1 주요 사용자

- 개인 Android 사용자
- 본인 구독 상태를 직접 점검하고 관리하려는 사용자

### 4.2 사용 전제

- Android 디바이스 사용
- Google 또는 Kakao 계정 사용 가능
- 알림 수신 여부는 선택 가능
- Usage Stats 권한은 핵심 분석 기능 사용을 위해 필요

## 5. 기능 요구사항

### FR-01 로그인 및 세션 시작

- 사용자는 온보딩 화면에서 Google 또는 Kakao로 로그인할 수 있어야 한다.
- Naver 로그인 버튼은 노출되지만 현재는 "곧 지원 예정" 토스트만 제공한다.
- 로그인 성공 시 액세스 토큰과 리프레시 토큰을 저장해야 한다.
- 로그인 성공 후 FCM 토큰이 있으면 서버에 동기화해야 한다.
- 신규 사용자이면서 약관/권한 플로우를 완료하지 않은 경우 약관 동의 화면으로 이동해야 한다.
- 그 외 사용자는 대시보드로 바로 진입해야 한다.

상태: 구현 완료, Naver만 부분 지원

### FR-02 약관 동의 및 권한 플로우

- 사용자는 서비스 이용약관, 개인정보 처리방침에 필수 동의해야 한다.
- 혜택 및 프로모션 정보 수신 동의는 선택 항목이어야 한다.
- 약관 상세는 외부 링크가 아니라 앱 내부 상세 화면으로 확인할 수 있어야 한다.
- 약관 동의 후 알림 권한 안내 화면을 거쳐야 한다.
- 알림 권한은 거부 또는 건너뛰기가 가능해야 한다.
- Usage Stats 권한은 분석 시작을 위해 요구되어야 하며, 허용 전까지 재안내가 가능해야 한다.

상태: 구현 완료

### FR-03 결제 내역 분석 로딩 및 구독 확인

- 로그인 및 권한 플로우 완료 후 사용자는 분석 로딩 화면을 거쳐야 한다.
- 분석이 끝나면 구독 후보 확인 화면으로 이동해야 한다.
- 사용자는 감지된 구독 후보를 확인하고 다음 단계로 진행할 수 있어야 한다.

상태: 구현 완료

### FR-04 대시보드 진입 및 구독 목록 제공

- 인증된 사용자의 기본 진입 화면은 대시보드여야 한다.
- 대시보드는 월 구독 총액, 잠재 절감 금액, 사용 리마인드, 검토 필요 항목 등을 보여줘야 한다.
- 현재 등록된 구독 목록과 빈 상태, 에러 상태를 구분해서 표시해야 한다.
- 알림함 진입 버튼과 읽지 않은 알림 점 표시를 제공해야 한다.

상태: 구현 완료

### FR-05 수동 구독 추가, 수동 매핑 및 편집

- 사용자는 대시보드에서 직접 구독을 추가할 수 있어야 한다.
- 수동 추가는 바텀시트 기반 다단계 흐름이어야 한다.
- 사용자는 서비스 또는 번들을 검색하고 선택할 수 있어야 한다.
- 선택한 서비스의 요금제와 결제일 정보를 설정할 수 있어야 한다.
- 사용자는 별도 수동 매핑 화면에서 검토가 필요한 결제 내역을 서비스에 연결할 수 있어야 한다.
- 수동 매핑 화면에서는 서비스 선택과 제거 액션을 수행할 수 있어야 한다.
- 기존 구독도 수정 및 삭제할 수 있어야 한다.

상태: 구현 완료

### FR-06 구독 상세 조회

- 사용자는 대시보드 목록에서 구독 상세를 볼 수 있어야 한다.
- 상세 화면은 서비스명, 요금제, 결제 예정 정보, 최근 사용량 요약, 추천 메시지, 해지 가이드 진입을 제공해야 한다.
- 상세에서 수정 또는 삭제 액션으로 이어질 수 있어야 한다.

상태: 구현 완료

### FR-07 인사이트 제공

- 사용자는 인사이트 탭에서 최근 사용량 기반 요약 정보를 볼 수 있어야 한다.
- 인사이트 화면은 전체 사용 요약 카드, TOP 3, 관련 인사이트 카드, 구독별 사용 목록을 제공해야 한다.
- 현재 인사이트는 리스트형/카드형 요약 구조를 사용한다.

상태: 구현 완료

### FR-08 프로모션 추천 제공

- 사용자는 추천 탭에서 카테고리별 추천 프로모션을 볼 수 있어야 한다.
- 추천 유형은 번들, 프로모션, 카드 혜택으로 구분되어야 한다.
- 각 카테고리에서 기본 3건 중심으로 노출하고 더보기 흐름을 제공해야 한다.
- 상세 화면에서 할인 정보, 적용 대상, 이동 링크를 볼 수 있어야 한다.

상태: 구현 완료

### FR-09 체크인 및 후속 행동 유도

- 사용자는 결제일이 다가오는 시점에 체크인 화면으로 이동할 수 있어야 한다.
- 체크인은 이번 달 사용 정도를 응답하는 구조여야 한다.
- 저사용 패턴일 경우 후속 화면에서 절감 금액과 행동 유도 메시지를 제공해야 한다.
- 사용자는 해지 가이드 화면으로 이동하거나 유지 의사를 선택할 수 있어야 한다.

상태: 구현 완료

### FR-10 해지 가이드 제공

- 해지 가이드 화면은 해지 방법 링크, 고객센터 전화번호, 이메일 등 제공 가능한 정보를 보여줘야 한다.
- 사용자는 해지 안내 링크로 이동하거나 전화 연결을 시도할 수 있어야 한다.
- 추후 처리 의사를 선택하고 이전 흐름으로 돌아갈 수 있어야 한다.

상태: 구현 완료

### FR-11 알림함

- 사용자는 별도 알림 화면에서 알림 목록을 볼 수 있어야 한다.
- 알림 유형은 체크인, 구독 검토, 프로모션을 지원해야 한다.
- 알림 선택 시 관련 화면으로 이동해야 한다.
- 사용자는 개별 알림 삭제, 전체 삭제, 읽음 처리를 할 수 있어야 한다.

상태: 구현 완료

### FR-12 마이페이지 및 설정

- 사용자는 마이페이지에서 시스템 알림 권한 상태를 확인할 수 있어야 한다.
- 사용자는 체크인 알림, 프로모션 알림, 마케팅 수신 동의를 조정할 수 있어야 한다.
- 사용자는 앱 버전, 약관, 개인정보 처리방침, 마케팅 동의 문서를 확인할 수 있어야 한다.
- 사용자는 로그아웃을 수행할 수 있어야 한다.

상태: 구현 완료

### FR-13 회원탈퇴

- 사용자는 별도의 회원탈퇴 화면에서 단계적으로 탈퇴를 진행할 수 있어야 한다.
- 탈퇴 전 주의사항 안내와 최종 확인 체크가 필요해야 한다.
- 탈퇴 성공 시 토큰과 온보딩 상태를 정리하고 온보딩 화면으로 복귀해야 한다.
- Kakao 로그인 사용자는 필요 시 연동 해제 요청을 수행해야 한다.

상태: 구현 완료

## 6. 비기능 요구사항

### NFR-01 상태 표현 일관성

- 주요 화면은 Loading, Empty, Error, Success 상태를 명확히 구분해야 한다.
- 대시보드와 목록형 화면은 빈 상태와 실패 상태를 혼동하지 않아야 한다.

### NFR-02 권한 처리 명확성

- 알림 권한과 Usage Stats 권한은 서로 다른 목적을 가진다는 점이 화면에서 명확해야 한다.
- 선택 권한과 필수 권한의 차이를 문구와 액션으로 구분해야 한다.

### NFR-03 보안 및 세션 처리

- 인증 토큰은 안전한 저장소 기반으로 관리되어야 한다.
- 인증 실패 또는 세션 만료 시 재인증 흐름으로 복귀할 수 있어야 한다.

### NFR-04 탐색 안정성

- 인증 여부에 따라 시작 목적지가 달라져야 한다.
- 알림 및 딥링크 진입 시 목적 화면으로 안전하게 이동해야 한다.

### NFR-05 문서 정합성

- 루트 `docs` 문서는 현재 구현 기준을 유지해야 한다.
- FE 세부 운영 기준은 `FE/docs/maintenance` 문서를 기준으로 삼아야 한다.

## 8. 현행 기준 API 범위

- 인증: `POST /api/v1/auth/login/google`, `POST /api/v1/auth/login/kakao`, `POST /api/v1/auth/refresh`, `POST /api/v1/auth/logout`
- 사용자: `GET /api/v1/users/me`, `PATCH /api/v1/users/me/notification-settings`, `PATCH /api/v1/users/me/optional-consent`, `DELETE /api/v1/users/me`
- 구독: `GET /api/v1/subscriptions`, `GET /api/v1/subscriptions/{id}`, `POST /api/v1/subscriptions`, `PATCH /api/v1/subscriptions/{id}`, `DELETE /api/v1/subscriptions/{id}`
- 서비스/번들: `GET /api/v1/services`, `GET /api/v1/services/{serviceId}/plans`, `GET /api/v1/bundles`
- 추천: `GET /api/v1/promotions/recommendations`, `GET /api/v1/promotions/{promotionId}`
- 체크인: `GET /api/v1/checkins`, `POST /api/v1/checkins`
- 알림: `POST /api/v1/notifications/tokens`, `GET /api/v1/notifications`, `PATCH /api/v1/notifications/{id}/read`, `DELETE /api/v1/notifications/{id}`, `DELETE /api/v1/notifications`
