## 👹공통 규칙

- 모든 이슈는 가능한 한 **Epic에 연결**한다.
- 이슈는 스토리 형식으로 등록한다.
- 모든 이슈는 기본 필드 입력:
    - **Assignee / Sprint / Priority / Estimate(Story Point 또는 시간) / Label(선택)**
- 제목(Summary)은 아래 형식으로 통일한다.
- 구조는 에픽→스토리→작업으로.
   
- 먼저 스토리 수준에서 스토리 포인트를 결정하고, 그 안의 작업을 미리 설정한 스토리 포인트를 나누어서 설정해주도록 합니다!! (시간 단위로 작업을 나누는 연습을 해볼 수 있는 현업 방법!!)

- **스토리 포인트 설정하기!**
    - 1 2 3 5 8 로 점수 설정하기
    - 1 : 1시간 이내로 완료 가능한 작업
    - 2 : 오전 또는 오후내로 완료 가능한 작업
    - 3 : 하루이내로 완료 가능한 작업
    - 5 : 최대 3일내로 완료 가능한 작업
    - 8 : 일주일내로 완료 가능한 작업

### 1-1. 제목(Summary) 컨벤션 (통일)

**`[파트] (도메인) 한 줄 목적/행동`**

- 파트 Prefix: `[FE]`
- 도메인 예: `(Auth) (Record) (Galaxy) (Search) (Artist)`
- 예시 : `[FE] (Record) Auto Logging 업로드 영역 Drag&Drop 구현`

## 🚌**작업유형**

### **에픽**

- 큰 단위의 업무(여러 User Story, Task 등을 묶은 단위)
- 매주 스프린트를 들어가기 전에 생성할 Epic에 있어서 이야기한다. **(필요한 Epic 조사)**
- 논의한 Epic을 기본으로 해당 Epic에 담당자를 지정하여 생성한다.
- **생성 Convention**
    - Epic을 생성할 때 파트, 기능을 적어준다.
    - ex) 결제 기능 구현/챗봇 기능 구현
    

### **스토리**

- 해당 Epic의 하위 단위 작업으로 직접적인 개발과 기능 구현을 기본으로 한다.
    - **ex ) 최종 고객에게 가치를 제공하는 기능**
    - 작성 방법 : “I as WHO want to do WHAT, so that WHY”
- Tip) User story의 크기는 sprint내에 완료 가능한 단위로 분할 필요
    - 예) 사용자 관리 개념
    - **생성 Convention**
    - **ex ) 로그인 기능**
    - 로그인 기능 개발(Epic이 [BE] User일 경우)
    - UI/UX 구현(Epic이 [FE] 목업 개발)
    - 합의 알고리즘 구현(Epic이 [BC] 합의 알고리즘)
    - [기획] Planning/[기획]설계/[기획] 문서 작업
    

### **작업**

- 해당 스토리가 필요하기 위한 작업으로 일반적으로 기술적, 관리적 업무를 지칭한다.
    - 하지만 초기 스프린트에서 비개발적인 작업이 많고 해당 에픽의 내용에 있어서는 작업을 문서 작업과 같은 작업을 작업 이슈로 관리한다.
- ex) User Story의 기술적, 관리적, 서류 작업
    - 설계, 서버 설치, 클라우드 도입 등
- **생성 Convention**
    - **[docker 학습] document 작업 진행**
    - **작업에 직관적으로 알 수 있도록 작성.**

### **하위작업**

- **Story, Task를 더 작은 단위로 나눈 업무**
    - 즉, 모든 Sub-Task가 끝나야 해당 업무 종료
    - Sub-task를 사용하는 것은 Story를 이루기 위한 단계별 작성으로 선택상으로 지정한다.
    - 예) 사용자 관리(UI) 개발, 사용자 관리(Service) 개발
    - **생성 Convention**
        - **ex) 자동 로그인 기능 구현**
        - 작업에 직관적으로 알 수 있도록 작성.
        - 작업을 진행하면서 해당 이슈를 분할하여 작성
