# 구독분석 후보추출 및 판단 파이프라인 정리

> 작성일: 2026-03-20
> 대상: FE / BE / AI
> 목적: 결제내역 기반 구독 분석에서 후보추출, 매핑, 온디바이스 AI, 사용자 확인의 책임과 순서를 정리한다.

## 1. 결론

권장 파이프라인은 아래와 같다.

1. FE가 `GET /api/v1/payment-history`로 결제내역을 조회한다.
2. FE가 기기 로컬에서 UsageStats를 수집한다.
3. FE가 서비스/요금제 카탈로그를 우리 BE API로 조회해 임시 메모리에 적재한다.
4. FE가 결제내역 전체에서 구독 가능성이 있는 항목만 후보추출한다.
5. FE가 카탈로그/규칙 기반 1차 필터를 수행한다.
6. 1차 필터로 확정되지 않은 후보만 `batch-lookup`으로 누적 매핑 DB를 조회한다.
7. 그래도 해결되지 않은 후보만 온디바이스 AI에 넘겨 종합 판단한다.
8. 여전히 애매한 항목만 사용자 판단 대상으로 남긴다.
9. 사용자가 확정한 결과는 `batch-confirm`으로 저장하고, merchant 매핑 히트카운트를 누적한다.

핵심은 다음과 같다.

- 후보추출은 AI가 아니라 규칙 기반 전처리로 수행한다.
- UsageStats는 서버가 아니라 FE 기기 로컬에서만 수집하고 활용한다.
- 온디바이스 AI는 전건 판정기가 아니라, 규칙/매핑으로 해결되지 않는 애매한 케이스 해석기 역할을 맡는다.

## 2. 왜 이 구조가 필요한가

### 2-1. 후보추출이 먼저 필요한 이유

전체 결제내역을 그대로 AI나 매핑에 태우면 아래 문제가 생긴다.

- OTT와 무관한 결제가 너무 많다.
- 명백한 비구독 항목까지 모두 분석 비용이 들어간다.
- AI 입력이 커지고, 판단 근거가 흐려진다.
- 매핑 조회도 불필요하게 많은 merchant를 대상으로 수행하게 된다.

따라서 먼저 결제내역 전체에서 "구독일 가능성이 있는 항목"만 후보로 추려야 한다.

### 2-2. 매핑만으로는 부족한 이유

`batch-lookup`은 `merchantRaw` 기반 누적 매핑 조회 API다.

- 사용량 데이터는 보지 않는다.
- 결제 주기 해석을 하지 않는다.
- 요금제 추론을 하지 않는다.
- 처음 보는 merchant에는 약하다.

즉 `batch-lookup`은 기억 DB이지, 최종 판단 엔진이 아니다.

### 2-3. 온디바이스 AI가 여전히 필요한 이유

후보추출과 매핑이 있어도 아래 케이스는 여전히 애매하다.

- `APPLE.COM/BILL`, `GOOGLE*...` 같이 merchant가 generic한 경우
- 금액은 비슷하지만 어떤 서비스/요금제인지 애매한 경우
- UsageStats와 결제 패턴을 함께 봐야 하는 경우
- 새 merchant alias라 누적 매핑 DB에 아직 없는 경우
- 저이용 판단, 체크인 생략, 해지 검토처럼 사용량 해석이 필요한 경우

즉 온디바이스 AI의 의미는 "모든 결제를 다 보는 것"이 아니라 "UsageStats가 필요한 어려운 케이스를 로컬에서 해석하는 것"이다.

## 3. 책임 분리

### 3-1. BE 책임

- `GET /api/v1/payment-history` 제공
- 결제내역 저장/조회
- 서비스/요금제 메타데이터 제공
- merchant 매핑 조회 및 확정 저장
- 분석 결과 저장

### 3-2. FE 책임

- UsageStats 권한 확인 및 기기 로컬 사용량 수집
- 서비스/요금제 카탈로그 조회 후 임시 메모리 적재
- 결제 원본 정규화
- 후보추출
- 카탈로그/규칙 기반 1차 필터
- `batch-lookup` 호출
- unresolved 후보만 온디바이스 AI에 전달
- 사용자 확인 화면 구성

### 3-3. 온디바이스 AI 책임

- FE가 정규화한 후보를 입력으로 받는다.
- UsageStats, 결제 패턴, 서비스/요금제 힌트를 종합해서 판단한다.
- 구조화된 결과만 FE에 반환한다.
- 확실한 구독 / 확실히 구독 아님 / 확인 필요를 구분한다.

## 4. 상세 파이프라인

### Step 1. 결제내역 조회 API bootstrap

- FE가 `GET /api/v1/payment-history`로 결제내역을 조회한다.
- BE는 저장된 결제내역을 분석 세션에서 사용할 수 있는 목록으로 제공한다.

필요 계약:

- `GET /api/v1/payment-history`
- `ApiResponse<PaymentHistoryListResponse>` 응답 계약

현재 상태:

- FE payment bootstrap 단계가 실제 결제내역 조회 API를 사용하도록 연결되어 있다.

### Step 2. 서비스/요금제 카탈로그 preload

FE는 분석 시작 시 아래 정보를 임시 메모리에 적재한다.

- OTT 서비스 목록
- 서비스별 요금제 목록

현재 사용 가능한 API:

- `GET /api/v1/services?category=OTT`
- `GET /api/v1/services/{serviceId}/plans`

이 카탈로그는 분석 세션 동안만 메모리에 유지하고, 분석 종료 후 휘발시킨다.

### Step 3. UsageStats 수집

FE는 `UsageStatsManager`로 기기 로컬 UsageStats를 수집한다.

- 서버 전송 없이 로컬에서만 사용
- OTT 관련 패키지 사용량만 수집
- 최근 7일, 30일 사용량과 마지막 사용 시각 등을 분석 입력으로 사용

### Step 4. 결제 정규화

FE는 결제 원본을 후보추출 전에 정규화한다.

정규화 예시:

- `merchantRaw` 대소문자 통일
- 공백/특수문자/PG 표기 정리
- 금액/날짜 형식 통일
- 가능하면 취소/환불/할부 여부 분리

주의:

- 정규화는 FE 로컬 로직이다.
- BE API는 정규화 로직 자체를 대신하는 것이 아니라, 정규화에 필요한 참조 데이터만 제공한다.

## 5. 후보추출 규칙

후보추출의 목표는 "전체 결제내역 중 구독 가능성이 있는 항목만 남기는 것"이다.

후보추출은 AI가 아니라 규칙 기반 로직으로 수행한다.

### 5-1. 입력

- 결제내역 원본 목록
- UsageStats 요약
- 서비스/요금제 카탈로그

### 5-2. 기본 규칙

- 최근 3~6개월 내 같은 merchant가 2회 이상 반복되면 가점
- 결제 간격이 월간, 주간, 연간 패턴이면 가점
- 금액 편차가 작으면 가점
- 디지털/OTT alias와 유사하면 가점
- 오프라인 상점, 일회성 결제, 할부, 환불은 감점 또는 제거

### 5-3. UsageStats 활용 방식

- 관련 앱 사용 흔적이 있으면 후보 유지/가점
- 사용 흔적이 없다고 바로 비구독으로 단정하지는 않는다
- 사용량은 제거 기준이 아니라 보정 신호로 사용한다

### 5-4. 후보추출 결과

후보추출 단계에서는 아래 셋 중 하나로 나누지 않는다.

- 확실한 구독
- 확실히 구독 아님
- 애매함

이 단계는 어디까지나 "분석 대상 후보를 넉넉하게 남기는 전처리"여야 한다.

## 6. 카탈로그/규칙 기반 1차 필터

후보추출 후에는 서비스/요금제 카탈로그를 활용해 1차 필터를 수행한다.

### 6-1. 카탈로그에서 필요한 정보

- `serviceId`
- `serviceName`
- `code`
- `plans[].servicePlanId`
- `plans[].planName`
- `plans[].billingCycle`
- `plans[].monthlyPrice`

장기적으로 더 있으면 좋은 정보:

- `aliases`
- `knownMerchantPatterns`
- `packageNames`
- `priceTolerance`

### 6-2. 1차 필터 판단 기준

- merchant가 서비스 코드/이름/alias와 강하게 매칭되는지
- 결제 금액이 특정 요금제 월환산 금액과 맞는지
- 반복 주기가 요금제 billing cycle과 맞는지
- UsageStats 패키지와 서비스 패키지가 맞는지

### 6-3. 1차 필터 결과

이 단계에서 아래로 나눈다.

- 규칙만으로 확실한 구독
- 규칙만으로 확실히 구독 아님
- 규칙만으로는 판단 불가

`규칙만으로는 판단 불가`인 항목만 다음 단계로 넘긴다.

## 7. merchant 매핑 조회

규칙만으로 판단이 안 된 항목만 `batch-lookup`을 호출한다.

### 7-1. 왜 후보 전체가 아니라 일부만 보내는가

- OTT와 무관한 결제를 매핑 DB까지 태울 필요가 없다.
- `batch-lookup`은 누적 매핑 확인용이지 전건 판정용이 아니다.

### 7-2. 배치 조회 특성

- 현재 API는 1건씩이 아니라 `items[]` 배치 조회다.
- 최대 100건까지 조회 가능하다.
- 동일 merchant는 BE 내부에서 dedupe 가능하다.

즉 "매핑-first는 한 건씩 API를 쳐야 해서 비합리적이다"는 전제는 현재 계약 기준으로 맞지 않는다.

### 7-3. 매핑 결과 해석

- `hitCount >= 3`이면 자동 매칭 가능
- `hitCount < 3`이면 자동 매칭 불가
- 사전에 없으면 미매칭

이 단계까지도 해결되지 않은 항목만 AI로 넘긴다.

## 8. 온디바이스 AI 판단

온디바이스 AI는 unresolved 후보만 입력으로 받는다.

### 8-1. AI 입력 형식

AI에 원본 결제 전체를 넘기지 않는다.

입력은 "정규화된 후보 1건 + UsageStats 요약 + 서비스/요금제 힌트"의 구조화된 목록이어야 한다.

예시 필드:

- `reviewId`
- `merchantRaw`
- `merchantNormalized`
- `amount`
- `paymentDate`
- `recurrence`
- `usageSignals`
- `serviceCatalogHints`
- 선택적으로 `ruleScores`

### 8-2. AI 출력 형식

AI 출력은 자유문이 아니라 구조화된 JSON이어야 한다.

최소 필드:

- `reviewId`
- `decision`
- `subscriptionConfidence`
- `guessedServiceId`
- `serviceConfidence`
- `guessedServicePlanId`
- `planConfidence`
- `reasonCodes`

`decision`은 아래 3개면 충분하다.

- `CONFIRMED_SUBSCRIPTION`
- `CONFIRMED_NON_SUBSCRIPTION`
- `REVIEW_NEEDED`

## 9. 사용자 확인

판단 관점에서는 AI까지 수행해도 애매한 항목만 사용자 확인 대상이다.

다만 현재 제품 UX는 화면명세서 기준으로 아래처럼 구성한다.

- `SCR-002b`에서 감지된 구독 목록과 확인 필요 결제 요약을 함께 보여준다.
- 신뢰도 높은 감지 구독과 추가 판단이 필요한 항목은 섹션으로 분리한다.
- 실제 상세 판단 액션은 `SCR-009`에서 수행한다.

사용자에게 제공하는 액션:

- 구독 아님
- OTT 구독 맞음

현재 정책:

- `구독 아님`은 현재 분석 세션의 프론트 로컬 상태에서만 제외 처리한다.
- 서버에는 사용자가 구독으로 확정한 항목만 `batch-confirm`으로 전송한다.
- 제외 결과의 영구 저장 API는 이번 단계 범위에서 제외하고, 재분석 시 동일 항목이 다시 후보로 나타날 수 있다.

사용자가 구독으로 확정하면:

- AI/BE가 제안한 서비스 후보를 확인한다.
- 필요 시 서비스 후보를 수정한다.
- AI/BE가 제안한 요금제 후보를 확인한다.
- 필요 시 요금제 후보를 수정한다.
- `batch-confirm` 호출

## 10. 왜 이 구조가 확장에 유리한가

이 파이프라인은 OTT에만 묶이지 않는다.

향후 확장 대상:

- 음악 구독
- 클라우드 구독
- 생산성 툴
- 게임패스
- 앱스토어 정기결제
- 멤버십

확장 시 바뀌는 것은 주로 아래다.

- 카탈로그 데이터
- 후보추출 규칙
- 카테고리별 alias/merchant pattern

반면 파이프라인 자체는 유지된다.

- 후보추출
- 규칙 필터
- 매핑
- AI
- 사용자 확인

## 11. FE 구현 권장 위치

후보추출 로직은 FE 분석 모듈에 둔다.

권장 위치:

- 분석 시작 orchestration: `feature/analysis/viewmodel/AnalysisViewModel`
- 후보추출 로직: `feature/analysis/SubscriptionCandidateExtractor`
- 분석용 모델:
  - `feature/analysis/model/PaymentRecord`
  - `feature/analysis/model/AnalysisCandidate`
  - `feature/analysis/model/ServiceCatalog`
- 임시 handoff:
  - `data/repository/AnalysisHandoffRepository`

역할 분리:

- `AnalysisViewModel`
  - 결제 원본 수집 트리거
  - UsageStats 수집
  - 카탈로그 preload
  - 후보추출 호출
  - unresolved 후보 handoff 저장
- `SubscriptionRepository`
  - `batch-lookup`
  - `batch-confirm`
  - pending review 관리

즉 후보추출 로직은 `SubscriptionRepository`가 아니라 `feature/analysis`에서 관리해야 한다.

## 12. 현재 기준으로 바로 쓸 수 있는 API

사용 가능한 API:

- 서비스 목록 조회
- 서비스별 요금제 조회
- 결제내역 조회
- merchant 매핑 배치 조회
- merchant 매핑 배치 확정

현재 문서상 공백:

- 온디바이스 분석 결과 전송 API
- 분석용 카탈로그 확장 필드(alias, packageNames, knownMerchantPatterns)
- 제외 결과 영구 저장 API [후순위]

## 13. 최종 권장안

현재 팀 기준 최종 권장안은 아래와 같다.

1. FE는 `GET /api/v1/payment-history`로 결제내역을 조회한다.
2. FE는 UsageStats와 서비스/요금제 카탈로그를 로컬 메모리에 적재한다.
3. FE가 규칙 기반 후보추출을 수행한다.
4. FE가 카탈로그/규칙 기반 1차 필터를 수행한다.
5. unresolved 후보만 `batch-lookup`으로 조회한다.
6. 그래도 unresolved인 후보만 온디바이스 AI에 넘긴다.
7. 최종 unresolved만 사용자 판단 대상으로 남긴다.
8. 제품 UI는 `SCR-002b`에서 감지 구독과 확인 필요 결제 요약을 함께 보여주고, 실제 상세 판단은 `SCR-009`에서 진행한다.
9. 사용자 확정 결과는 `batch-confirm`으로 저장하고 매핑을 누적한다.

이 구조의 핵심 메시지는 아래 한 줄로 요약된다.

> 쉬운 것은 규칙과 매핑으로 처리하고, UsageStats가 필요한 애매한 케이스만 온디바이스 AI로 해석한다.
