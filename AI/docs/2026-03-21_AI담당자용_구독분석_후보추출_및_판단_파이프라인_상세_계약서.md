# AI 담당자용 구독분석 후보추출 및 판단 파이프라인 상세 계약서

> 작성일: 2026-03-21  
> 대상: AI/Data 담당, FE 담당, 필요 시 BE 담당  
> 목적: OTT 구독분석 파이프라인에서 AI가 언제 호출되고, 어떤 데이터를 입력받고, 어떤 형식으로 결과를 반환해야 하는지 구현 가능한 수준으로 고정한다.

---

## 1. 이 문서의 핵심 요약

현재 권장 파이프라인은 아래 순서를 따른다.

1. FE가 결제 원본을 bootstrap seam으로 확보한다.
2. FE가 OTT 서비스/요금제 카탈로그를 preload 한다.
3. FE가 기기 로컬 UsageStats를 수집한다.
4. FE가 결제 원본을 정규화한다.
5. FE가 규칙 기반으로 구독 후보를 추출한다.
6. FE가 규칙/카탈로그 기반 1차 필터로 확정 가능한 항목을 먼저 처리한다.
7. 1차 필터로 확정되지 않은 항목만 `batch-lookup`에 보낸다.
8. 그래도 확정되지 않은 항목만 AI에 보낸다.
9. AI 결과까지 반영한 최종 세션 결과를 handoff에 저장한다.

즉 AI는 "전 결제내역 판정기"가 아니라 "규칙/매핑으로도 어려운 후보 해석기"다.

## 2. 현재 코드 기준 실제 책임 분리

### FE 책임

- 결제 원본 bootstrap
- UsageStats 수집
- 서비스/요금제 카탈로그 preload
- 결제 정규화
- 후보추출
- 1차 필터
- `batch-lookup` 호출
- AI 호출용 입력 payload 생성
- AI 응답을 `CandidateResolution`로 매핑
- 최종 `AnalysisSessionResult` handoff 저장

### AI 책임

- FE가 만든 unresolved 후보만 입력으로 받는다.
- 각 후보에 대해 구독 여부 판단값과 서비스/요금제 추정치를 구조화된 형태로 반환한다.
- `reviewId`를 절대 바꾸지 않고 그대로 round-trip 한다.
- 서비스/요금제는 FE가 넘겨준 카탈로그 범위 안에서만 선택한다.
- 확신이 낮으면 무리하게 확정하지 말고 `REVIEW_NEEDED`를 반환한다.

### BE 책임

- 서비스 목록/요금제 목록 제공
- merchant mapping `batch-lookup`, `batch-confirm` 제공
- `GET /api/v1/payment-history` 제공

## 4. 파이프라인 단계별 입출력 계약

## Step 1. 결제 원본 bootstrap seam

### 목적

- `GET /api/v1/payment-history` 응답을 FE 분석 세션에서 사용할 수 있는 `PaymentRecord` 리스트로 변환해 확보한다.

### 현재 상태

- FE가 `GET /api/v1/payment-history`를 호출한다.
- 성공 시 실제 결제내역 또는 실제 빈 목록을 반환한다.
- 실패 시 앱이 죽지 않도록 placeholder fallback(`isPlaceholder=true`)을 반환한다.

### 목표 입력

- 없음. 로그인 사용자 BearerToken 기반으로 결제내역을 조회한다.

### 목표 출력 타입

```kotlin
data class PaymentBootstrapPayload(
    val paymentRecords: List<PaymentRecord>,
    val isPlaceholder: Boolean,
    val placeholderReason: String? = null,
)
```

### `PaymentRecord` 계약

```kotlin
data class PaymentRecord(
    val paymentId: String,
    val merchantRaw: String,
    val amount: Int,
    val paymentDate: LocalDate,
    val currency: String = "KRW",
    val installmentMonths: Int? = null,
    val canceled: Boolean = false,
    val refunded: Boolean = false,
)
```

### 필드 설명

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| `paymentId` | `String` | Y | 세션 내 stable 식별자. 결제 1건당 유일해야 함 |
| `merchantRaw` | `String` | Y | 결제내역 조회 API 응답의 `merchantRaw` |
| `amount` | `Int` | Y | 결제 금액. KRW 기준 정수 |
| `paymentDate` | `LocalDate` | Y | 결제일 (`yyyy-MM-dd`) |
| `currency` | `String` | Y | 현재는 `"KRW"` 고정 가정 |
| `installmentMonths` | `Int?` | N | 할부 개월 수. 있으면 후보추출에서 감점/제외 가능 |
| `canceled` | `Boolean` | Y | 취소 여부 |
| `refunded` | `Boolean` | Y | 환불 여부 |

### 예시 JSON

```json
{
  "paymentRecords": [
    {
      "paymentId": "pay_20260301_001",
      "merchantRaw": "NETFLIX.COM",
      "amount": 17000,
      "paymentDate": "2026-03-01",
      "currency": "KRW",
      "installmentMonths": null,
      "canceled": false,
      "refunded": false
    }
  ],
  "isPlaceholder": false,
  "placeholderReason": null
}
```

## Step 2. 서비스/요금제 카탈로그 preload

### 목적

- AI가 service/plan을 추정할 때 사용할 정본 catalog를 FE 세션 메모리에 확보한다.

### 현재 API

- `GET /api/v1/services?category=OTT`
- `GET /api/v1/services/{serviceId}/plans`

### FE 내부 표준 타입

```kotlin
data class ServiceCatalog(
    val services: List<Service> = emptyList(),
    val loadedAtEpochMs: Long? = null,
)

data class Service(
    val serviceId: Long? = null,
    val code: String,
    val name: String,
    val logoUrl: String? = null,
    val aliases: Set<String> = emptySet(),
    val packageNames: Set<String> = emptySet(),
    val knownMerchantPatterns: Set<String> = emptySet(),
    val priceTolerance: Int? = null,
    val plans: List<Plan> = emptyList(),
)

data class Plan(
    val servicePlanId: Long? = null,
    val planName: String,
    val billingCycle: String,
    val monthlyPrice: Int,
)
```

### AI가 반드시 신뢰해야 하는 필드

- `serviceId`
- `servicePlanId`
- `hitCount`
- `planName`
- `billingCycle`
- `monthlyPrice`

### AI가 선택하면 안 되는 것

- catalog에 없는 `serviceId`
- catalog에 없는 `servicePlanId`
- FE가 넘기지 않은 임의 서비스명

## Step 3. UsageStats 수집

### 목적

- OTT 앱 사용 흔적을 기기 로컬에서만 수집해 후보 유지/판단 보조 신호로 사용한다.

### 타입

```kotlin
data class AppUsageSnapshot(
    val packageName: String,
    val usage7dMs: Long,
    val usage30dMs: Long,
    val lastUsedEpochMs: Long?,
    val permissionGranted: Boolean,
    val reason: String?,
    val queriedAtEpochMs: Long,
    val timezone: String,
)
```

### 필드 해석 규칙

- `usage7dMs > 0` 이면 최근 7일 사용 흔적 있음
- `usage30dMs > 0` 이면 최근 30일 사용 흔적 있음
- `lastUsedEpochMs != null` 이면 최근 30일 범위 내 최종 사용 시점 존재
- `reason == "no_history"` 는 조회는 성공했지만 흔적 없음
- `permissionGranted == false` 면 이 세션의 사용량 신호를 신뢰하면 안 됨

### 중요한 해석 원칙

- 사용 흔적이 있다고 해서 무조건 구독 확정은 아님
- 사용 흔적이 없다고 해서 무조건 비구독은 아님
- UsageStats는 제거 기준이 아니라 보정 신호다

## Step 4. 결제 정규화

### 목적

- merchant 원문을 AI/규칙 엔진이 비교하기 쉬운 형태로 정리한다.

### 타입

```kotlin
data class NormalizedPaymentRecord(
    val paymentRecord: PaymentRecord,
    val merchantNormalized: String,
    val merchantTokens: List<String> = emptyList(),
    val normalizedAmount: Int = paymentRecord.amount,
    val paymentDate: LocalDate = paymentRecord.paymentDate,
    val currency: String = paymentRecord.currency,
    val ruleTags: Set<String> = emptySet(),
)
```

### 현재 정규화 규칙

- merchant를 대문자화
- 영문/숫자/한글 외 문자를 공백으로 치환
- 연속 공백 제거
- token 분리
- generic gateway 패턴이면 `ruleTags`에 `generic_gateway`
- 취소/환불/할부면 `ruleTags`에 관련 태그 부여

### 예시

| merchantRaw | merchantNormalized |
| --- | --- |
| `Netflix.com` | `NETFLIX COM` |
| `APPLE.COM/BILL` | `APPLE COM BILL` |
| `쿠팡플레이 정기결제` | `쿠팡플레이 정기결제` |

## Step 5. 후보추출

### 목적

- 전체 결제 중 "구독 가능성이 있는 결제"만 AI/매핑 대상으로 남긴다.

### 출력 타입

```kotlin
data class AnalysisCandidate(
    val reviewId: Long,
    val paymentRecord: PaymentRecord,
    val normalizedPaymentRecord: NormalizedPaymentRecord,
    val usageSignals: List<AppUsageSnapshot> = emptyList(),
    val serviceCatalogHints: List<ServiceCatalog.Service> = emptyList(),
    val recurrenceLabel: String? = null,
    val ruleScores: Map<String, Double> = emptyMap(),
)
```

### `reviewId` 규칙

- FE가 세션 내 stable key로 만든다
- AI는 입력으로 받은 `reviewId`를 결과에 그대로 반환해야 한다
- AI가 새 ID를 만들면 안 된다

### `ruleScores` 예시 키

- `catalogAlias`
- `repeatMerchant`
- `planPriceHint`
- `usageSignals`
- `genericSubscriptionToken`
- `offlinePenalty`
- `candidateScore`

### `recurrenceLabel` 예시 값

- `WEEKLY`
- `MONTHLY`
- `QUARTERLY`
- `YEARLY`
- `REPEATED`

## Step 6. 1차 필터

### 목적

- 규칙/카탈로그만으로 확정 가능한 것은 AI까지 보내지 않고 FE에서 먼저 정리한다.

### 출력

1. `CONFIRMED_SUBSCRIPTION`
2. `CONFIRMED_NON_SUBSCRIPTION`
3. unresolved candidate 리스트

### 현재 FE 판단 기준

- strong alias match
- plan price 유사도
- usage package 일치 여부
- 월간/연간 recurrence 패턴

### 중요 포인트

- 1차 필터에서 unresolved로 남은 것만 `batch-lookup`으로 간다
- AI는 1차 필터를 통과하지 못한 어려운 항목만 받는다

## Step 7. batch-lookup

### 목적

- unresolved candidate의 merchant를 누적 mapping DB에 조회해 확정 또는 suggestion을 얻는다.

### 요청 API

`POST /api/v1/merchant-mappings/batch-lookup`

### 요청 형식

```json
{
  "items": [
    {
      "merchantRaw": "NETFLIX.COM",
      "predictedServiceId": 1,
      "predictedServicePlanId": 12
    },
    {
      "merchantRaw": "APPLE.COM/BILL",
      "predictedServiceId": 6,
      "predictedServicePlanId": 61
    },
    {
      "merchantRaw": "GOOGLE*YOUTUBE",
      "predictedServiceId": 7,
      "predictedServicePlanId": 71
    }
  ]
}
```

### 응답 형식

```json
{
  "success": true,
  "data": {
    "results": [
      {
        "merchantRaw": "NETFLIX.COM",
        "matched": true,
        "serviceId": 1,
        "servicePlanId": 12,
        "hitCount": 5
      },
      {
        "merchantRaw": "APPLE.COM/BILL",
        "matched": true,
        "serviceId": 6,
        "servicePlanId": 61,
        "hitCount": 2
      },
      {
        "merchantRaw": "GOOGLE*YOUTUBE",
        "matched": false,
        "serviceId": 7,
        "servicePlanId": 71,
        "hitCount": null
      }
    ]
  }
}
```

### 해석 규칙

- `matched == true && hitCount >= 3`
  - FE가 `CONFIRMED_SUBSCRIPTION` + `highConfidence=true` 로 확정
  - AI로 보내지 않음
- `matched == true && hitCount < 3`
  - FE가 suggestion으로 보존
  - AI 입력에 `batchLookupHint`로 포함
- `matched == false`
  - FE가 unresolved로 간주
  - AI 입력에 hint 없이 포함

### 중요한 변경사항

예전에는 `matched == true` 결과를 pending review 생성에서 버릴 수 있었지만, 현재는 절대 버리면 안 된다.  
`matched == true`는 아래 둘 중 하나로 반드시 세션 결과에 남아야 한다.

1. high-confidence 확정 결과
2. low-confidence suggestion 결과

### AI 입력 필드 해석 규칙

| 필드 | 설명 |
| --- | --- |
| `reviewId` | 결과에 반드시 그대로 반환해야 하는 key |
| `payment` | 원본 결제 정보 |
| `normalizedPayment` | 정규화 결과 |
| `recurrenceLabel` | 반복 주기 힌트 |
| `ruleScores` | FE 규칙 엔진의 중간 점수 |
| `usageSignals` | 후보 서비스 앱 사용 흔적 |
| `serviceCatalogHints` | FE가 narrowing 한 서비스 후보군 |
| `batchLookupHint` | 누적 merchant 매핑 suggestion. 없을 수 있음 |

### AI가 지켜야 하는 강제 규칙

1. 입력에 없는 `reviewId`를 만들면 안 된다.
2. 입력에 없는 `serviceId`를 추론 결과로 만들면 안 된다.
3. `servicePlanId`는 반드시 해당 `serviceId`의 plan 목록 안에서만 선택해야 한다.
4. 확신이 낮으면 `REVIEW_NEEDED`를 반환해야 한다.
5. 장문의 자연어 설명 대신 구조화된 `reasonCodes`를 우선 사용한다.

## Step 9. AI 출력 계약

### 최소 출력 envelope

```json
{
  "schemaVersion": "analysis-ai-output-v1",
  "sessionId": "analysis-1774100000000",
  "results": [
    {
      "reviewId": 812341234,
      "decision": "CONFIRMED_SUBSCRIPTION",
      "subscriptionConfidence": 0.91,
      "guessedServiceId": 6,
      "guessedServiceName": "디즈니+",
      "serviceConfidence": 0.92,
      "guessedServicePlanId": 602,
      "guessedServicePlanName": "프리미엄",
      "planConfidence": 0.88,
      "reasonCodes": [
        "AI_USAGE_MATCH",
        "AI_MONTHLY_RECURRENCE",
        "AI_PLAN_PRICE_MATCH",
        "BATCH_LOOKUP_SUGGESTION_USED"
      ]
    }
  ]
}
```

### 필드 정의

| 필드 | 타입 | 필수 | 규칙 |
| --- | --- | --- | --- |
| `reviewId` | `Long` | Y | 입력값 그대로 |
| `decision` | `String` | Y | 아래 enum만 허용 |
| `subscriptionConfidence` | `Float` | Y | 0.0 ~ 1.0 |
| `guessedServiceId` | `Long?` | N | catalog 범위 내에서만 |
| `guessedServiceName` | `String?` | N | FE가 넘긴 catalog 이름과 일치 권장 |
| `serviceConfidence` | `Float?` | N | 0.0 ~ 1.0 |
| `guessedServicePlanId` | `Long?` | N | 해당 service의 plan 범위 내에서만 |
| `guessedServicePlanName` | `String?` | N | catalog planName과 일치 권장 |
| `planConfidence` | `Float?` | N | 0.0 ~ 1.0 |
| `reasonCodes` | `List<String>` | Y | snake/upper case 권장 |

### `decision` 허용값

- `CONFIRMED_SUBSCRIPTION`
- `CONFIRMED_NON_SUBSCRIPTION`
- `REVIEW_NEEDED`

### FE 매핑 대상 타입

AI 출력은 최종적으로 아래 FE 표준 타입으로 매핑된다.


### AI 출력 해석 기준

- `decision == CONFIRMED_SUBSCRIPTION && subscriptionConfidence >= 0.85`
  - FE에서 high-confidence 구독으로 분류 가능
- `decision == REVIEW_NEEDED`
  - FE에서 확인 필요 결제로 분류
- `decision == CONFIRMED_NON_SUBSCRIPTION`
  - FE에서 이번 세션 구독 후보에서 제외

## 7. AI 구현 시 체크리스트

- `reviewId` round-trip 보장
- 입력 후보 수와 결과 수를 1:1로 맞추기
- confidence 범위는 0.0~1.0 준수
- `serviceId/servicePlanId`는 catalog 범위 밖 값 생성 금지
- `REVIEW_NEEDED`를 과감하게 사용하기
- batch-lookup suggestion이 있으면 참고하되 맹신하지 않기
- usage signal과 recurrence가 conflict하면 `REVIEW_NEEDED` 우선
- generic merchant(`APPLE`, `GOOGLE`, `BILL`)는 usage/price/recurrence를 함께 보고 판단하기

### 교체 후에도 유지되어야 하는 계약

- FE는 unresolved candidate만 AI에 전달한다

- AI는 catalog 안의 service/plan만 선택한다
