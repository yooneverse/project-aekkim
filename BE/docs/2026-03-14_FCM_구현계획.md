| 대상 범위 | Backend, Android Frontend, Infra |

---

## 1. 목표

- 앱 사용자의 FCM 토큰을 서버에 저장할 수 있어야 한다.
- 백엔드가 특정 디바이스 토큰으로 테스트 푸시를 전송할 수 있어야 한다.
- 이후 체크인/프로모션 알림 및 Batch 작업이 같은 발송 로직을 재사용할 수 있어야 한다.

---

## 2. 선행 준비물

### 2.1 Firebase 프로젝트

- Backend 담당자가 Firebase 프로젝트 생성
- Android 앱 등록
- `google-services.json` 발급
- Cloud Messaging 사용 준비
- Firebase Admin SDK용 서비스 계정 키(JSON) 발급

### 2.2 환경 변수 / 시크릿

- FE
  - `google-services.json`
- BE
  - Firebase 서비스 계정 키 파일 또는 JSON secret
  - `GOOGLE_CLIENT_ID`
  - `JWT_REFRESH_TTL`

### 2.3 현재 전제

- auth 로그인/재발급/로그아웃은 구현 완료
- `fcm_tokens` 테이블은 ERD에 존재
- 알림 API 문서 초안은 [2026-03-12_알림_API.md](/Users/SSAFY/workspace/S14P21E106/docs/api/2026-03-12_알림_API.md)에 존재

---

## 3. 구현 원칙

- 운영 사용자용 알림 전송 API는 만들지 않는다.
- 초기에는 dev/local 전용 테스트 발송 API로 FCM 연동 자체를 검증한다.
- FCM 발송 로직은 컨트롤러에서 직접 구현하지 않고 서비스로 분리한다.
- 테스트 발송 성공 후 FCM 토큰 등록 API를 연결한다.
- Batch는 마지막 단계에서 붙인다.

---

## 4. 백엔드 작업

### 4.0 0단계 - Firebase 프로젝트 생성 및 권한 관리

- Backend 담당자가 Firebase 프로젝트 생성
- Android 앱 등록에 필요한 패키지명 확인
- FE에 `google-services.json` 전달
- Firebase Admin SDK용 서비스 계정 키(JSON) 발급 및 보관 방식 결정

완료 기준:
- Firebase 프로젝트가 생성되어 있음
- FE/BE가 각자 필요한 설정 파일 또는 시크릿을 전달받음

### 4.1 1단계 - Firebase Admin SDK 연동

- `firebase-admin` 의존성 추가
- Firebase 설정 클래스 추가
- 서비스 계정 키 로딩 방식 결정
  - 파일 경로 기반
  - 환경 변수 JSON 문자열 기반
- `FirebaseApp`, `FirebaseMessaging` 빈 등록

완료 기준:
- 애플리케이션이 Firebase Admin 초기화까지 성공
- local/dev 환경에서 서비스 계정 키를 읽을 수 있음

### 4.2 2단계 - FCM 발송 서비스 구현

- 예시 클래스명
  - `FcmService`
  - `NotificationSender`
- 최소 기능
  - 단일 token 발송
  - 입력: `token`, `title`, `body`
  - 출력: Firebase messageId
- 실패 처리
  - invalid / expired token 구분 가능하도록 로그 남김
  - 추후 토큰 정리 로직 확장 가능하게 구조 설계

완료 기준:
- 서비스 메서드 단독 호출로 테스트 푸시 전송 가능

### 4.3 3단계 - dev 전용 테스트 발송 API

- 컨트롤러 예시
  - `DevNotificationController`
- 프로필 제한
  - `@Profile({"local", "dev"})`
- 엔드포인트 예시
  - `POST /api/v1/dev/notifications/test`
- 요청 필드 예시
  - `token`
  - `title`
  - `body`

목적:
- DB 저장 없이 “푸시가 실제로 오는지” 먼저 검증

완료 기준:
- Swagger/Postman으로 직접 발송 테스트 가능
- 실제 디바이스에서 푸시 수신 확인 가능

### 4.4 4단계 - FCM 토큰 등록/갱신 API

- API 문서 기준
  - `POST /api/v1/notifications/tokens`
- 인증 필요
  - `Authorization: Bearer {accessToken}`
- 처리 로직
  - 현재 로그인 사용자 기준 저장
  - 동일 토큰 존재 시 갱신
  - 중복 저장 방지

검토 항목:
- 유저당 다중 디바이스 허용 여부
- 플랫폼(Android) 저장 필요 여부
- 마지막 갱신 시각 저장 여부

완료 기준:
- FE가 받은 FCM 토큰을 서버에 등록 가능
- 등록/갱신 응답이 API 문서와 일치

### 4.5 5단계 - 이후 확장 준비

- userId 기준 토큰 조회 메서드
- 다중 토큰 발송 구조
- 실패 토큰 정리 정책
- 알림 이력 저장 필요 여부 검토

## 5. 프론트 작업

### 5.1 1단계 - Firebase Android 연결

- Backend가 생성한 Firebase 프로젝트 기준으로 Android 앱 연결
- 전달받은 `google-services.json` 추가
- Firebase Messaging 설정 확인
- 앱 빌드 및 Firebase 초기화 확인

완료 기준:
- 앱 실행 시 Firebase 설정 로드 성공

### 5.2 2단계 - FCM registration token 획득

- 앱에서 FCM 토큰 조회
- 디버그 로그 또는 화면에서 토큰 확인 가능하게 준비
- 테스트용으로 토큰을 백엔드에 전달 가능하게 함

완료 기준:
- 디바이스별 FCM 토큰 확인 가능

### 5.3 3단계 - 테스트 푸시 수신 확인

- 백엔드 dev 테스트 API에 FCM token 전달
- foreground / background 수신 동작 확인

완료 기준:
- 실제 디바이스에서 푸시 수신 성공

### 5.4 4단계 - FCM 토큰 등록 API 연동

- 로그인 후 또는 앱 시작 시 토큰 등록 호출
- 토큰 갱신 시 재등록 처리
- 로그아웃/탈퇴 시 정책 협의

완료 기준:
- 서버에 현재 사용자 FCM 토큰 등록 완료

- EC2 내 파일 경로 표준화
