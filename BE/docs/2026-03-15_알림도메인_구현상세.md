# 알림 도메인 구현 상세

| 항목 | 내용 |
| --- | --- |
| 문서 목적 | 알림 도메인 1차 구현 범위와 백엔드/프론트 작업 기준 정리 |
| 작성일 | 2026-03-15 |
| 대상 범위 | Backend, Android Frontend, Infra |
| 관련 이슈 | S14P21E106-135, S14P21E106-121, S14P21E106-122, S14P21E106-123 |
| 상태 | 구현 상세 |

---

## 1. 목표

- `fcm_tokens`, `notifications`를 실제 도메인 코드로 구현한다.
- FCM 토큰 등록/갱신 API를 제공한다.
- 현재 로그인 사용자의 알림 목록 조회 API를 제공한다.
- 특정 알림의 읽음 처리 API를 제공한다.
- 이번 단계 이후 후속 구현으로 CHECKIN 대상 선정 배치, CHECKIN 발송 배치, PROMO 대상 선정 배치, PROMO 발송 배치, Redis 기반 중복 방지 로직이 추가되었다.

---

## 2. 범위

### 2.1 포함

- `POST /api/v1/notifications/tokens`
- `GET /api/v1/notifications`
- `PATCH /api/v1/notifications/{notificationId}/read`
- `notifications`, `fcm_tokens` 엔티티/리포지토리/서비스/컨트롤러 구현
- 프론트의 토큰 등록, 목록 조회, 읽음 처리 연동

### 2.2 제외

- 프로모션 알림 생성
- 프로모션 추천 알림 생성
- churn review 알림 생성
- 프로모션 Batch 스케줄링
- 구독/프로모션 이벤트와의 자동 연결

---

## 3. 전제 조건

- auth 리팩토링이 완료되어 `users` 기준 인증이 동작한다.
- FCM 테스트 발송 API(`/api/v1/dev/notifications/test`)가 동작한다.
- Firebase 프로젝트, Android 앱 등록, 서비스 계정 JSON 준비가 완료되어 있다.
- FE는 Firebase Messaging SDK 설정과 `google-services.json` 적용이 완료되어 있다.

## 4. ERD 기준 모델

### 4.1 `fcm_tokens`

| 컬럼 | 설명 |
| --- | --- |
| `fcm_token_id` | PK |
| `user_id` | `users.user_id` FK |
| `fcm_token` | UNIQUE, 등록 토큰 |
| `created_at`, `updated_at` | `BaseEntity` 상속 기준 |

### 4.2 `notifications`

| 컬럼 | 설명 |
| --- | --- |
| `notification_id` | PK |
| `user_id` | `users.user_id` FK |
| `type` | `CHECKIN`, `PROMO`, `CHURN_REVIEW` |
| `reference_id` | 타입별 참조 ID |
| `title` | 알림 제목 |
| `body` | 알림 본문 |
| `sent_at` | 발송 시각 |
| `read_at` | 읽음 시각 |

### 4.3 `reference_id` 의미

| type | reference_id 의미 |
| --- | --- |
| `CHECKIN` | `subscriptions.subscription_id` |
| `PROMO` | `promotions.promotion_id` |
| `CHURN_REVIEW` | `subscriptions.subscription_id` |

참고:
- 이번 단계에서는 `referenceId`를 저장/조회만 하고, 실제 참조 대상 검증은 생성 로직 단계에서 처리한다.

## 5. 백엔드 구현 상세

### 5.1 엔티티

#### `FcmToken`

- 위치: `domain/notification/entity/FcmToken`
- 필드
  - `Long fcmTokenId`
  - `User user`
  - `String fcmToken`
- 매핑
  - `@ManyToOne(fetch = FetchType.LAZY)`로 `User` 연결
  - `fcm_token`은 unique
- 역할
  - 로그인 사용자와 디바이스 토큰 연결

#### `Notification`

- 위치: `domain/notification/entity/Notification`
- 필드
  - `Long notificationId`
  - `User user`
  - `NotificationType type`
  - `Long referenceId`
  - `String title`
  - `String body`
  - `LocalDateTime sentAt`
  - `LocalDateTime readAt`
- 매핑
  - `@ManyToOne(fetch = FetchType.LAZY)`로 `User` 연결
  - `NotificationType`은 `@Enumerated(EnumType.STRING)`
- 역할
  - 사용자별 알림 이력 저장

### 5.2 리포지토리

#### `FcmTokenRepository`

- 필요 메서드
  - `Optional<FcmToken> findByFcmToken(String fcmToken)`
  - 필요 시 `List<FcmToken> findAllByUser_UserId(Long userId)`

#### `NotificationRepository`

- 필요 메서드
  - `Optional<Notification> findByNotificationIdAndUser_UserId(Long notificationId, Long userId)`
  - 사용자 기준 목록 조회 메서드
- 정렬
  - `sentAt desc`, 그 다음 `notificationId desc`

### 5.3 서비스

#### `NotificationService`

- 책임
  - DB 기준 알림 비즈니스 로직 전담
- 메서드
  - `upsertFcmToken(Long userId, UpsertFcmTokenRequest request)`
  - `getNotifications(Long userId, boolean unreadOnly, int size)`
  - `markAsRead(Long userId, Long notificationId)`

#### `FcmService`

- 책임
  - Firebase 발송 전담
- 현재 역할 유지
  - 토큰 저장/조회와 분리

### 5.4 API 121 - FCM 토큰 등록/갱신

- URL: `POST /api/v1/notifications/tokens`
- 인증: 필요
- 요청
  - `fcmToken`
- 응답
  - `fcmTokenId`
  - `fcmToken`
  - `updatedAt`

처리 규칙:
- `userId`는 access token에서 추출
- 동일 `fcmToken` row가 있으면 user와 timestamp 갱신
- 없으면 신규 insert
- 성공 시 `200 OK`

구현 포인트:
- upsert 정책을 명확히 유지
- `fcmToken` 값 검증은 `@NotBlank`

### 5.5 API 122 - 알림 목록 조회

- URL: `GET /api/v1/notifications`
- 인증: 필요
- 쿼리
  - `unreadOnly` 기본값 `false`
  - `size` 기본값 `20`, 최대 `100`

응답 항목:
- `notificationId`
- `type`
- `referenceId`
- `title`
- `body`
- `isRead`
- `readAt`
- `sentAt`

처리 규칙:
- 현재 로그인 사용자 기준 조회
- `unreadOnly=true`면 `readAt is null`
- `size`는 범위 검증 필요
- `isRead`는 `readAt != null`로 계산

### 5.6 API 123 - 알림 읽음 처리

- URL: `PATCH /api/v1/notifications/{notificationId}/read`
- 인증: 필요

응답 항목:
- `notificationId`
- `isRead`
- `readAt`

처리 규칙:
- 현재 로그인 사용자의 알림만 읽음 처리 가능
- 타 사용자 알림이면 `FORBIDDEN`
- 알림이 없으면 `RESOURCE_NOT_FOUND`
- 이미 읽은 상태면 기존 `readAt` 유지 후 반환 가능

### 5.7 컨트롤러

#### `NotificationController`

- 엔드포인트
  - `POST /api/v1/notifications/tokens`
  - `GET /api/v1/notifications`
  - `PATCH /api/v1/notifications/{notificationId}/read`
- 공통
  - `@SecurityRequirement(name = "bearerAuth")`
  - `@Operation` 추가
  - `ApiResponse<T>`로 응답

### 5.8 보안/설정

- `SecurityConfig`에 `/api/v1/notifications/**`를 인증 필요 경로로 추가
- 이 단계부터 `notification` 엔티티/리포지토리가 실제 구현되므로, `E106Application`의 JPA 스캔 범위에 `notification`을 포함하는 것을 다시 반영한다.

### 5.9 예외 처리

- `INVALID_INPUT`
  - `fcmToken`, `size` 검증 실패
- `UNAUTHORIZED`
  - 인증 실패
- `AUTH_USER_NOT_FOUND`
  - 토큰 사용자 없음
- `RESOURCE_NOT_FOUND`
  - 알림 없음
- `FORBIDDEN`
  - 타 사용자 알림 읽음 처리 시도
- `INTERNAL_ERROR`
  - 서버 예외

## 6. 프론트 구현 상세

### 6.1 API 121 연동 - FCM 토큰 등록/갱신

- 시점
  - 로그인 직후
  - 앱 시작 후 토큰 갱신 감지 시
- 처리
  - Firebase Messaging으로 FCM registration token 확보
  - `POST /api/v1/notifications/tokens` 호출
  - `Authorization: Bearer {accessToken}` 포함

응답 처리:
- `fcmTokenId`, `updatedAt`는 로컬 저장 필수는 아님
- 성공 여부만 확인해도 충분

### 6.2 API 122 연동 - 알림 목록 조회

- 화면
  - 알림함 진입 시점
- 호출
  - `GET /api/v1/notifications`
  - 필요 시 `unreadOnly=true`
  - 초기 `size=20`

UI 요구사항:
- 최신순 목록
- 읽지 않음 표시
- 빈 목록 상태
- 로딩 상태
- 에러 상태

### 6.3 API 123 연동 - 읽음 처리

- 트리거
  - 알림 클릭 시
  - 알림 상세 진입 시
- 호출
  - `PATCH /api/v1/notifications/{notificationId}/read`

UI 처리:
- 성공 시 해당 항목 `isRead=true`
- `readAt` 반영
- 필요하면 낙관적 업데이트 적용 가능

### 6.4 `referenceId` 처리

- 이번 단계에서는 `referenceId`를 화면 분기용으로만 저장 가능
- 실제 상세 이동은 다음 단계에서 연결
  - `CHECKIN` -> 구독 상세
  - `PROMO` -> 프로모션 상세
  - `CHURN_REVIEW` -> 구독 상세

### 6.5 수신 처리

- foreground/background 푸시 수신 자체는 기존 FCM 연동 기반 유지
- 이번 이슈에서는 “등록/조회/읽음” API 연동이 우선
