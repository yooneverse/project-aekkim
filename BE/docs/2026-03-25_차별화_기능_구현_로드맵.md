# 차별화 기능 구현 로드맵

## 1. 문서 목적

- 서비스 차별화에 직접 연결되는 7개 기능을 구현 관점에서 다시 정리한다.
- 각 기능마다 무엇이 필요한지, 어떤 방식으로 구현할지, 지금 바로 가능한지 여부를 명확히 정리한다.
- 현재 구조에서 바로 할 수 있는 것과, 사용량 저장 구조를 먼저 만들어야 하는 것을 구분한다.

## 2. 대상 기능

이번 문서에서 다루는 기능은 아래 6개다.

1. 구독 연관 리포트
2. 목표 기반 동기부여
3. 구독료 자극 멘트
4. 앱 사용 시간 / 시간 대비 사용량
5. 잔디형 사용량 시각화
6. 추세 그래프 / 차트

## 3. 현재 구조와 한계

### 3.1 지금 프론트/백엔드가 보는 최근 사용량

현재 사용량 기능의 기준 데이터는 `subscription_usage_daily` 기반의 실제 일별 사용 시간이다.

- 기준 데이터: 서비스별 일별 사용량 집계
- 단위: 일 단위
- 값: `used_minutes`
- 상세 조회 범위: 첫 사용 기록이 있는 시점부터 오늘까지

즉 지금 구조는 “최근 3개월 체크인 요약”이 아니라, 일별 사용 기록을 월 단위로 탐색하는 형태에 가깝다.

### 3.2 지금 상태로 가능한 것

현재 구조로도 일부 기능은 규칙 기반으로 구현 가능하다.

- 구독료 자극 멘트
- 구독 연관 리포트의 단순 버전

### 3.3 지금 상태로 부족한 것

아래 기능은 체크인 이력만으로는 완성도 있게 구현하기 어렵다.

- 앱 사용 시간 / 시간 대비 사용량
- 잔디형 사용량 시각화
- 추세 그래프 / 차트
- 정교한 구독 연관 리포트
- 목표 기반 동기부여의 실제 진행률 계산

결론적으로 차별화 기능을 제대로 만들려면 “일별 사용량 집계”를 저장하는 구조가 필요하다.

## 4. 공통 데이터 설계

## 4.1 왜 별도 사용량 저장이 필요한가

7개 기능의 공통 핵심은 “최근 사용 흐름”이다.

이때 필요한 것은 raw 이벤트 전체가 아니라, 하루 단위의 최소 집계다.

예시:

- 사용자 1
- 넷플릭스
- 2026-03-26
- 42분 사용

이런 row가 하루에 서비스별 1건만 쌓이면 대부분의 기능을 계산할 수 있다.

## 4.2 최소 저장 필드

객관적 필요성 기준으로 꼭 필요한 컬럼은 아래 5개다.

- `usage_id`
- `user_id`
- `service_id`
- `usage_date`
- `used_minutes`

### 각 필드가 필요한 이유

- `usage_id`
  - 행 식별자
  - 단일 PK로 관리하는 것이 가장 단순하다.

- `user_id`
  - 누구의 사용량인지 구분하기 위해 필요하다.

- `service_id`
  - 어떤 서비스 사용량인지 구분하기 위해 필요하다.

- `usage_date`
  - 최근 30일, 잔디형, 추세 그래프, 패턴 판단의 기준 축이다.

- `used_minutes`
  - 시간 대비 사용량, 저사용 판단, 패턴 분석의 핵심 값이다.

## 4.3 권장 테이블 형태

권장 테이블명:

- `subscription_usage_daily`

권장 컬럼:

- `usage_id`
- `user_id`
- `service_id`
- `usage_date`
- `used_minutes`
- `created_at`
- `updated_at`

주의:

- `created_at`, `updated_at`은 이 프로젝트의 `BaseEntity` 구조로 자동 처리 가능하다.
- 즉 실제로 핵심 설계 대상은 `usage_id`, `user_id`, `service_id`, `usage_date`, `used_minutes`다.

## 4.4 권장 제약 및 조회 기준

권장 제약:

- `(user_id, service_id, usage_date)` 유니크

권장 인덱스:

- 유니크 인덱스: `(user_id, service_id, usage_date)`
- 조회 인덱스: `(user_id, usage_date)`

## 4.5 최근 30일의 정의

`최근 30일`은 월 기준이 아니라 rolling window 기준으로 본다.

예시:

- 오늘이 `2026-03-26`이면
- `2026-02-25 ~ 2026-03-26`
- 이 기간의 `usage_date` row를 전부 합산한다.

즉 `최근 30일`은 “최근 달”이 아니라 “오늘 기준 직전 30일 구간”이다.

## 4.6 기존 저사용 필드 유지 기준

현재 `subscriptions` 테이블에는 아래 필드가 있다.

- `low_usage_detected`
- `low_usage_cycle_ym`

이 필드는 지금 당장은 제거하면 안 된다.

이유:

- 현재 체크인/프로모션 알림 배치가 이 필드를 기준으로 발송 대상을 고른다.
- 즉 현재 운영 기준에서는 아직 실제 참조 중인 필드다.

### 제거 가능한 시점

아래 3개가 모두 완료된 뒤 제거한다.

1. `subscription_usage_daily` 기반으로 최근 30일 저사용 여부를 서버에서 직접 계산
2. 체크인/프로모션 배치가 더 이상 `low_usage_detected`, `low_usage_cycle_ym`를 읽지 않음
3. `POST /api/v1/subscriptions/usage-qualification` 경로를 더 이상 사용하지 않음

즉 현재는 유지하고, 사용량 집계 기반 배치 전환 이후 제거하는 것이 맞다.

## 4.7 카테고리 색상 기준

카테고리별 시각 표현은 아래 3축으로 통일한다.

- `OTT` = 주황 계열
- `MUSIC` = 민트/청록 계열
- `AI` = 블루 계열

### 색상 방향

- `OTT`
  - 엔터/영상 서비스 느낌
  - 메인 브랜드 톤과도 자연스럽게 연결

- `MUSIC`
  - 가볍고 리듬감 있는 느낌
  - OTT 주황과 구분이 잘 됨

- `AI`
  - 기능적/생산성/도구 느낌
  - 차분하고 안정적인 강조색으로 사용 가능

### 권장 사용 방식

각 카테고리는 1개의 대표색만 쓰지 않고, 같은 계열의 농도 단계를 함께 사용한다.

- `50` = 아주 연한 배경
- `100` = 연한 칩/카드 배경
- `300` = 일반 강조
- `500` = 강한 포인트

### 예시 색상

- `OTT`
  - `50`: `#FFF4E8`
  - `100`: `#FFD7A8`
  - `300`: `#FFB25B`
  - `500`: `#F28C28`

- `MUSIC`
  - `50`: `#EAFBF7`
  - `100`: `#BCEFE4`
  - `300`: `#5FD0B6`
  - `500`: `#1DAA8A`

- `AI`
  - `50`: `#EEF4FF`
  - `100`: `#CFE0FF`
  - `300`: `#7EA9FF`
  - `500`: `#3F73E0`

### 적용 위치

- 카테고리 탭
- 카테고리 배지
- 차트/잔디 색상 계열
- 리포트 카드 보조 포인트

주의:

- 카테고리 색상은 배경/보조 포인트 위주로 쓰고, 본문 텍스트 색상은 중립 톤을 유지한다.
- 강한 원색을 넓게 깔기보다 연한 배경 + 진한 포인트 조합으로 사용하는 것이 안정적이다.

## 4.8 FE 번들 레지스트리 운영 기준

- 이 번들 레지스트리는 사용량 측정용이 아니라, 결제내역 분석 시 “어떤 번들 결제인지” 후보를 추론하기 위한 보조 데이터다.
- 따라서 단기적으로는 FE 하드코딩 번들 목록을 실제 [bundle.json](/C:/Users/SSAFY/workspace/S14P21E106/seed/bundle.json) 기준과 최대한 동일하게 유지한다.
- 즉 현재 구조에서는 번들 레지스트리를 일부만 두지 않고, 실제 번들 카탈로그에 있는 항목을 최대한 반영하는 것이 맞다.
- 전체 사용량/인사이트 기능 구현이 마무리된 뒤에는 FE가 실제 번들 카탈로그를 직접 읽어 `ServiceCatalog.bundles`를 구성하도록 전환한다.

## 5. 기능별 구현 방식

## 5.1 구독 연관 리포트

### 목표

- 구독을 하나씩 따로 보지 않고, 서로 어떤 관계인지 해석한다.

예시:

- 넷플릭스는 거의 안 보는데 티빙은 자주 봄
- ChatGPT는 자주 쓰는데 Claude는 거의 안 씀
- 같은 카테고리 구독이 중복 유지되고 있음

### 필요한 데이터

- `user_id`
- `service_id`
- `usage_date`
- `used_minutes`
- 기존 구독료
- 기존 서비스 카테고리

### 어떻게 구현할 것인가

- 같은 카테고리 서비스끼리 최근 30일 사용 시간을 비교한다.
- 많이 쓰는 구독과 거의 안 쓰는 구독을 비교 문구로 만든다.
- 카테고리 내 2개 이상 유지 중인 구독만 리포트 대상으로 본다.

### 1차 구현 예시

- `넷플릭스 40분 / 티빙 320분` -> “티빙 중심으로 사용 중”
- `ChatGPT 600분 / Claude 40분` -> “Claude 대비 ChatGPT 활용도가 높음”

### 구현 난이도

- 중상

### 비고

- 처음에는 같은 카테고리 내 2개 비교부터 시작하는 것이 안전하다.

## 5.2 목표 기반 동기부여

### 목표

- 사용자가 절약과 정리 행동을 목표로 잡고 추적할 수 있게 한다.

예시:

- 이번 달 1만 원 줄이기
- 안 쓰는 구독 1개 정리하기
- 체크인 100% 완료하기

### 필요한 데이터

- `user_id`
- `service_id`
- `usage_date`
- `used_minutes`
- 기존 구독료
- 별도 목표 저장 구조

### 어떻게 구현할 것인가

- 자유 입력 목표 대신 고정 템플릿 목표를 먼저 제공한다.
- 최근 30일 사용량과 현재 구독료를 바탕으로 진행률을 계산한다.

### 1차 구현 예시

- “안 쓰는 구독 1개 정리하기”
  - 최근 30일 사용시간이 매우 낮은 구독 수로 계산
- “이번 달 1만 원 줄이기”
  - 저사용 구독의 월 구독료 합으로 후보 제시

### 구현 난이도

- 중상

### 비고

- 목표 저장 테이블은 `usage`와 별개로 추가해야 한다.

## 5.3 구독료 자극 멘트

### 목표

- 사용량 대비 비용을 직관적인 문장으로 보여줘 점검 행동을 유도한다.

예시:

- 이번 달 거의 안 쓴 구독에 또 결제가 예정돼 있어요
- 최근 이용이 적은데 월 구독료는 그대로 나가고 있어요

### 필요한 데이터

- `service_id`
- `used_minutes`
- 기존 구독료
- 기존 체크인 응답
- 기존 다음 결제일

### 어떻게 구현할 것인가

- 최근 30일 사용시간과 월 결제액을 함께 본다.
- 사용량이 낮고 결제일이 가까운 구독에 자극 멘트를 만든다.
- 체크인 이력은 보조 근거로만 활용한다.
- 한 구독당 문구는 최대 1개만 보여준다.
- 어떤 규칙에도 걸리지 않으면 문구를 노출하지 않는다.
- 번들은 포함 서비스 사용시간을 합산한 뒤 공통 문구만 사용한다.

### 1차 구현 예시

- `used_minutes = 0` + 결제일 임박 -> 강한 점검형 문구
- `used_minutes` 낮음 + 월 구독료 높음 -> 완곡한 유도형 문구

### 1차 노출 문구 세트

- `결제만 유지 중이에요`
- `요즘은 좀 뜸해요`
- `자주 쓰는 편이에요`

### 1차 노출 기준

공통 계산값:

- `totalUsedMinutes`: 최근 30일 총 사용시간
- `usedDays`: 최근 30일 사용한 날짜 수
- `daysUntilBilling`: 다음 결제일까지 남은 일수

노출 우선순위:

1. `결제만 유지 중이에요`
   - `totalUsedMinutes = 0`
   - 또는 `usedDays <= 1` 이고 `daysUntilBilling <= 7`

2. `요즘은 좀 뜸해요`
   - `usedDays <= 3`
   - 또는 `totalUsedMinutes <= 30`

3. `자주 쓰는 편이에요`
   - `usedDays >= 12`
   - 또는 `totalUsedMinutes >= 300`

위 조건에 걸리지 않으면 문구를 보여주지 않는다.

### 구현 난이도

- 하

### 비고

- 표현은 강하되 사용자 거부감이 생기지 않게 톤 조절이 중요하다.
- 1차에서는 문구를 늘리지 않고 위 3개만 사용한다.
- 구독별 카드에는 작은 잔디 아래 1줄로 배치한다.

## 5.4 앱 사용 시간 / 시간 대비 사용량

### 목표

- 구독료와 사용시간을 연결해 사용 효율을 직관적으로 보여준다.

예시:

- 넷플릭스 최근 30일 180분 사용
- 시간당 비용 4,500원

### 필요한 데이터

- `service_id`
- `usage_date`
- `used_minutes`
- 기존 월 구독료

### 어떻게 구현할 것인가

- 최근 30일 `used_minutes` 합계를 구한다.
- `월 구독료 / 최근 30일 사용시간`으로 시간당 비용을 계산한다.

### 1차 구현 예시

- 월 구독료 13,500원
- 최근 30일 180분 사용
- 시간당 비용 4,500원

### 구현 난이도

- 중

### 비고

- 사용 시간이 너무 낮으면 “시간당 비용” 대신 “거의 사용 안 함”으로 표시하는 것이 낫다.

## 5.5 잔디형 사용량 시각화

### 목표

- 일별 사용 흐름을 한눈에 보여준다.

### 일별 흐름의 의미

- “일별 흐름”은 최근 N일 동안 날짜별 사용량이 어떻게 변했는지를 뜻한다.
- 예:
  - 3/1 -> 0분
  - 3/2 -> 18분
  - 3/3 -> 42분
- 즉 요약값이 아니라 날짜별 원자료에 가까운 목록이다.

예시:

- 최근 30일 연속 사용 여부
- 공백 구간
- 특정 날짜 집중 사용

### 필요한 데이터

- `usage_date`
- `used_minutes`

### 어떻게 구현할 것인가

- 저장은 서비스별 일별 집계로 유지한다.
- 구독 탭의 리스트에서는 잔디를 제거하고, 구독 상세에서만 큰 잔디를 보여준다.
- 구독 상세의 잔디는 해당 구독 서비스의 월별 사용 기록을 보여준다.
- 월은 한 번에 하나만 보여주고, `< >`로 이전/다음 월을 이동한다.
- 구독 상세 잔디의 셀 탭 시 툴팁은 날짜와 시청 분을 간단히 보여준다.
- 인사이트 탭에서는 전체 잔디 대신 최근 30일 기준 구독별 비교 순위를 보여준다.

### 1차 화면 배치

- 하단 탭에 `인사이트` 탭을 별도로 둔다.
- `구독` 탭의 각 구독 카드에는 잔디를 두지 않는다.
- 구독 상세에서 혜택/추천 아래에 큰 잔디를 배치한다.
- `인사이트` 탭 상단에는 `가장 많이 본 구독 TOP 3`를 배치한다.
- 즉 개별 흐름은 `구독 상세`, 전체 비교는 `인사이트`에서 보여준다.

### 구독 상세 잔디 방향

- 구독 상세 잔디는 서비스 하나의 월별 사용 기록을 보여준다.
- 한 달 단위 heatmap 형태를 유지하되, 셀 크기를 키우고 중앙 정렬한다.
- 결제일에 해당하는 날짜는 별도 테두리로 강조한다.
- 셀 색상은 단순 사용량 단계만 보여줘도 충분하다.

### 셀 선택 시 상호작용

- 구독 상세 잔디 셀은 탭 가능하게 만든다.
- 셀을 누르면 셀 위에 작은 다크 툴팁을 띄운다.
- 툴팁은 약 2초간 유지한 뒤 자동으로 사라진다.
- 툴팁 내용은 날짜와 시청 분을 2줄로 보여준다.

예시:

- `3월 26일`
- `42분 시청`

### 구현 기준

- 툴팁은 바텀시트나 우측 고정 패널 대신, Github contribution 툴팁에 가까운 작은 오버레이 톤으로 구현한다.
- 툴팁은 선택한 셀 근처에 겹치게 띄운다.
- 폰트는 크게 키우지 않고 날짜 `12~13sp` 수준의 작은 정보 카드 느낌으로 유지한다.
- 모바일 잔디 셀 특성상 hover 개념이 없으므로 `tap -> 2초 노출 -> fade out` 흐름으로 본다.

### 메인 구독 탭 리마인드 배너

- 구독 탭 상단, `이번 달 구독료` 카드 바로 위에 리마인드 배너를 둔다.
- 노출 조건은 `최근 사용일로부터 7일 이상 경과`로 단순화한다.
- 후보가 1개면 서비스명을 직접 노출한다.
  - 예: `넷플릭스를 최근에 다시 보지 않았어요`
- 후보가 2개 이상이면 묶음 문구를 사용한다.
  - 예: `최근 다시 보지 않은 구독이 3개 있어요`
- 사용자가 `X`를 누르면 현재 배너 상태(signature)를 FE 로컬에 저장해 숨긴다.
- 사용량이 다시 감지되거나 후보 조합이 바뀌면 signature가 달라지므로 재노출 가능하다.

### 1차 구현 예시

- 구독 상세에서 월별 사용 기록 heatmap 표시
- 메인 구독 탭에서 7일 이상 미사용 리마인드 배너 1개 표시
- 인사이트 탭에서 `많이 본 구독 TOP 3`를 비교형 카드로 표시

### 구현 난이도

- 중

### 비고

- 잔디는 구독 카드 보조 정보가 아니라, 개별 구독 점검 정보로 보는 것이 더 맞다.
- 인사이트 탭은 전체 흐름을 예쁘게 보여주기보다, 유지 판단을 돕는 비교형 화면으로 전환한다.
- 여러 미사용 후보가 동시에 생겨도 배너는 1개만 보여주고, 단일 서비스 또는 묶음 문구로 처리한다.

## 5.6 추세 그래프 / 차트

### 목표

- 시간에 따라 사용량이 어떻게 변하고 있는지 보여준다.

예시:

- 최근 4주 사용량 감소
- 최근 3개월 사용량 증가

### 필요한 데이터

- `usage_date`
- `used_minutes`
- 기존 구독료

### 어떻게 구현할 것인가

- 일별 집계를 다시 주간/월간으로 묶어 시계열 차트를 만든다.
- 사용량과 구독료 해석 문구를 함께 붙인다.

### 1차 구현 예시

- 최근 4주 주간 사용시간 합
- 최근 3개월 월간 사용시간 합
- “결제는 유지되는데 사용량은 감소 중” 같은 해석 추가

### 구현 난이도

- 중

### 비고

- 그래프만 두지 말고 해석 문구를 같이 보여줘야 차별점이 생긴다.

## 6. 단계별 구현 순서

## 6.1 1단계: 공통 기반 만들기

- `subscription_usage_daily` 테이블 추가
- FE에서 최근 사용시간을 읽어 일별 집계 업로드하는 구조 설계

## 6.2 2단계: 빠르게 체감되는 기능

- 구독료 자극 멘트
- 앱 사용 시간 / 시간 대비 사용량
- 구독 연관 리포트 1차

## 6.3 3단계: 시각화 강화

- 잔디형 사용량 시각화
- 추세 그래프 / 차트

## 6.4 4단계: 해석 확장

- 구독 연관 리포트
- 목표 기반 동기부여

## 7. 우선순위 결론

가장 먼저 구현할 가치가 큰 조합은 아래 3개다.

- 구독료 자극 멘트
- 앱 사용 시간 / 시간 대비 사용량
- 구독 연관 리포트

이후 시각화 기능을 붙이면 설득력이 커진다.

- 잔디형 사용량 시각화
- 추세 그래프 / 차트

그 다음에 서비스의 해석력을 더 높이는 기능을 붙인다.

- 구독 연관 리포트
- 목표 기반 동기부여

## 8. 최종 결론

- 현재 체크인 이력만으로는 7개 기능 전체를 완성도 있게 구현하기 어렵다.
- 하지만 `subscription_usage_daily` 같은 최소 일별 사용량 집계 테이블이 있으면 7개 기능 모두 구현 가능 범위로 들어온다.
- 핵심은 raw 이벤트 전체 저장이 아니라, “하루에 서비스별 몇 분 썼는지”만 저장하는 최소 집계 구조다.
