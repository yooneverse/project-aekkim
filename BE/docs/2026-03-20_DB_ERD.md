# E106 DB ERD 정의서

## 도메인 구분

### 사용자 도메인

| 테이블 | 역할                                                                                                                |
| --- |-------------------------------------------------------------------------------------------------------------------|
| `users` | 사용자 정보. 소셜 로그인 계정을 provider / provider_user_id로 연결하고, 체크인·프로모션 알림 수신 여부와 선택 동의 여부를 관리한다. 금융망 연동 시 사용자별 finance_user_key를 저장한다. |

### 구독 도메인

| 테이블 | 역할 |
| --- | --- |
| `services` | OTT 서비스 마스터. `code`, `name`, `category`, `logo_url`을 보관한다. |
| `service_plans` | 서비스별 요금제 정보. Python 스크래퍼가 자동 수집·갱신한다. |
| `bundles` | 번들 마스터. 번들 코드, 이름, 요금제명, 현재가·정가, 로고를 관리한다. |
| `bundle_services` | 번들에 포함된 서비스 매핑. 하나의 번들이 어떤 서비스 조합인지 관리한다. |
| `subscriptions` | 사용자의 구독 현황. 단일 서비스 구독과 번들 구독을 함께 관리한다. |
| `checkin_records` | 월별 구독 사용 확인 기록. 같은 달 중복 체크인을 방지한다. |
| `subscription_usage_daily` | 사용자별 서비스 일별 사용량 집계. 최근 30일 사용량, 그래프, 패턴 분석의 공통 기반 데이터다. |
| `merchant_service_map` | 카드 가맹점명과 서비스 간 매핑 사전. `hit_count`를 기반으로 자동 매칭 신뢰도를 관리한다. |

### 프로모션 도메인

| 테이블 | 역할 |
| --- | --- |
| `promotions` | 프로모션 마스터. 일반 프로모션과 카드 혜택 중심의 가격/기간/출처 정보를 관리한다. |
| `promotion_services` | 프로모션과 서비스 간 N:N 매핑. 일반 프로모션과 카드 혜택 연결에 사용한다. |

### 알림 도메인

| 테이블 | 역할 |
| --- | --- |
| `notifications` | 알림 내역. 타입, 제목, 본문, 발송 시각, 읽음 시각을 관리한다. |
| `fcm_tokens` | 사용자 디바이스별 FCM 토큰. Push 발송 대상을 식별한다. |

## 테이블 정의

### 1. `users` - 사용자

| --- | --- | --- | -- | --- |

| `provider` | VARCHAR(20) | Y | DEFAULT `'GOOGLE'` | 소셜 로그인 제공자 |

| `last_login_at` | DATETIME | N | | 최근 로그인 시각 |
| `last_synced_at` | DATETIME | Y |  | 결제내역 증분 수집 기준이 되는 마지막 동기화 시각 |

### 6-1. `subscription_usage_daily` - 일별 서비스 사용량 집계

| 컬럼명 | 타입 | 필수 | 키/제약 | 설명 |
| --- | --- | --- | --- | --- |
| `usage_id` | BIGINT | Y | PK, AUTO_INCREMENT | 사용량 집계 ID |
| `user_id` | BIGINT | Y | FK -> `users.user_id` | 사용자 ID |
| `service_id` | BIGINT | Y | FK -> `services.service_id` | 서비스 ID |
| `usage_date` | DATE | Y |  | 사용일 |
| `used_minutes` | INT | Y |  | 해당 날짜의 총 사용 시간(분) |

추가 제약:
- UNIQUE(`user_id`, `service_id`, `usage_date`)

운영 규칙:
- FE가 Usage Stats를 읽어 하루 단위로 집계한 결과만 저장한다.
- raw 이벤트 전체를 저장하지 않고, 서비스별 일별 집계 1건만 저장한다.
- 최근 30일 사용량, 잔디형 시각화, 추세 그래프, 패턴 분석의 공통 기반 데이터로 사용한다.

### 7. `promotions` - 프로모션

| 컬럼명 | 타입 | 필수 | 키/제약 | 설명 |
| --- | --- | --- | --- | --- |
| `promotion_id` | BIGINT | Y | PK, AUTO_INCREMENT | 프로모션 ID |
| `promotion_type` | VARCHAR(20) | Y |  | `CARD_BENEFIT`, `PROMO`, `BUNDLE` |
| `title` | VARCHAR(150) | Y |  | 프로모션명 |
| `summary` | VARCHAR(500) | N |  | 요약. 가격, 조건, 혜택 등 |
| `original_price` | INT | N |  | 할인 전 정가. 번들/이벤트의 기준 가격 |
| `discount_price` | INT | N |  | 할인 후 적용 가격. 번들가, 프로모션 적용가 등 |
| `starts_at` | DATETIME | Y |  | 시작 시각 |
| `ends_at` | DATETIME | Y |  | 종료 시각. 상시 상품은 `2099-12-31` 사용 |
| `source_url` | VARCHAR(500) | N |  | 출처 또는 신청 URL |

참고:
- `service_id`는 이 테이블에 두지 않는다.
- 서비스 연결은 `promotion_services` 매핑 테이블로 관리한다.
- 현재 번들 상품 마스터는 `bundles`, `bundle_services`에서 관리한다.
- 추천 응답에서는 번들을 `promotionType = BUNDLE`로 표현할 수 있다.

프로모션 유형:

| `promotion_type` | 설명 | 예시 |
| --- | --- | --- |
| `CARD_BENEFIT` | 특정 카드 사용 시 OTT 구독료 할인 또는 적립 혜택 | JADE Classic - 넷플릭스 적립 50% |
| `PROMO` | 서비스 자체 할인 또는 이벤트 프로모션 | 티빙 광고형 스탠다드 첫 달 100원 |
| `BUNDLE` | 번들 추천 응답에 사용되는 타입 | 디즈니+ 넷플릭스 번들 추천 |

가격 필드 사용 규칙:

| `promotion_type` | `original_price` | `discount_price` | 설명 |
| --- | --- | --- | --- |
| `CARD_BENEFIT` | 미사용 가능 | 미사용 가능 | 정액/정률 할인, 캐시백, 적립은 `summary` 중심으로 표현한다. |
| `PROMO` | 사용 가능 | 사용 가능 | 정상가와 프로모션 적용가를 저장한다. |
| `BUNDLE` | 사용 가능 | 사용 가능 | 번들 추천 응답을 구성할 때 가격 정보를 표현하는 데 사용한다. |

활용:
- 일반 프로모션과 카드 혜택은 사용자가 구독 중인 서비스(`subscriptions`)를 기준으로 `promotion_services` 매핑을 조회한다.
- 번들 추천은 `bundles`, `bundle_services`를 기준으로 별도 계산하고, 응답에서만 `promotionType = BUNDLE`로 표현할 수 있다.

### 8. `promotion_services` - 프로모션 ↔ 서비스 매핑

| 컬럼명 | 타입 | 필수 | 키/제약 | 설명 |
| --- | --- | --- | --- | --- |
| `promotion_service_id` | BIGINT | Y | PK, AUTO_INCREMENT | 매핑 ID |
| `promotion_id` | BIGINT | Y | FK -> `promotions.promotion_id` | 프로모션 ID |
| `service_id` | BIGINT | Y | FK -> `services.service_id` | 서비스 ID |

추가 제약:
- UNIQUE(`promotion_id`, `service_id`)

참고:
- 단일 프로모션은 매핑 1건이다.
- 번들 상품 자체의 서비스 연결은 `bundle_services`에서 관리한다.
- `promotion_services`는 일반 프로모션과 카드 혜택 연결이 기본이며, 번들 상품의 서비스 연결 주체는 아니다.

프로모션 추천 로직:

```text
사용자 구독: 넷플릭스, 티빙
        |
        v
promotion_services에서 매핑 조회
        |
        v
관련 프로모션 전부 반환
  - CARD_BENEFIT: 현대카드 넷플릭스 캐시백
  - PROMO: 티빙 첫 달 100원
```

### 10. `merchant_service_map` - 공용 가맹점 ↔ 서비스 사전

| 컬럼명 | 타입 | 필수 | 키/제약 | 설명 |
| --- | --- | --- | --- | --- |
| `merchant_map_id` | BIGINT | Y | PK, AUTO_INCREMENT | 매핑 ID |
| `merchant_raw` | VARCHAR(160) | Y | UNIQUE | 가맹점명 원문 |
| `service_id` | BIGINT | Y | FK -> `services.service_id` | 매핑된 서비스 |
| `hit_count` | INT | Y | DEFAULT 1 | 이 매핑이 확인된 횟수 |
| `last_confirmed_at` | DATETIME | Y |  | 마지막 확인 시각 |

운영 규칙:
- `hit_count >= 3`이면 이후 같은 가맹점명은 자동 매칭한다.
- `hit_count < 3`이면 LLM이 서비스를 유추한 뒤 사용자 확인을 받고 카운트를 증가시킨다.
### 11. `payment_history` - 결제내역 저장소

| 컬럼명 | 타입 | 필수 | 키/제약 | 설명 |
| --- | --- | --- | --- | --- |
| `payment_history_id` | BIGINT | Y | PK, AUTO_INCREMENT | 결제내역 ID |
| `payment_id` | VARCHAR(100) | Y | UNIQUE | 결제 식별자 |
| `merchant_raw` | VARCHAR(255) | Y |  | 가맹점 원문명 |
| `amount` | INT | Y |  | 결제 금액 |
| `payment_date` | DATE | Y |  | 결제일 |
| `category` | VARCHAR(50) | N |  | 결제 카테고리 |

운영 규칙:
- `payment_history`는 특정 사용자에 종속되지 않는 공용 결제내역 저장소로 사용한다.
- 회원가입/로그인 전에도 결제내역 저장/조회 API에서 사용할 수 있다.
- 중복 저장 방지를 위해 `payment_id`에 UNIQUE 제약을 둔다.
