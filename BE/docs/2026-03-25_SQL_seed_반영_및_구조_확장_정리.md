# SQL Seed 반영 및 구조 확장 정리

## 목적

- 전달받은 SQL 데이터를 현재 E106 코드베이스에 반영할 수 있는 범위를 정리한다.
- 현재 구조에서 바로 가능한 것과, 코드 확장이 필요한 것을 구분한다.
- 각 이슈별로 원인, 해결 방법, 권장 우선순위를 함께 문서화한다.

## 현재 결론 요약

- `services`, `service_plans`, `promotions` 데이터는 현재 seed import 구조로 반영 가능하다.
- 다만 `MUSIC`, `AI` 카테고리는 아직 코드상 지원하지 않아 선행 수정이 필요하다.
- `BUNDLE`은 `services.category`로 넣는 구조가 아니라 별도 `bundles` 도메인으로 다뤄야 한다.
- 번들 데이터는 현재 `PromotionType.BUNDLE` seed에서 파생 생성되므로, SQL의 `bundles` 테이블을 그대로 1:1 반영하려면 seed 구조 확장이 필요하다.
- 다만 번들 할인 전 가격 표시를 위해 `bundles.originalPrice`는 코드/seed에 반영하는 방향으로 정리한다.
- `payment_history`, `subscriptions`는 현재 seed import 대상이 아니므로 별도 import 기능 또는 직접 SQL 반영이 필요하다.

## 현재 코드 상태

### 이미 구현된 것

- 서비스 seed import
  - `ServiceSeedItem`, `ServicePlanSeedItem` 기반으로 서비스/요금제 upsert 지원
- 프로모션 seed import
  - `PromotionSeedItem` 기반으로 프로모션 upsert 지원
  - `services[]` 배열을 통해 `promotion_services` 매핑 처리
- 번들 도메인
  - `Bundle`, `BundleServiceMap` 엔티티 존재
  - `SubscriptionType.BUNDLE` 생성/조회/수정 로직 존재
- FE 분석 파이프라인
  - 번들 힌트, 번들 가격 비교, 번들 겹침 사용량 기반 추론 로직 존재
- 연간 요금제
  - `billingCycle = YEARLY` 자체는 이미 지원

### 최근 반영된 수정

- `AdminSeedImportService`
  - `Promotion.billingCycle` import 누락 보정
  - Java `uppercase` 오용 수정

## 고쳐야 할 항목 정리

### 1. 서비스 카테고리 확장

#### 문제

- 현재 `ServiceCategory` enum이 `OTT` 하나만 지원한다.
- 따라서 SQL의 `MUSIC`, `AI` 서비스는 현재 seed import 시 바로 실패한다.

#### 원인

- 서비스 seed DTO가 문자열이 아니라 `ServiceCategory` enum을 직접 받는다.
- enum에 없는 값은 역직렬화 단계에서 실패한다.

#### 해결 방법

1. `ServiceCategory`에 `MUSIC`, `AI`를 추가한다.
2. `services.category` DB 컬럼이 enum 제약을 가진 환경이라면 DB enum도 함께 확장한다.
3. 카테고리 필터 API와 FE 카탈로그 사용처에서 신규 카테고리를 허용하는지 확인한다.

#### 수정 대상

- `BE/src/main/java/com/ssafy/e106/domain/subscription/enums/ServiceCategory.java`
- 카테고리 관련 DB 스키마 또는 migration
- 카테고리 필터를 소비하는 FE 화면/API 호출부

#### 우선순위

- 최우선

---

### 2. `BUNDLE`을 서비스 카테고리로 넣지 않도록 구조 정리

#### 문제

- 전달받은 SQL은 `services.category`에 `BUNDLE`을 추가하려는 형태다.
- 현재 코드베이스는 번들을 서비스의 한 종류가 아니라 별도 도메인으로 분리해 두었다.

#### 원인

- 백엔드에 이미 `bundles`, `bundle_services`, `SubscriptionType.BUNDLE` 구조가 존재한다.
- 같은 개념을 `services.category = BUNDLE`로도 넣으면 도메인이 중복된다.

#### 해결 방법

1. `BUNDLE`은 `services.category`에 추가하지 않는다.
2. 번들은 기존처럼 `Bundle`, `BundleServiceMap`으로 관리한다.
3. 서비스 카테고리 확장은 `OTT`, `MUSIC`, `AI`까지만 반영한다.

#### 권장 방침

- `services.category`: `OTT`, `MUSIC`, `AI`
- `bundles`: 번들 마스터 데이터
- `bundle_services`: 번들 포함 서비스 매핑

#### 우선순위

- 최우선

---

### 3. 번들 seed 구조를 독립 마스터 데이터 기준으로 확장

#### 문제

- 현재 번들은 `PromotionType.BUNDLE` seed를 import하면서 파생 생성된다.
- 따라서 SQL의 `bundles(code, name, plan_name, billing_cycle, monthly_price, original_price, logo_url)`를 그대로 반영할 수 없다.

#### 현재 한계

- 현재 구현은 bundle code를 title과 service 조합으로 유도한다.
- `name`, `planName`, `code`를 SQL처럼 명시적으로 통제하기 어렵다.
- 번들 프로모션과 번들 마스터 데이터를 분리하지 못한다.

#### 해결 방법

1. 번들 전용 seed payload를 추가한다.
2. 번들 seed는 아래 정보를 직접 받도록 한다.
   - `code`
   - `name`
   - `planName`
   - `billingCycle`
   - `monthlyPrice`
   - `originalPrice`
   - `logoUrl`
   - `services[]`
   - 선택: `aliases`, `knownMerchantPatterns`, `priceTolerance`
3. `AdminSeedImportService`에 `importBundles()`를 추가한다.
4. 프로모션 import와 번들 마스터 import를 분리한다.

#### 권장 DTO 예시

- `BundleSeedPayload`
- `BundleSeedItem`
- `BundleSeedServiceItem`

#### 기대 효과

- SQL의 `bundles`, `bundle_services`를 JSON으로 거의 1:1 변환 가능
- 번들 마스터와 번들 프로모션을 분리 관리 가능
- FE 분석/확정 흐름에서 안정적으로 `bundleCode`를 사용할 수 있음

#### 우선순위

- 높음

---

### 4. 서비스/요금제 seed 변환 규칙 명확화

#### 문제

- SQL의 `services`, `service_plans`는 현재 JSON import 구조로 옮길 수 있지만 변환 규칙이 명시돼 있지 않다.

#### 해결 방법

1. `services` SQL은 `SERVICE_CATALOG.json`의 `services[]`로 변환한다.
2. `service_plans` SQL은 `SERVICE_CATALOG.json`의 `servicePlans[]`로 변환한다.
3. `service_id` 조인은 seed에서 `serviceCode`로 치환한다.
4. `YEARLY` 플랜은 현재 구조에 맞게 `monthlyPrice = 월 환산값`으로 유지한다.

#### 주의사항

- `yearlyPrice` 별도 컬럼은 현재 없다.
- 지금 구조에서는 `billingCycle = YEARLY`와 `monthlyPrice` 조합이 표준이다.
- UI도 이미 연간 요금제를 `월 환산` 기준으로 보여주고 있다.

#### 우선순위

- 높음

---

### 5. 연간 요금제 가격 의미를 문서와 코드에서 명확히 고정

#### 문제

- `service_plans.monthly_price`에 연간 총액을 넣을지, 월 환산값을 넣을지 혼동 가능성이 있다.
- 현재 seed는 이미 연간 요금제를 월 환산값으로 넣고 있다.

#### 해결 방법

1. `monthlyPrice`의 의미를 문서로 명확히 고정한다.
2. 규칙은 아래처럼 통일한다.
   - `MONTHLY`: 실제 월 결제 금액
   - `YEARLY`: 연간 총액이 아닌 월 환산 금액
3. 관리자/크롤링/seed 생성 쪽에서 같은 규칙을 강제한다.

#### 대안

- 만약 연간 총액도 반드시 보여야 한다면 추후 별도 컬럼 추가를 검토한다.
  - 예: `displayPrice`, `annualPrice`
- 단, 현재 MVP 단계에서는 컬럼 추가보다 규칙 통일이 더 현실적이다.

#### 우선순위

- 높음

### 6. 프로모션 데이터는 `promotions + promotion_services` SQL을 하나의 JSON으로 합치기

#### 문제

- 현재 import 구조는 `promotion_services` SQL을 따로 받지 않는다.
- 서비스 매핑은 promotion JSON 내부의 `services[]` 배열로 처리한다.

#### 해결 방법

1. `promotions` SQL 레코드를 `PromotionSeedItem`으로 변환한다.
2. `promotion_services` SQL은 promotion title 기준으로 매칭하지 말고, 변환 시 같은 JSON 객체 안의 `services[]`로 합친다.
3. 카드 혜택, 일반 프로모션, 번들 프로모션 모두 동일하게 처리한다.

#### 주의사항

- title 기반 조인은 데이터가 바뀌면 깨지기 쉽다.
- 변환 스크립트를 만들 경우 `serviceCode`를 명시적으로 넣는 방식이 안전하다.

#### 우선순위

- 높음

### 7. `payment_history`와 `subscriptions`는 seed import 대상에서 분리

#### 문제

- 전달받은 SQL에는 `payment_history`, `subscriptions`가 포함되어 있다.
- 현재 admin seed import 구조에는 이 데이터를 받는 DTO와 서비스가 없다.

#### 해결 방법

선택지는 두 가지다.

1. 단기 대응
   - 이 데이터는 seed JSON로 억지로 넣지 말고 DB SQL로 직접 반영한다.
2. 구조 확장
   - 전용 import 기능을 추가한다.
   - 예:
     - `PaymentHistorySeedPayload`
     - `SubscriptionSeedPayload`
     - `AdminSeedImportService#importPaymentHistory`
     - `AdminSeedImportService#importSubscriptions`

#### 권장 판단

- 현재 범위에서는 direct SQL 또는 별도 관리 스크립트가 더 현실적이다.
- seed import에까지 넣으려면 데이터 정합성 검증 로직이 많이 필요하다.

#### 우선순위

- 중간

### 8. 번들 분석 품질을 위해 alias/merchant pattern 데이터 보강

#### 문제

- FE 분석기는 이미 번들 추론을 지원하지만, 실제 성능은 catalog의 alias와 merchant pattern 품질에 크게 의존한다.
- AI/MUSIC 서비스와 신규 번들은 merchant 표기가 다양해 오탐/누락이 생길 수 있다.

#### 해결 방법

1. 서비스 seed와 번들 seed에 alias/merchant pattern 필드를 추가한다.
2. 최소한 아래 종류를 관리한다.
   - 결제사 표기
   - 영문 표기
   - 앱명/브랜드명
   - 카드 명세서 축약 표기
3. FE `ServiceCatalog` 로딩 데이터에도 해당 필드를 내려준다.

#### 관련 코드

- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/model/ServiceCatalog.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/SubscriptionCandidateExtractor.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/RuleBasedCandidateResolver.kt`

#### 우선순위

- 중간

### 9. SQL 변환 파이프라인을 수작업이 아닌 스크립트로 고정

#### 문제

- 지금처럼 SQL 덩어리를 사람이 직접 JSON으로 옮기면 누락과 불일치가 생기기 쉽다.

#### 해결 방법

1. SQL 원본에서 아래 산출물을 자동 생성하는 변환 스크립트를 둔다.
   - `SERVICE_CATALOG.json`
   - `promotions.json`
   - 향후 `bundles.json`
2. 변환 규칙을 고정한다.
   - `service_id` → `serviceCode`
   - `promotion_services` → `services[]`
   - `bundle_services` → `services[]`
3. 사람이 마지막 검수만 하도록 바꾼다.

#### 우선순위

- 중간

### 10. 검증 시나리오 추가

#### 문제

- 카테고리 확장, 번들 독립 seed, 연간 요금제, AI/MUSIC 서비스가 들어오면 검증 범위가 커진다.

#### 해결 방법

아래 검증을 추가한다.

1. 서비스 seed import 성공
   - `OTT`, `MUSIC`, `AI`
2. 연간 요금제 조회 성공
   - `YEARLY` 플랜 노출
3. 번들 seed import 성공
   - `Bundle`, `BundleServiceMap` 생성
4. 프로모션 import 성공
   - `Promotion`, `PromotionServiceMapping` 생성
5. 구독 생성/조회 검증
   - `SINGLE`
   - `BUNDLE`
6. FE 분석 검증
   - Spotify, ChatGPT, 유독, 번들 merchant 표기 인식

#### 우선순위

- 높음

## 권장 구현 순서

1. `ServiceCategory`에 `MUSIC`, `AI` 추가
2. `services/service_plans/promotions` SQL을 현재 JSON 포맷으로 변환
3. 번들 전용 seed 구조 추가
4. `bundles/bundle_services` SQL을 신규 bundle seed로 반영
5. 필요 시 `payment_history`, `subscriptions` 전용 import 기능 추가
6. 테스트 및 검증 시나리오 보강

## 실무 적용 방침

### 지금 바로 가능한 것

- `services`
- `service_plans`
- `promotions`

단, `ServiceCategory` 확장 후 진행

### 코드 확장 후 가능한 것

- `bundles`
- `bundle_services`

### 별도 범위로 분리할 것

- `payment_history`
- `subscriptions`
- FK 삭제 같은 DDL

## 최종 정리

- 현재 구조는 이미 서비스, 요금제, 프로모션, 번들 구독, 번들 추론까지 꽤 많이 구현돼 있다.
- 가장 먼저 막히는 부분은 `ServiceCategory`가 `OTT`만 지원한다는 점이다.
- 가장 크게 손봐야 하는 부분은 번들 마스터 데이터를 프로모션 파생 생성이 아니라 독립 seed로 관리하도록 바꾸는 것이다.
- `payment_history`, `subscriptions`는 현재 seed import와 별도 범위로 다루는 것이 맞다.
