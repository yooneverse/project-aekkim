# 번들 추천 구조 전환 구현 계획

## 1. 목적

- 현재 추천 API에서 번들 추천이 노출되지 않는 원인을 정리한다.
- 기존 `promotions` 기반 번들 추천 구조를 `bundles` 기반 구조로 전환하기 위한 구현 계획을 정리한다.

## 2. 현재 문제

현재 번들 데이터는 다음 구조로 분리되어 있다.

- 번들 마스터: `bundles`
- 번들 포함 서비스: `bundle_services`

하지만 추천 API는 아직 아래 구조를 기준으로 동작한다.

- 추천 조회 대상: `promotions`
- 번들 추천 조건: `promotions.promotion_type = BUNDLE`
- 서비스 연결 기준: `promotion_services`

즉 현재 상태는 다음과 같다.

1. 번들 seed는 `bundle.json`으로 `bundles`에 적재된다.
2. `promotions.json`에서는 `BUNDLE` 타입을 제거했다.
3. 추천 API는 여전히 `PromotionType.BUNDLE`만 조회한다.
4. 결과적으로 번들 추천 결과가 0건이 된다.

## 3. 원인 요약

문제의 핵심은 도메인 구조와 추천 조회 경로가 서로 다르다는 점이다.

- 데이터 저장 구조: `bundles`
- 추천 조회 구조: `promotions`

즉 번들을 독립 도메인으로 분리한 뒤에도 추천 로직은 이전 구조에 남아 있어, 새 번들 데이터를 읽지 못한다.

## 4. 목표 구조

추천 API는 아래 3축을 기준으로 동작해야 한다.

- 번들 추천: `bundles`, `bundle_services`
- 일반 프로모션 추천: `promotions` 중 `PROMO`
- 카드 혜택 추천: `promotions` 중 `CARD_BENEFIT`

즉 번들 추천만큼은 더 이상 `promotions.promotion_type = BUNDLE`에 의존하지 않는다.

## 5. 구현 원칙

### 5.1 응답 스키마는 최대한 유지

FE는 이미 추천 응답에서 아래 구조를 사용하고 있다.

- `bundles`
- `promotions`
- `cardBenefits`

따라서 가능하면 API 응답 스키마는 그대로 유지하고, 내부 생성 경로만 바꾸는 것이 안전하다.

### 5.2 번들 추천은 별도 계산 경로로 분리

현재 `PromotionService` 안에서 번들, 프로모션, 카드 혜택을 모두 `getTopRecommendationsByType(...)`로 처리하고 있다.

이 구조는 프로모션 엔티티 기반이기 때문에 번들 추천에 그대로 사용할 수 없다.

따라서 번들 추천은 아래처럼 별도 경로로 분리한다.

- 번들 추천: `getTopBundleRecommendations(...)`
- 프로모션 추천: `getTopRecommendationsByType(..., PROMO, ...)`
- 카드 혜택 추천: `getTopRecommendationsByType(..., CARD_BENEFIT, ...)`

### 6.2 번들 추천 평가 모델 추가

현재 추천 평가는 `PromotionRecommendationEvaluation` 하나로 통일되어 있다.

번들 추천은 프로모션과 평가 기준이 다르기 때문에 선택지가 2개 있다.

1. 기존 `PromotionRecommendationEvaluation`을 재사용
2. 번들 전용 평가 모델을 분리

권장:

- 점수와 이유 라벨 구조는 같으므로, 초반에는 기존 형태와 유사한 전용 모델을 추가해도 무방하다.
- 이름 예시:
  - `BundleRecommendationEvaluation`

필수 평가 요소:

- 포함 서비스 overlap 수
- 현재 사용 중인 서비스와의 관련성
- 번들 가격 절감액
- 현재 구독 중복 여부

### 6.3 번들 추천 응답 매핑 추가

대상 파일:

현재는 `Promotion` 엔티티만 받아 응답을 생성한다.

번들 추천도 같은 응답 타입으로 내려주려면 `Bundle` 기반 정적 팩토리 메서드를 추가해야 한다.

예시:

매핑 규칙:

- `startsAt`, `endsAt`는 번들에는 없으므로 별도 정책 필요

### 6.4 번들 추천 기간값 처리 정책

현재 추천 응답은 `startsAt`, `endsAt`를 포함한다.

번들은 프로모션이 아니라 상시 카탈로그이므로 기간 정보가 없다.

선택지:

1. `startsAt`, `endsAt`를 null 허용으로 변경
2. 상시 추천용 더미 기간값 사용

권장:

- DTO는 그대로 두되 번들 응답에서는 null 허용으로 맞추는 것이 가장 자연스럽다.
- FE에서 기간 null일 때도 정상 동작하는지 함께 확인해야 한다.

### 6.6 추천 점수 계산 규칙

초기 규칙은 단순하고 설명 가능해야 한다.

권장 기준:

1. 번들 포함 서비스 중 현재 사용자와 겹치는 서비스 수가 많을수록 가산점
2. 번들 절감액(`originalPrice - monthlyPrice`)이 클수록 가산점
3. 이미 동일 번들을 구독 중이면 제외
4. 포함 서비스가 사용 중 카테고리와 전혀 겹치지 않으면 제외 또는 감점

추천 이유 라벨 예시:

- `지금 이용 중인 서비스를 묶어 절약할 수 있어요`
- `현재 구독 조합보다 더 저렴해요`
- `사용 중인 서비스 2개 이상이 포함돼 있어요`

## 9. 예상 리스크

### 9.1 추천 상세와의 연결 문제

번들 추천은 현재 프로모션 상세 API와 1:1로 연결되지 않는다.

따라서 초기 구현에서는 다음 중 하나를 정해야 한다.

- 번들 추천은 상세 진입 없이 목록 노출만 우선 지원
- 번들 추천 전용 상세 API를 별도로 만든다
- 기존 프로모션 상세 화면을 번들용으로도 확장한다

### 9.2 점수 계산 중복

프로모션 추천과 번들 추천의 평가 구조가 비슷해 보여도 기준은 다르다.

- 프로모션: 기간, 할인 이벤트, 카드 혜택 중심
- 번들: 서비스 조합, 중복 구독, 절감액 중심

무리하게 공통화하면 오히려 코드가 복잡해질 수 있다.

## 10. 결론

현재 번들 추천이 나오지 않는 이유는 번들 데이터가 `bundles`로 분리된 뒤에도 추천 로직이 여전히 `PromotionType.BUNDLE` 프로모션만 조회하고 있기 때문이다.

따라서 수정 방향은 다음으로 확정한다.

- 번들 추천은 `promotions`가 아니라 `bundles`에서 조회한다.
- 추천 API 응답 구조는 최대한 유지한다.
- 번들 추천만 별도 계산 경로와 별도 매핑 경로로 분리한다.
- 이후 FE에서 번들 추천 노출과 상세 동작을 함께 점검한다.
