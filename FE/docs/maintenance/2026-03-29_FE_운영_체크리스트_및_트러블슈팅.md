# FE 운영 체크리스트 및 트러블슈팅

> 작성일: 2026-03-29
> 목적: 실행 전 점검, 변경 후 회귀 확인, 자주 발생하는 장애 대응 기준 정리

## 1. 실행 전 체크리스트

- [ ] `fe/local.properties`에 `sdk.dir`가 올바르게 들어가 있다.
- [ ] `BASE_URL_DEV`, `BASE_URL_EMULATOR`, `BASE_URL_DEVICE`, `BASE_URL_PROD`가 현재 환경과 맞다.
- [ ] `fe/app/google-services.json`이 준비되어 있다.
- [ ] Google 로그인 테스트용 계정이 단말 또는 에뮬레이터에 로그인되어 있다.
- [ ] 필요 시 `KAKAO_NATIVE_APP_KEY`가 채워져 있다.
- [ ] 실기기 테스트라면 단말과 서버가 같은 네트워크에 있거나 접근 가능한 주소를 사용한다.

## 2. 권장 확인 명령

```bash
.\gradlew.bat :app:assembleDebug
.\gradlew.bat :app:lintDebug
.\gradlew.bat :app:testDebugUnitTest
```

메모

- 테스트 task는 정의되어 있지만 실제 자동화 테스트 커버리지는 낮다.
- 그래서 `assembleDebug` + 수동 회귀가 사실상 기본 점검 절차다.

## 3. 변경 후 스모크 테스트

### 인증/온보딩

- [ ] 앱 첫 실행 시 온보딩 또는 세션 복구 진입이 올바르다.
- [ ] Google 로그인 성공 후 약관 화면으로 이동한다.
- [ ] 로그아웃 후 재실행 시 세션이 남지 않는다.

### 권한/분석

- [ ] 알림 권한 다이얼로그와 UsageStats 권한 흐름이 막히지 않는다.
- [ ] 분석 완료 후 구독 확인 또는 대시보드로 흐름이 이어진다.

### 메인 플로우

- [ ] 대시보드 로딩/에러/Empty가 모두 정상 표시된다.
- [ ] 수동 추가, 상세 진입, 편집/삭제 흐름이 깨지지 않는다.
- [ ] 체크인 딥링크와 해지 지원 진입이 정상이다.

### 프로모션/알림/마이페이지

- [ ] 프로모션 목록/상세 진입이 정상이다.
- [ ] 알림 탭 후 읽음 처리와 딥링크 이동이 정상이다.
- [ ] 마이페이지 토글, 로그아웃, 회원탈퇴 진입이 정상이다.

## 4. 장애별 확인 포인트

### 4-1. 앱은 뜨지만 API가 전부 실패한다

우선 확인

- `fe/local.properties`의 base URL 값
- 에뮬레이터인지 실기기인지에 따라 `BaseUrlResolver.kt`가 어떤 값을 고르는지
- 백엔드 서버가 실제로 해당 주소에서 열려 있는지

추가 메모

- 네트워크 기준 URL은 `currentBaseUrl()`에서 결정된다.
- 디버그에서는 에뮬레이터면 `BASE_URL_EMULATOR`, 실기기면 `BASE_URL_DEVICE`를 우선 사용한다.

### 4-2. 실기기에서만 서버 연결이 안 된다

우선 확인

- `BASE_URL_DEVICE`가 현재 PC IP 또는 접근 가능한 서버 주소인지
- 실기기와 서버가 같은 네트워크에 있는지
- 로컬 핫스팟/사내망 변경으로 IP가 바뀌지 않았는지

### 4-3. Google 로그인이 실패한다

우선 확인

- `fe/app/google-services.json` 존재 여부
- `GOOGLE_WEB_CLIENT_ID` 또는 `default_web_client_id` 설정 여부
- debug keystore SHA가 Firebase/Google 설정과 맞는지
- 테스트 단말에 Google 계정이 실제 로그인되어 있는지

### 4-4. Kakao 로그인이 반응하지 않거나 초기화되지 않는다

우선 확인

- `KAKAO_NATIVE_APP_KEY` 값이 비어 있지 않은지
- placeholder 값이 들어가 있지 않은지
- Manifest placeholder로 만들어지는 `kakao{nativeKey}` scheme가 맞는지

추가 메모

- `App.kt`는 키가 비어 있으면 Kakao SDK 초기화를 건너뛴다.

### 4-5. 알림이 오지 않거나 눌러도 화면이 이상하다

우선 확인

- FCM 토큰 등록이 호출됐는지
- payload에 `type`, `referenceId`가 정확히 들어오는지
- `type` 값이 `CHECKIN`, `PROMO`, `CHURN_REVIEW` 중 하나인지
- 딥링크가 `aekkim://checkin/{id}`, `aekkim://subscriptions/{id}`, `aekkim://promotions/{id}` 규칙을 따르는지

### 4-6. 로그아웃 이후 세션이 남거나 토큰 상태가 이상하다

우선 확인

- `TokenDataStore`가 에뮬레이터에서 plain `SharedPreferences` fallback을 쓰는지
- 앱 데이터 초기화 후 재현되는지
- 토큰 refresh 흐름과 `AuthInterceptor`, `TokenAuthenticator` 로그

추가 메모

- 디버그 에뮬레이터에서는 암호화 저장소 대신 plain prefs를 쓸 수 있다.
- 로컬 세션 문제 재현 시 앱 데이터 삭제 후 다시 보는 것이 빠르다.

### 4-7. UsageStats 권한 허용 이후에도 분석이 안 된다

우선 확인

- 단말 제조사별 설정 화면에서 권한이 실제 부여됐는지
- 권한 확인 로직이 `UsagePermissionChecker` 기준으로 true가 되는지
- 분석 자산 파일이 누락되지 않았는지

### 4-8. AI 분석 또는 모델 로딩에서 예외가 난다

우선 확인

- `fe/app/src/main/assets/ml`에 아래 4개 파일이 있는지
  - `ft_model.bin`
  - `xgb_model_android.json`
  - `feature_map.json`
  - `label_map.json`
- 앱 실행 후 `filesDir/models` 복사가 정상 완료됐는지

## 5. 회귀 범위가 넓은 변경

아래를 건드렸다면 최소 스모크 테스트가 아니라 전체 플로우 회귀를 권장한다.

- `app/navigation/*`
- `data/auth/*`, `data/interceptor/*`, `data/local/TokenDataStore.kt`
- `feature/auth/*`
- `feature/dashboard/*`, `feature/checkin/*`, `feature/promotion/*`
- `NotificationRepository`, `AekkimFirebaseMessagingService`
- `Theme.kt`, `Type.kt`, 공용 컴포넌트

## 6. 유지보수 시 남은 리스크

- 자동화 테스트가 부족해 수동 회귀 의존도가 높다.
- 릴리즈 서명과 실운영 배포 절차가 `fe/docs`에 완전히 문서화되어 있지 않다.
- Kakao/Naver 로그인 정책은 최종 제품 기준으로 다시 확인이 필요하다.
- Firebase Analytics는 의존성은 있지만 운영 이벤트 기준이 명확하지 않다.
