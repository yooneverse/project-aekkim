# 바이브코딩 프로젝트 구조 설계서

## 1) 목표
- 앱개발 초심자가 "빠르게 만들고, 나중에 망가지지 않게" 확장할 수 있는 기본 구조를 제안합니다.
- 이 문서는 Android Compose 기준으로 작성했지만, 핵심 분리 원칙(UI/도메인/데이터)은 다른 프론트엔드에도 그대로 적용 가능합니다.

## 2) 설계 원칙 (바이브코딩용)
- **작게 시작**: 기능 1개를 끝까지(화면+상태+API+예외) 완성한 뒤 확장.
- **파일 규칙 고정**: 기능마다 동일한 뼈대 사용(팀원/AI가 길을 잃지 않음).
- **상태 단방향**: UI는 상태를 그리기만 하고, 변경은 ViewModel로만.
- **외부 의존성 격리**: 네트워크/DB/블루투스/AI는 `data` 계층에서만 다룸.
- **오프라인 대비 기본 탑재**: 최소한 read 기능은 fallback 데이터 제공.

## 4) 기능(feature) 단위 파일 템플릿

각 기능 폴더에 아래 4개를 기본으로 둡니다.

1. `XxxScreen.kt`
- Compose UI만 포함
- `collectAsState()`로 상태 구독
- 사용자 액션을 `viewModel.onIntent(...)`로 전달

2. `XxxViewModel.kt`
- `UiState(StateFlow)` + `UiEvent(SharedFlow)` 관리
- 비즈니스 로직과 use-case 호출

3. `XxxState.kt`
- 화면에서 필요한 상태만 데이터 클래스로 정의

4. `XxxIntent.kt`
- 버튼 클릭/입력/재시도 등 사용자 액션을 sealed class로 정의

## 5) 추천 데이터 흐름

```text
Screen -> ViewModel(Intent) -> Repository -> (Remote API / Local DB / Manager)
       <-      UiState/UiEvent  <-
```

핵심 규칙:
- Screen에서 Retrofit/Room/Bluetooth 직접 호출 금지
- ViewModel에서 Android Context 직접 의존 최소화
- Repository에서 온라인/오프라인 소스 선택

## 6) 네비게이션 분리 전략
- `MainActivity`는 `AppNavHost`만 호출.
- 라우트 등록은 기능별 NavGraph 파일로 분리.
- 결과: 파일 충돌 감소, 신규 기능 추가 속도 증가.

예시:
- `navigation/AuthNavGraph.kt`
- `navigation/GestureNavGraph.kt`
- `navigation/SettingsNavGraph.kt`

## 7) 바이브코딩 운영 규칙 (실전)

### 7-1. 개발 사이클
1. "화면 1개" 스켈레톤 생성
2. ViewModel 상태 연결
3. Repository API 1개 연결
4. 로딩/에러/빈 상태 처리
5. 저장/수정/삭제 확장

### 7-2. PR/커밋 단위
- 하나의 사용자 시나리오 기준으로 잘게 쪼갬.
- 예: `로그인 성공 흐름 완성` / `오프라인 목록 조회 fallback`.

### 7-3. 파일 생성 규칙
- 새 기능 만들 때 같은 폴더 구조 복붙 후 이름만 변경.
- ViewModel에서 먼저 상태 골격을 만들고 UI를 붙이는 순서 권장.

## 8) AI(바이브코딩) 프롬프트 템플릿

아래 템플릿으로 요청하면 결과 품질이 안정적입니다.

```text
[목표]
gesture 상세 화면에 '매핑 삭제' 기능 추가

[제약]
- Compose + MVVM 패턴 유지
- Screen은 UI만, 로직은 ViewModel/Repository로 분리
- 로딩/성공/실패 상태 모두 처리

[수정 범위]
- feature/gesture/view/GestureDetailScreen.kt
- feature/gesture/viewmodel/GestureViewModel.kt
- data/repository/GestureRepository.kt

[완료 조건]
- 삭제 성공 시 목록 즉시 갱신
- 실패 시 Snackbar 이벤트 노출
- 기존 동작 회귀 없음
```

## 9) 최소 품질 게이트
- Lint/타입체크 통과
- 바뀐 ViewModel 단위테스트 1개 이상
- 로딩/에러/빈 상태 스크린샷(또는 동작 확인 로그)
- 네비게이션 뒤로가기 동작 확인

## 10) 이 구조를 바로 적용하는 순서
1. 현재 프로젝트에서 `navigation` 분리부터 적용.
2. 새 기능은 무조건 `feature/{name}` 템플릿으로 시작.
3. Repository에서 온라인/오프라인 분기 공통 함수화.
4. 마지막으로 테스트/품질 게이트 자동화.

---

이 설계의 핵심은 "속도"와 "유지보수"의 균형입니다.
처음엔 템플릿처럼 보여도, 팀 규모가 커질수록 개발 속도와 안정성 차이가 크게 납니다.
