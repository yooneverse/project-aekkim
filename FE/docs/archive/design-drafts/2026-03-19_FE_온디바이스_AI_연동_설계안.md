# FE 온디바이스 AI 연동 설계안
> 작성일: 2026-03-19  
> 목적: 현재 FE 구조 기준으로 온디바이스 AI를 어디에 붙여야 하는지 파일 단위로 정리한다.  
> 상태: 설계 문서 only, 코드 변경 없음

## 1. 결론

- 온디바이스 AI의 실행 주체는 `FE(Android 앱)`이다.
- 현재 구조에서는 `feature/analysis`가 AI 진입점이므로, 온디바이스 추론은 `AnalysisViewModel`을 중심으로 붙이는 것이 맞다.
- 금융망 결제내역 원본은 `BE -> FE`로 전달받고, FE는 `UsageStats + 결제내역`을 합쳐 기기 내에서 추론한 뒤, 결과만 `BE`로 업로드해야 한다.
- 따라서 FE 기준 책임 분리는 아래가 맞다.
  - `feature/analysis`: 추론 오케스트레이션, 상태 관리, UI
  - `feature/analysis/ondevice`: 모델 로딩, 전처리, 추론, 후처리
  - `data/api`, `data/dto`, `data/repository`: bootstrap 조회 / 결과 업로드
  - `di`: Retrofit, 추론 엔진, 모델 provider 주입

## 2. 현재 FE 구조 기준 삽입 지점

현재 이미 존재하는 분석 관련 파일:

- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/UsagePermissionChecker.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/UsageStatsCollector.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/viewmodel/AnalysisViewModel.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/view/AnalysisLoadingScreen.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/analysis/model/AnalysisContract.kt`
- `FE/app/src/main/java/com/ssafy/e106/app/navigation/AuthNavGraph.kt`

현재 이 구조는 `UsageStats 권한 확인 -> 로딩 UI -> SubscriptionConfirm 이동`까지만 담당한다.  
즉 AI를 새로 꽂아야 하는 자리는 이미 정해져 있고, `AnalysisViewModel`이 그 중심이 된다.

## 3. 권장 데이터 흐름

1. `AnalysisLoadingRoute` 진입
2. `AnalysisViewModel`이 UsageStats 권한 확인
3. `AnalysisRepository`가 BE에서 결제내역 bootstrap 세션 조회
4. `UsageStatsCollector`가 OTT 사용 기록 수집
5. `OnDeviceSubscriptionAnalysisEngine`이 기기 내 추론 수행
6. `AnalysisRepository`가 추론 결과를 BE에 업로드
7. FE는 결과 요약 상태를 보여주고 `SubscriptionConfirm`으로 이동

핵심은 `AI 추론은 FE`, `원본 데이터 공급과 DB 반영은 BE`라는 점이다.

## 4. 파일 단위 설계

### 4-1. 수정 대상

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/model/AnalysisContract.kt`
- 역할:
  - 현재 단순 progress 상태에 더해 bootstrap/추론/업로드 상태를 표현한다.
- 추가할 항목:
  - `analysisError`
  - `detectedSubscriptionCount`
  - `analysisSourceLabel`
  - `RetryClicked`
- 이유:
  - 온디바이스 추론은 실패 지점이 여러 개이므로, 기존 `loading -> done`만으로는 상태를 표현할 수 없다.

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/model/TrackedOttPackages.kt`
- 역할:
  - package name 집합만 두지 말고 `packageName -> serviceName` 매핑까지 가지게 한다.
- 이유:
  - 모델 입력과 결과 후처리에서 서비스명을 일관되게 사용해야 한다.

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/viewmodel/AnalysisViewModel.kt`
- 역할:
  - 현재 mock 단계 전환 로직을 버리고 실제 분석 파이프라인 orchestration을 담당한다.
- 넣어야 할 책임:
  - 권한 확인
  - bootstrap 세션 조회
  - UsageStats 수집
  - 온디바이스 추론 호출
  - 업로드 호출
  - 성공/부분성공/실패/재시도 상태 분기
- 주입 대상:
  - `UsagePermissionChecker`
  - `UsageStatsCollector`
  - `AnalysisRepository`
  - `OnDeviceSubscriptionAnalysisEngine`

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/view/AnalysisLoadingScreen.kt`
- 역할:
  - 기존 3단계 mock 카피를 실제 단계 UX로 맞춘다.
- 반영할 상태:
  - bootstrap 중
  - 기기 내 AI 추론 중
  - 결과 업로드 중
  - 실패 + 재시도
  - 성공 후 confirm 이동
- 이유:
  - 온디바이스 AI는 “기기에서 분석 중”이라는 메시지를 화면에 명확히 드러내야 한다.

#### `FE/app/src/main/java/com/ssafy/e106/app/navigation/AuthNavGraph.kt`
- 역할:
  - 현행 route 구조는 유지하되 `AnalysisLoading -> SubscriptionConfirm` 흐름을 분석 완료 기준으로만 이동시키도록 유지한다.
- 이유:
  - AI를 별도 route로 빼지 않아도 현 구조에서 충분히 수용 가능하다.

#### `FE/app/src/main/java/com/ssafy/e106/di/NetworkModule.kt`
- 역할:
  - 분석 bootstrap / 결과 업로드용 API provider 등록
- 이유:
  - 현재 NetworkModule에 `AnalysisApi`가 없으므로 Repository를 주입할 수 없다.

### 4-2. 신규 생성 대상

#### `FE/app/src/main/java/com/ssafy/e106/data/api/AnalysisApi.kt`
- 역할:
  - 분석 관련 서버 통신 endpoint 정의
- 권장 API:
  - `GET /api/v1/analysis/bootstrap`
  - `POST /api/v1/analysis/results`
- 이유:
  - `SubscriptionApi`에 섞지 말고, 분석 lifecycle을 별도 API로 분리하는 편이 책임이 명확하다.

#### `FE/app/src/main/java/com/ssafy/e106/data/dto/analysis/AnalysisBootstrapResponse.kt`
- 역할:
  - BE가 내려주는 분석 세션 + 결제내역 원본 DTO

#### `FE/app/src/main/java/com/ssafy/e106/data/dto/analysis/AnalysisResultUploadRequest.kt`
- 역할:
  - FE의 온디바이스 추론 결과 업로드 요청 DTO

#### `FE/app/src/main/java/com/ssafy/e106/data/dto/analysis/AnalysisResultUploadResponse.kt`
- 역할:
  - 업로드 완료 응답 DTO

#### `FE/app/src/main/java/com/ssafy/e106/data/repository/AnalysisRepository.kt`
- 역할:
  - bootstrap 조회 / 결과 업로드 / FE용 domain mapping
- 넣어야 할 책임:
  - `AnalysisApi` 호출
  - DTO -> domain 변환
  - 에러 파싱
  - mock fallback 여부 정책
- 이유:
  - 현재 `SubscriptionRepository`는 체크인/구독 관리가 중심이라 분석 lifecycle을 여기에 섞는 것은 경계가 흐려진다.

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/ondevice/OnDeviceSubscriptionAnalysisEngine.kt`
- 역할:
  - 온디바이스 추론 인터페이스
- 이유:
  - ViewModel이 TensorFlow Lite, ML Kit, ONNX Runtime 구현 세부를 직접 알면 안 된다.

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/ondevice/TfLiteSubscriptionAnalysisEngine.kt`
- 역할:
  - 실제 모델 로딩 및 추론 실행
- 권장 책임:
  - 입력 feature 생성
  - 모델 inference
  - raw output 후처리

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/ondevice/OnDeviceAnalysisInputMapper.kt`
- 역할:
  - 결제내역 DTO + UsageStats를 모델 입력 feature로 변환
- 이유:
  - 전처리 로직을 engine 내부에 전부 넣으면 테스트가 어려워진다.

#### `FE/app/src/main/java/com/ssafy/e106/feature/analysis/ondevice/OnDeviceAnalysisOutputMapper.kt`
- 역할:
  - 모델 출력값을 FE domain / 업로드 payload로 변환

#### `FE/app/src/main/java/com/ssafy/e106/di/AnalysisModule.kt`
- 역할:
  - `OnDeviceSubscriptionAnalysisEngine` 구현체 바인딩
- 이유:
  - ViewModel이 구현체에 직접 결합되지 않도록 한다.

#### `FE/app/src/main/assets/ml/subscription_detector.tflite`
- 역할:
  - 온디바이스 모델 파일
- 이유:
  - 온디바이스 고정이라면 모델은 기기 내에 있어야 한다.
- 비고:
  - 초기 배포는 assets 포함 방식이 가장 단순하다.
  - 추후 모델 용량이 커지면 앱 실행 후 다운로드 방식으로 전환 가능하다.

## 6. FE 기준 책임 분리 원칙

### FE가 해야 하는 일
- UsageStats 권한 확인
- 결제내역 bootstrap 조회
- 기기 내 모델 추론
- 결과 요약 UI 표시
- 추론 결과 업로드

### BE가 해야 하는 일
- 금융망 결제내역 조회
- bootstrap 세션 생성
- FE 결과 검증
- DB 반영

### FE가 하면 안 되는 일
- 금융망 API 직접 호출
- DB 반영 책임 소유
- 서버 AI 호출을 온디바이스라고 포장하는 구조

## 8. 구현 순서 제안

1. `AnalysisApi`, `dto/analysis`, `AnalysisRepository` 추가
2. `OnDeviceSubscriptionAnalysisEngine` 인터페이스와 구현체 추가
3. 모델 파일 assets 배치
4. `AnalysisModule`, `NetworkModule` DI 연결
5. `AnalysisContract` 상태 확장
6. `AnalysisViewModel`에 bootstrap -> 추론 -> 업로드 흐름 연결
7. `AnalysisLoadingScreen`에 실패/재시도/진행 메시지 반영
8. `SubscriptionConfirm`와 결과 handoff 정책 정리

## 9. 이번 설계에서 의도적으로 제외한 것

- 대규모 화면 리디자인
- 금융망 API를 FE가 직접 호출하는 구조
- 서버 AI 호출 구조
- 분석 결과를 FE 로컬 DB에 영구 저장하는 구조
- 모델 다운로드 서버/버전 관리 정책 상세

## 10. 확인 필요 사항

- `SubscriptionConfirm`가 실제 분석 결과를 어떤 소스로 읽을지
  - 서버 재조회인지
  - navigation argument인지
- 모델 런타임 선택
  - TensorFlow Lite
  - ML Kit custom model
  - ONNX Runtime Mobile
- bootstrap API가 한 번에 주는 결제내역 범위
  - 최근 1개월
  - 최근 3개월
- 업로드 실패 시 정책
  - 재시도만 허용할지
  - 임시 보관 후 재전송할지

## 11. 최종 판단

- 현재 FE 구조 기준으로 온디바이스 AI는 `feature/analysis`에 붙이는 것이 맞다.
- 새 도메인 경계는 `data/api + data/dto/analysis + data/repository/AnalysisRepository + feature/analysis/ondevice + di/AnalysisModule` 조합으로 나누는 것이 가장 자연스럽다.
- 즉 “분석 화면에서 시작하고, 기기에서 추론하고, 결과만 업로드한다”는 축으로 설계해야 한다.
