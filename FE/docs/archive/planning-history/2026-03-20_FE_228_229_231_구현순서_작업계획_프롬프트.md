# FE `228·229·231` 구현 순서 / 작업계획 / 프롬프트

> 작성일: 2026-03-20  
> 범위: `S14P21E106-228`, `S14P21E106-229`, `S14P21E106-231`  
> 제외: `S14P21E106-223`은 이번 작업에서 보류  
> 기준 코드: `FE/app/src/main/java/com/ssafy/e106`

## 1. 작업 범위 요약

- 이번 사이클은 `223`을 닫지 않고, `228`, `229`, `231`만 구현 대상으로 본다.
- 현재 코드 기준 핵심 미완료는 아래 3개다.
  - `228`: `AuthRoute.SubscriptionConfirm`가 아직 placeholder
  - `231`: `HomeRoute.CancelGuide`가 아직 placeholder이고, `customerServicePhone`, `contactEmail`이 UI에 반영되지 않음
  - `229`: OTT 서비스 노출/정렬/브랜딩이 화면마다 일관되지 않음

## 2. 권장 구현 순서

### 1순위. `228` SCR-002b 구독 확인 화면 구현

- 이유:
  - 온보딩 메인 플로우가 여기서 끊긴다.
  - `AnalysisLoading -> SubscriptionConfirm -> Dashboard` 흐름을 먼저 닫아야 이후 QA가 가능하다.
  - `229` 브랜딩 정리 전에 `SCR-002b` 화면 구조가 먼저 나와야 공통 컴포넌트 적용 범위를 확정할 수 있다.

### 2순위. `231` 구독 상세 CTA 데이터 슬롯 확장

- 이유:
  - 현재 DTO에는 `cancelGuideUrl`, `customerServicePhone`, `contactEmail`이 내려오지만 UI는 `cancelGuideUrl`만 일부 사용한다.
  - `SCR-004 -> SCR-008` 흐름을 실제 화면으로 연결해야 대시보드 상세 플로우가 완성된다.
  - `229` 브랜딩 정리는 이 단계에서 추가될 `CancelGuide` 화면까지 포함해 한 번에 정리하는 편이 낫다.

### 3순위. `229` OTT 6종 fixture 정리

- 이유:
  - 이 작업은 기능 막힘을 푸는 것보다 화면 일관성 보강 성격이 강하다.
  - `228`, `231`에서 새로 생기는 서비스 카드/배지/UI를 포함해 마지막에 공통화하는 편이 중복 수정이 적다.

## 3. 현재 코드 기준 사실관계

### `228`

- `AuthNavGraph.kt`에서 `AuthRoute.SubscriptionConfirm`는 `RoutePlaceholderScreen`으로 남아 있다.
- `AnalysisLoadingRoute`는 이미 `NavigateToSubscriptionConfirm` effect를 발행한다.
- 아직 `feature/subscriptionconfirm` 패키지와 전용 contract/viewmodel/screen은 없다.
- 분석 결과를 `SubscriptionConfirm`에 넘기는 고정 소스는 아직 없다.

### `231`

- `HomeNavGraph.kt`에서 `HomeRoute.CancelGuide`는 `PlaceholderScreen("Cancel Guide")` 상태다.
- `SubscriptionDetailResponse`에는 아래 필드가 이미 있다.
  - `cancelGuideUrl`
  - `customerServicePhone`
  - `contactEmail`
- 그러나 `SubscriptionRepository -> SubscriptionDetailData -> DashboardUiMapper -> SubscriptionDetailUiModel` 경로에서는 `cancelGuideUrl`만 반영 중이다.
- `SubscriptionDetailBottomSheet`는 추천 프로모션 CTA와 `해지 방법 보기` 버튼만 제공한다.

### `229`

- 서비스 선택/대시보드/수동확인 쪽은 대부분 첫 글자 fallback 배지 중심이다.
- 프로젝트에는 이미 `Coil AsyncImage`가 들어가 있고, 프로모션 화면은 `logoUrl` 기반 로고 렌더링 패턴을 사용한다.
- 따라서 `229`는 새 라이브러리 추가가 아니라 공통 서비스 배지 컴포넌트 정리 문제로 보는 것이 맞다.

## 4. 태스크별 작업계획

### 4-1. `228` 작업계획

#### 목표

- `SubscriptionConfirm` placeholder를 실제 화면으로 교체한다.
- 사용자가 감지된 구독 목록을 검토하고, `구독 아님`으로 제외한 뒤 `확인완료`로 대시보드에 진입할 수 있게 만든다.

#### 구현 포인트

1. `feature/subscriptionconfirm` 패키지 추가
2. `Contract / ViewModel / Route / Screen` 기본 구조 생성
3. 감지 구독 리스트 UI 구현
4. `구독 아님` optimistic remove 처리
5. empty 상태에서도 `확인완료` CTA 유지
6. `AuthNavGraph` placeholder 제거
7. 저장 계약이 불명확하면 FE 한정 handoff 구조를 명시적으로 두고, 백엔드 API를 임의로 만들지 않는다

#### 권장 파일 범위

- `FE/app/src/main/java/com/ssafy/e106/app/navigation/AuthNavGraph.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/subscriptionconfirm/SubscriptionConfirmContract.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/subscriptionconfirm/SubscriptionConfirmViewModel.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/subscriptionconfirm/ui/SubscriptionConfirmRoute.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/subscriptionconfirm/ui/SubscriptionConfirmScreen.kt`
- 필요 시 결과 handoff용 bridge/repository 파일

#### 구현 결정 메모

- 현재 가장 위험한 부분은 "감지된 구독 목록 소스"다.
- 이번 사이클에서는 다음 원칙으로 정리하는 것이 안전하다.
  - 백엔드에 없는 신규 API를 invent 하지 않는다.
  - 현재 분석 플로우가 넘겨줄 수 있는 최소 handoff 모델을 FE 내부에 만든다.
  - 저장 계약이 없으면 `확인완료`는 Dashboard 이동까지 담당하고, persistence는 후속 계약으로 분리한다.

#### 완료 기준

- `AuthRoute.SubscriptionConfirm`가 placeholder가 아니다.
- 감지 리스트, 제거 액션, empty 상태, 확인완료 CTA가 동작한다.
- 분석 완료 후 해당 화면으로 진입하고, 완료 시 Dashboard로 이동한다.

### 4-2. `231` 작업계획

#### 목표

- 구독 상세 DTO의 연락 수단 필드를 UI에 반영한다.
- `SCR-004 -> SCR-008` 흐름을 실제 화면으로 연결한다.

#### 구현 포인트

1. `SubscriptionDetailResponse`의 연락처 필드를 domain/UI model로 확장
2. `DashboardUiMapper`, `DashboardContract`, `SubscriptionDetailBottomSheet` 반영
3. `feature/cancelguide` 패키지 추가
4. `CancelGuideRoute / Screen / ViewModel` 구현
5. `HomeNavGraph` placeholder 제거
6. `해지 방법 보기` 버튼이 실제 `CancelGuide` 화면으로 이동하도록 연결
7. 전화번호, 이메일, 가이드 URL을 조건부 렌더링

#### 권장 파일 범위

- `FE/app/src/main/java/com/ssafy/e106/data/dto/subscription/SubscriptionDetailResponse.kt`

- `FE/app/src/main/java/com/ssafy/e106/feature/dashboard/DashboardContract.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/dashboard/DashboardUiMapper.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/dashboard/ui/component/DashboardBottomSheets.kt`

- `FE/app/src/main/java/com/ssafy/e106/feature/cancelguide/ui/CancelGuideScreen.kt`

#### 구현 결정 메모

- `recommendations` CTA는 이미 "추천 보기"로 동작하므로 깨지지 않게 유지한다.
- 이번 작업의 핵심은 "추천 프로모션" 확장이 아니라 "해지/문의 경로 확장"이다.
- `CancelGuide` 화면은 `subscriptionId` 기반 조회 구조로 두는 것이 현재 네비게이션과 맞다.

#### 완료 기준

- `HomeRoute.CancelGuide`가 placeholder가 아니다.
- `customerServicePhone`, `contactEmail`, `cancelGuideUrl`가 UI에 반영된다.
- 구독 상세에서 해지 지원 화면으로 실제 이동 가능하다.

### 4-3. `229` 작업계획

#### 목표

- OTT 6종 기준 서비스 노출/정렬/브랜딩을 통일한다.
- 첫 글자 fallback 배지 위주의 화면들을 `logoUrl` 우선 렌더링 패턴으로 정리한다.

#### 구현 포인트

1. 공통 서비스 배지/로고 컴포넌트 설계
2. `logoUrl`가 있으면 `AsyncImage`, 없으면 첫 글자 fallback
3. 적용 대상 화면 정리
  - Dashboard 서비스 카드/바텀시트
  - ManualMapping
  - 신규 `SubscriptionConfirm`
  - 신규 `CancelGuide`
4. OTT 6종 노출/정렬/명칭 일관성 점검
5. 문구와 정렬 기준을 화면명세와 맞춘다

#### 권장 파일 범위

- `FE/app/src/main/java/com/ssafy/e106/core/ui/component/...` 공통 컴포넌트 신설 위치
- `FE/app/src/main/java/com/ssafy/e106/feature/dashboard/ui/component/DashboardBottomSheets.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/dashboard/ui/DashboardScreen.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/mapping/ui/ManualMappingScreen.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/subscriptionconfirm/ui/SubscriptionConfirmScreen.kt`
- `FE/app/src/main/java/com/ssafy/e106/feature/cancelguide/ui/CancelGuideScreen.kt`

#### 구현 결정 메모

- `229`는 API 연동 작업이 아니라 "실데이터를 어떻게 일관되게 보이게 할지"를 닫는 작업이다.
- 프로모션 화면의 `PromotionServiceAvatar` 패턴을 재사용하거나 일반화하는 방향이 가장 안전하다.
- 브랜딩 정리 전에 새 화면(`228`, `231`)이 먼저 있어야 적용 범위가 고정된다.

#### 완료 기준

- 주요 서비스 노출 화면에서 로고/라벨/정렬 기준이 일관된다.
- `logoUrl`가 있으면 동일한 시각 규칙으로 렌더링된다.
- `logoUrl`가 없을 때 fallback 정책이 화면마다 다르지 않다.

## 5. 작업 분할 방식

### 분할 권장

1. `228` 단독 구현
2. `231` 단독 구현
3. `229` 마지막 일괄 정리

### 이유

- `228`과 `231`은 신규 feature/route 추가라서 서로 write set이 비교적 분리된다.
- `229`는 공통 UI 정리라서 먼저 하면 뒤 작업에서 다시 건드릴 가능성이 높다.

- `CancelGuide`에서 URL/전화/이메일 유무별 조건부 렌더링
- Dashboard / ManualMapping / SubscriptionConfirm / CancelGuide에서 서비스 로고 렌더링 일관성
