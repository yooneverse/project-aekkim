# FE 발표주간 QA 및 회귀 테스트계획

> 작성일: 2026-03-23  
> 범위: `[FE] (QA) 핵심 사용자 플로우 종단 테스트`, `[FE] (Bugfix) 발표 전 P0/P1 버그 수정 및 회귀 대응`, `[FE] (Release) 실기기 회귀 테스트 및 발표 빌드 점검`

## 1. 문서 목적

- 최종 발표 전 FE 핵심 사용자 플로우를 전수 점검한다.
- P0/P1 버그를 빠르게 식별하고, 수정 후 회귀 테스트까지 같은 문서에서 관리한다.
- 발표 당일 사용할 계정, 환경, 빌드, 단말 조건을 사전 확정한다.

## 2. 이번 주 테스트 목표

### 2-1. QA 목표

- 온보딩부터 구독확인까지 첫 사용자 플로우가 끊기지 않는다.
- 대시보드, 수동확인, 구독상세, 해지지원 흐름이 자연스럽게 이어진다.
- 체크인, 알림함, 프로모션 흐름이 발표 시연 기준으로 동작한다.

### 2-2. Bugfix 목표

- 로그인/권한/네비게이션 관련 `P0` 이슈를 먼저 제거한다.
- API 실패, 재시도, 에러 토스트, 상태 동기화, 새로고침, 중복 클릭 이슈를 정리한다.
- 수정 이슈는 반드시 같은 날 회귀 테스트까지 완료한다.

### 2-3. Release 목표

- 발표용 계정/환경값/네트워크 조건을 사전에 고정한다.
- 실기기와 에뮬레이터 양쪽에서 회귀 테스트를 수행한다.
- 세션 만료, 권한 거부, 네트워크 끊김 같은 현장 리스크를 예외 시나리오로 점검한다.

## 3. 테스트 범위

### 포함

- `SCR-001` 온보딩 로그인
- `SCR-001-2` 약관 동의 및 권한
- `SCR-002` 분석중
- `SCR-002b` 구독 확인
- `SCR-003` 대시보드
- `SCR-004` 구독 상세
- `SCR-007`, `SCR-007-2` 체크인
- `SCR-008` 해지지원
- `SCR-009` 수동확인
- `SCR-010` 마이페이지
- 알림함, 푸시, 딥링크, 프로모션

### 제외

- 발표 범위 밖의 신규 기능 탐색
- FE 코드와 무관한 백엔드 운영 장애 분석
- 화면명세에 없는 장기 개선 항목

## 4. 사전 준비

### 4-1. 실행 환경

- `FE/local.properties`의 `BASE_URL_DEV`를 발표용 대상 서버 기준으로 고정한다.
- `FE/app/google-services.json` 존재 여부를 확인한다.
- Android SDK Build Tools `35.0.0` 설치 여부를 확인한다.
- 발표 빌드 기준 브랜치/커밋을 문서 상단 또는 빌드 체크리스트에 기록한다.

### 4-2. 테스트 계정/데이터셋

| 구분 | 용도 | 필요 조건 |
| --- | --- | --- |
| `QA-NEW-01` | 온보딩, 분석, 구독 0건 Empty 확인 | 신규 계정, 로그인 가능, 구독 없음 |
| `QA-SUB-01` | 대시보드, 구독상세, 해지지원 | 구독 2건 이상, 상세 데이터 존재 |
| `QA-MAP-01` | 수동확인 흐름 | 확인 필요 결제 1건 이상 |
| `QA-CHK-01` | 체크인 진입 | 미응답 체크인 1건 이상 |
| `QA-CHK-02` | 후속 권유/해지지원 | 최근 2회 연속 부정 응답 이력 |
| `QA-NOTI-01` | 알림함/딥링크/프로모션 | `CHECKIN`, `PROMO`, `CHURN_REVIEW` 알림 보유 |

### 4-3. 단말/네트워크

- Android 실기기 1대 이상
- Android 에뮬레이터 1대 이상
- 정상 Wi-Fi
- 느린 네트워크 또는 핫스팟
- 네트워크 off/on 전환 가능 상태

## 5. 테스트 운영 순서

### 5-1. 1차 Smoke

- 앱 설치 및 실행
- 로그인 가능 여부
- 홈 진입 가능 여부
- 치명적인 크래시 여부
- 필수 API 실패 시 복구 가능 여부

### 5-2. 핵심 플로우 E2E

1. 온보딩 → 분석 → 구독확인
2. 대시보드 → 수동확인 → 구독상세 → 해지지원
3. 체크인 → 알림함 → 프로모션

### 5-3. 버그 수정 루프

1. 결함 기록
2. `P0/P1/P2` 분류
3. 당일 수정
4. 영향 플로우 재실행
5. 관련 회귀 테스트

### 5-4. 발표 빌드 검증

- 실기기 재설치
- 로그인 재확인
- 시연 플로우 전체 1회 연속 실행
- 예외 시나리오 최소 1회 확인

## 6. 세부 테스트 시나리오

### 6-1. 온보딩 → 분석 → 구독확인 E2E

#### 목표

- 신규 사용자 첫 진입 흐름이 중단 없이 진행된다.

#### 실행 순서

1. 로그아웃 상태에서 앱 실행
2. Google 로그인
3. 약관/선택동의/권한 처리
4. 분석 로딩 진입
5. 구독확인 화면 진입

#### 체크포인트

- Google 로그인 버튼은 중복 클릭이 막힌다.
- Kakao/Naver는 준비중 정책이 일관되다.
- 필수 약관 미동의 시 CTA가 비활성이다.
- 약관 상세 진입 후 복귀가 자연스럽다.
- 알림 권한, UsageStats 권한 흐름이 끊기지 않는다.
- 분석 로딩은 단계 진행 또는 명확한 fallback UX를 가진다.
- 분석 완료 후 `SCR-002b`로 이동한다.
- 구독확인에서 `구독 아님`, `확인완료`, Empty 상태가 자연스럽다.

#### 실패 시 우선 분류

- 앱 크래시, 로그인 불가, 분석 후 진입 불가: `P0`
- 권한 안내 부정확, 토스트/문구 불일치: `P1`

### 6-2. 대시보드 → 수동확인 → 구독상세 → 해지지원 E2E

#### 목표

- 발표 핵심 가치인 구독 관리 흐름이 한 번에 이어진다.

#### 실행 순서

1. 대시보드 진입
2. 확인 필요 결제 배너 또는 CTA 진입
3. 수동확인 처리
4. 구독상세 바텀시트 진입
5. 해지지원 화면 진입

#### 체크포인트

- 대시보드 `Loading/Empty/Error/Stale` 상태가 분리되어 보인다.
- `추가`, `편집`, `새로고침`이 의도대로 동작한다.
- 수동확인에서 `구독 아님`과 서비스 선택 흐름이 충돌하지 않는다.
- 수동확인 완료 후 대시보드 상태가 즉시 동기화된다.
- 구독상세에 서비스명, 요금제, 금액, 다음 결제일이 정확히 표시된다.
- `해지 방법 보기` CTA가 실제 해지지원 화면으로 이동한다.
- 해지지원에서 URL/전화/이메일이 값 유무에 맞게 조건부 노출된다.
- 복귀 후 이전 화면 컨텍스트가 유지된다.

#### 실패 시 우선 분류

- 구독상세 진입 불가, 해지지원 placeholder, 상태 동기화 실패: `P0`
- CTA 문구/정렬/브랜딩 불일치: `P1`

### 6-3. 체크인 → 알림함 → 프로모션 E2E

#### 목표

- 발표 중 보조 시연 흐름이 끊기지 않도록 한다.

#### 실행 순서

1. 체크인 배너 또는 딥링크 진입
2. 체크인 응답 제출
3. 후속 권유 화면 확인
4. 알림함 진입
5. `CHECKIN`, `CHURN_REVIEW`, `PROMO` 알림 이동 확인
6. 프로모션 목록/상세 확인

#### 체크포인트

- 풀스크린 체크인과 배너 진입 정책이 다르게 동작한다.
- 중복 제출이 막힌다.
- 실패 시 재시도 UX가 있다.
- 알림함은 목록, Empty, 읽음 처리, 재진입을 지원한다.
- `PROMO` 알림 이동이 placeholder가 아닌 실제 목적지로 연결된다.
- 프로모션 목록/상세는 링크 CTA, 만료 상태, Empty/Error 상태를 가진다.

#### 실패 시 우선 분류

- 알림 클릭 후 목적지 불일치, 프로모션 진입 불가: `P0`
- 읽음 표시, 카피, 배지 스타일 이슈: `P1`

## 7. Bugfix 회귀 계획

### 7-1. 집중 영역

- 로그인/권한/네비게이션
- API 실패/재시도/에러 토스트
- 상태 동기화/새로고침
- 중복 클릭/중복 제출

### 7-2. 회귀 원칙

- 수정한 화면만 다시 보지 않는다.
- 수정 이슈가 속한 플로우의 앞단과 뒷단까지 다시 실행한다.
- 저장형 액션 수정 후에는 `성공`, `실패`, `재시도`, `뒤로가기 후 재진입`을 모두 확인한다.

### 7-3. 회귀 체크 매트릭스

| 이슈 유형 | 최소 재검증 범위 |
| --- | --- |
| 로그인/세션 | 온보딩, 약관, 앱 재실행, 로그아웃 |
| 권한 | 권한 거부, 재허용, resume |
| 구독 상태 | 대시보드, 수동확인, 구독상세, 해지지원 |
| 체크인 | 배너 진입, 딥링크 진입, 후속 권유 |
| 알림 | 목록, 읽음, 목적지 이동 |
| 프로모션 | 목록, 상세, 외부 링크 |

## 8. 결함 관리 기준

### 8-1. 우선순위 정의

- `P0`
  - 발표 시연이 끊기거나 앱이 멈춘다.
  - 핵심 플로우 진입이 안 된다.
  - 저장/조회/이동이 완전히 실패한다.
- `P1`
  - 발표는 가능하지만 품질 저하가 크다.
  - 에러 처리, 상태 전이, 문구, 일관성 문제가 눈에 띈다.
- `P2`
  - 후순위 UI polish, 세부 정렬, 비치명 문구 이슈

### 8-2. 결함 기록 포맷

| ID | 우선순위 | 화면/플로우 | 재현 절차 | 기대 결과 | 실제 결과 | 담당 | 상태 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| BUG-001 | P0 | 온보딩 → 분석 | ... | ... | ... | FE | Open |

### 8-3. 종료 기준

- `P0`는 0건
- `P1`은 발표 시연을 방해하지 않는 수준으로 정리
- 남은 `P1/P2`는 PM 관점 메모와 우회 시나리오를 남긴다.

## 9. 발표 빌드 점검

### 9-1. 빌드 전 체크

- 발표용 서버 주소 확정
- 발표용 계정 로그인 가능 여부 확인
- 테스트 데이터셋 고정
- 푸시/알림 시연 필요 여부 확정

### 9-2. 빌드 후 체크

- 클린 설치 후 첫 실행
- 자동 로그인/세션 복구
- 네트워크 정상 상태에서 시연 플로우 1회
- 네트워크 끊김/복구
- 세션 만료 시 fallback
- 권한 거부 후 재진입

### 9-3. 발표 직전 최종 체크리스트

- [ ] 실기기 빌드 설치 완료
- [ ] 발표용 계정 로그인 완료
- [ ] 온보딩 신규 계정 플로우 확인 완료
- [ ] 핵심 시연 계정 대시보드 데이터 확인 완료
- [ ] 수동확인/체크인/알림/프로모션 데이터 확인 완료
- [ ] 세션 만료/권한 거부/네트워크 끊김 예외 확인 완료
- [ ] 치명 결함 `P0` 0건 확인

## 10. 완료 판정 기준

- 핵심 E2E 3개 플로우가 실기기에서 연속 통과한다.
- 수정된 `P0/P1` 버그는 회귀 테스트까지 완료된다.
- 발표용 빌드와 계정/환경값이 고정된다.
- 발표 당일 사용할 우회 시나리오와 잔여 리스크가 문서로 정리된다.
