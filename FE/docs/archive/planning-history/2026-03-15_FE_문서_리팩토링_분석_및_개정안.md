# 1. 전체 요약

- 프로젝트 현재 기준선은 `docs/2026-03-17_요구사항정의서_v1.9_AEKKIM.md`다. 기존 FE 계획서는 2026-03-09 시점의 계약을 전제로 작성되어 있어 그대로 유지하면 안 된다.
- 기존 FE 개발계획서의 유효성 총평은 "구조는 재사용 가능하지만 계약과 범위는 재정의 필요"다. 특히 인증 API, 알림 enum, 마이페이지 저장 흐름, 수동 구독 추가 범위, 알림함 범위가 최신 문서와 어긋난다.
- 현재 FE는 온보딩/약관/대시보드/마이페이지의 화면 뼈대와 일부 실제 인증 연동까지 진행되었다. 반면 구독, 프로모션, 알림 도메인은 대부분 목업 또는 플레이스홀더다.
- 현재 BE는 인증과 사용자 API만 실질 구현 상태다. 구독, 프로모션, 알림 도메인은 문서가 먼저 있고 컨트롤러/서비스/엔티티는 대부분 비어 있다.
- 따라서 이번 문서 개정의 핵심은 "기존 계획 유지"가 아니라 "실제 구현 가능한 범위와 BE 의존 범위를 분리한 실행계획으로 재편"이다.

## 가장 큰 변경점

1. 인증 계약이 `POST /api/v1/auth/login` + `googleAccessToken`에서 `POST /api/v1/auth/login/google` + `googleIdToken`으로 변경되었다.
2. Kakao 로그인 API가 추가되었고 Naver는 노출만 하되 준비중 상태로 정리되었다.
3. OTT MVP 범위가 5종이 아니라 6종으로 확정되었고 Watcha가 포함되었다.
4. 마이페이지는 조회만이 아니라 알림 설정 PATCH와 선택 동의 PATCH까지 즉시 저장 흐름이 필수다.
5. 알림 도메인은 FCM 토큰 등록, 알림 목록, 읽음 처리, 타입별 라우팅까지 MVP 범위로 승격되었다.
6. 알림 타입 정본이 `CHECKIN`, `PROMO`, `CHURN_REVIEW`로 정리되었고 기존 FE의 `CHECK_IN`, `CANCEL_REVIEW`는 폐기해야 한다.
7. 구독 상세는 단순 보기 화면이 아니라 고객센터 전화, 이메일, 해지 가이드 URL, 최근 사용량, 추천 프로모션 데이터 연결이 필요하다.
8. 체크인 흐름은 FCM 유무와 D-3 이후 미응답 배너 진입까지 고려해야 한다.
9. 수동 구독 추가와 확인 필요 결제 처리 흐름이 화면명세 기준으로 더 구체화되었다.
10. 현재 코드 기준으로 실제 연동 가능한 영역은 인증/사용자뿐이며, 나머지는 목업 전략을 문서에 명시해야 한다.

## 가장 큰 리스크 5개

1. FE 문서가 오래된 인증 계약을 기준으로 남아 있으면 로그인, 세션 저장, QA 케이스가 모두 틀어진다.
2. 알림 enum과 payload 키 충돌을 정리하지 않으면 FCM 수신, 딥링크, 알림함 라우팅이 동시에 깨진다.
3. 화면명세에는 있는데 BE 구현이 없는 구독/프로모션/알림 API를 완료로 오인하면 일정 산정이 붕괴한다.
4. 마이페이지 토글을 로컬 상태만 바꾸는 현재 구현을 완료로 간주하면 실제 저장 실패와 설정 불일치가 프로덕션 결함으로 이어진다.
5. 화면명세에 전용 알림함 화면 정의가 빠져 있어 FE 네비게이션 구조와 QA 기준이 비어 있다.

## 문서 업데이트 우선순위

1. 공통 규칙 및 최종 체크리스트
2. 스토리1 온보딩/로그인 계획서
3. 스토리4 마이페이지 계획서
4. 스토리5 체크인/해지검토/알림 계획서
5. 스토리3 구독확인/대시보드 계획서
6. 횡단 실행 계획
7. 스토리2 UsageStats/분석시작 계획서

# 3. FE/BE 진행상황 반영 전제

## 현재 FE 완료 범위

- Google 로그인 API 연동, 토큰 저장, 리프레시 토큰 기반 세션 복구
- 온보딩 화면, 약관 동의 화면, 알림 권한 요청, UsageStats 권한 설정 이동
- 대시보드 기본 레이아웃, 수동 구독 추가 바텀시트, 구독 상세 바텀시트, 마이페이지 기본 화면
- 공통 디자인 토큰과 핵심 컴포넌트 일부 (`AppButton`, `AppCard`, `AppAgreementRow`, `AppBottomSheet`, `AppBottomTabBar`, Empty/Loading/Error 상태 컴포넌트)
- FCM 서비스/채널 초기 셸

## 현재 FE 미완 범위

- Kakao 로그인 실연동, Naver 준비중 처리, 온보딩 후속 화면 완성
- 선택 동의 서버 저장, FCM 토큰 등록, 알림 설정 PATCH 연동
- 알림함 목록/읽음 처리/라우팅
- 프로모션 목록/상세, 체크인 화면, 해지 가이드 화면, 구독 확인 화면
- 대시보드/수동 구독 추가/확인 필요 결제의 실API 연동

## 현재 BE 구현 완료 API

- `POST /api/v1/auth/login/google`
- `POST /api/v1/auth/login/kakao`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/users/me`
- `DELETE /api/v1/users/me`
- `PATCH /api/v1/users/me/notification-settings`
- `PATCH /api/v1/users/me/optional-consent`
- 개발용 JWT 발급, 개발용 FCM 테스트 발송

## 현재 BE 미구현 또는 불안정 API

- 구독 도메인 전체: 서비스 목록/상세, 요금제, 내 구독 목록/상세, 등록/수정/삭제, 체크인 제출/이력
- 프로모션 도메인 전체: 목록/상세
- 알림 도메인 전체: FCM 토큰 등록, 목록 조회, 읽음 처리
- 알림 생성 배치 규칙

## FE에서 목업 처리해야 하는 영역

- 대시보드 구독 리스트와 월 합계
- 수동 구독 추가용 OTT 서비스/요금제 데이터
- 프로모션 목록/상세
- 확인 필요 결제 리스트와 매핑 후속 처리
- 알림함 목록과 타입별 상세 이동
- 체크인 질문/응답 후속 분기

## 연동 우선순위

1. 인증/사용자 API 정합성 마무리
2. 마이페이지 토글 PATCH와 선택 동의 PATCH
3. FCM 토큰 등록 계약 확정
4. 구독 서비스 목록/내 구독 목록/수동 등록 API
5. 체크인 제출/조회 API
6. 알림 목록/읽음 처리 API
7. 프로모션 목록/상세 API

# 4. 스토리별 정밀 점검

## 공통 체크리스트 점검

### 4-0-1. 원래 목표

- FE 전 스토리에 공통 적용할 규칙, 컴포넌트, API 계약, 상태 처리, QA 기준을 정리하는 문서였다.

### 4-0-2. 최신 기준에서 유지되는 범위

- Route 구조 분리
- 공통 디자인 토큰/컴포넌트 사용
- UiState/UiEffect/Intent 패턴
- mock/real 분리 원칙

### 4-0-3. 범위 변경/삭제/추가 사항

- 인증 계약, 알림 enum, 알림함 범위, 마이페이지 PATCH 규칙은 최신 계약으로 교체해야 한다.
- Watcha 포함 6종 OTT 범위를 공통 기준으로 승격해야 한다.
- "완료" 기준에 문서 정합성과 예외처리를 명시해야 한다.

# 5. 횡단 이슈 정리

- 공통 컴포넌트: `AppButton`, `AppCard`, `AppAgreementRow`, `AppBottomSheet`, `AppBottomTabBar`, `AppLoadingState`, `AppErrorState`, `AppEmptyState`를 유지한다. 다만 토글 저장중, 알림함 Empty, 체크인 Error 상태 프리셋이 추가로 필요하다.
- 디자인 토큰 반영: 현재 토큰 구조는 사용 가능하다. 스토리별 문서에 typography/spacing/state 규칙을 다시 매핑해야 한다.
- 인증/토큰 처리: Google 로그인과 refresh는 동작 중이다. Kakao 로그인, provider 표시, 로그아웃 후 초기화 범위를 정리해야 한다.
- 상태 관리: `UiState/UiEffect/Intent` 패턴은 유지한다. 즉시 저장 토글과 알림 읽음 처리에는 optimistic update + rollback 규칙이 필요하다.
- 네비게이션: Auth route의 `AnalysisLoading`, `SubscriptionConfirm`, Home route의 `PromotionList`, `PromotionDetail`, `CheckIn`, `CancelGuide`는 아직 placeholder다. 알림함 route가 신규 추가되어야 한다.
- API 에러 처리: 현재 사용자/인증 쪽은 기본 처리만 있다. 마이페이지 토글 저장, 알림 읽음, 체크인 제출에는 재시도와 롤백 정책이 필요하다.
- 로딩/에러/Empty: 화면명세 기준으로 대시보드, 알림함, 확인 필요 결제, 프로모션 리스트 각각의 상태 정의가 필요하다.
- FCM: Android 셸은 준비되었지만 토큰 등록 API, enum 정본, type별 referenceId 라우팅이 미완이다.
- UsageStats: 권한 재유도 UX는 있으나 분석 시작 화면과 대시보드 진입 브리지가 약하다.
- 더미데이터/프리뷰/목업 전략: 구독/알림/프로모션은 당분간 mock이 필요하다. 다만 mock 모델명과 real DTO를 분리해야 전환 비용이 줄어든다.
- QA 체크리스트: "완료"는 UI 존재가 아니라 저장 성공, 오류 처리, 문서 정합성, 목업 제거 조건까지 충족해야 한다.
- 팀원 분업 포인트: FE는 Story1/4 P0를 먼저 고정하고, Story3/5는 contract-first와 mock UI를 병행한다. BE는 구독/알림/프로모션 도메인 스켈레톤보다 실제 contract endpoint 구현 우선순위를 먼저 정해야 한다.
