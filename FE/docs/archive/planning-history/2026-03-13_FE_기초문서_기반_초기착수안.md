## 1) 문서 기준 해석

### 1-1. 확정사항

- 플랫폼: Android (Kotlin, Jetpack Compose)
- MVP 범위: OTT 5개(Netflix, Tving, Wavve, 쿠팡플레이, Disney+)
- 인증: UI는 3개 소셜 버튼 노출, MVP 실제 인증은 Google 우선
- 권한: UsageStats 온보딩 필수, FCM 권한은 허용/거부 모두 진행
- 대시보드: 앱 재진입 시 서버 DB 조회 기반(앱 진입마다 SSAFY 직접 호출 없음)
- 상태 패턴: `UiState / Intent / UiEffect`, `StateFlow / SharedFlow(replay = 0)`
- 라우트 네이밍: `AuthRoute.*`, `HomeRoute.*`, `SettingsRoute.*`

### 1-2. 제약사항

- 문서 밖 정책/플로우를 코드로 선확정하지 않음
- API 미확정은 `[추가 필요]`, 계약 협의는 `[협의 필요: FE/BE]`로 분리
- 빠른 구현보다 mock/real 전환 가능한 Repository 구조 유지
- 에러 문구는 공통 규칙 우선 적용

### 1-3. 미확정/충돌 포인트

1. 인증 로그인 endpoint 표기 충돌
   - FE 공통 규칙: `/api/v1/auth/login/{provider}`
   - 인증 API 문서: `/api/v1/auth/login`
2. Login 응답 계약 표기 차이
   - FE 공통 규칙: `LoginResponse`와 `TokenPair` 분리
   - 인증 API 문서: 로그인 응답에 `accessToken`, `refreshToken`, `user` 동시 포함
3. 수동 구독 추가 서비스 범위
   - 상위 MVP 범위 5개 vs 일부 화면 시안(왓챠 포함 6개)

## 2) 구현 전 계획(초기 착수)

### 2-1. 작업 목표

- 문서 기준으로 현재 FE 코드의 정렬 상태를 가시화하고,
- 바로 구현 가능한 영역과 협의 선행 영역을 분리해 다음 작업의 재현성을 확보한다.

### 2-2. 영향 범위

- 화면/라우팅: `app/navigation/*`, `feature/auth/*`
- 상태관리: `feature/*/Contract, ViewModel` 패턴
- API/DTO: `data/api/*`, `data/dto/*`, `data/repository/*`

### 2-3. 선행 작업

1. 로그인 API 계약 충돌 해소(문서 단일 기준 확정)
2. 수동 구독 추가 서비스 범위(5개/6개) 확정
3. UsageStats 권한 요청 UX 문구 및 재유도 정책 문구 확정

### 2-4. 리스크

- 계약 미확정 상태에서 DTO/Repository를 선구현하면 재작업 비용 증가
- 화면 시안과 MVP 범위 불일치(서비스 개수)로 QA 기준 흔들림

## 3) 코드/수정안 (초기 작업 반영)

### 3-1. 현재 코드 기준 정렬 결과

| 영역 | 현재 상태 | 문서 정렬 판단 |
| --- | --- | --- |
| Route 네이밍 | `AuthRoute`, `HomeRoute` 사용 중 | 정렬됨 |
| Auth 상태 패턴 | `AuthUiState/Intent/UiEffect` 사용 중 | 정렬됨 |
| 구글 로그인 flow | `AuthViewModel`에서 토큰 저장 후 약관 이동 | 정렬됨 |
| 카카오 API 교환 코드 | `AuthApi/AuthRepository`에 잔존 | 정리 필요(활성 범위 외) |
| 로그인 응답 계약 | `TokenPair` 중심 처리 | 협의 후 정렬 필요 |

### 3-2. 바로 수행 가능한 작업(확정 기반)

1. `Story 1/2` 범위 내 화면 상태머신 검증(mock 기준)
2. 공통 에러/토스트 문구를 Screen 단에서 `UiEffect.ShowToast`로 일원화 점검
3. 미구현 화면은 route contract 유지 + placeholder 책임 범위 명시

### 3-3. 협의 후 수행할 작업

1. 로그인 DTO 계약 최종 확정 후 `AuthApi/AuthRepository/AuthViewModel` 동시 정렬
2. 수동 구독 추가 서비스 목록(5 vs 6) 확정 후 `SCR-003-2` 메타데이터 반영

## 4) 반영 내용 요약

- 기초문서(요구사항, 화면명세, 인증/구독 API, FE 공통 규칙) 기준으로 초기 착수 기준선을 정리했다.
- 현재 코드에서 즉시 진행 가능한 구현 영역과 협의 선행 영역을 분리했다.
- 문서 간 충돌 포인트(인증 endpoint/로그인 응답 계약/서비스 개수)를 후속 논의 항목으로 명시했다.
- 다음 구현은 본 문서를 기준으로 Story 단위로 진행하고, 미확정 항목은 코드 선확정 없이 분리 관리한다.
