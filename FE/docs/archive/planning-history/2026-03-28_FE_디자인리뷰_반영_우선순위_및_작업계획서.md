# FE 디자인리뷰 반영 우선순위 및 작업계획서

> 작성일: 2026-03-28  
> 기준 자료: `OTT_앱_디자인_리뷰_종합_수정_우선순위.md`, `FE/docs/2026-03-12_FE_fdesign_css_definition__v2.md`, `FE/docs/2026-03-12_FE_mobile_component_spec_v1.md`, 현재 `FE/app` 구현 상태

## 1. 문서 목적

- 외부 디자인 리뷰 항목을 현재 FE 코드에 직접 매핑해 `이미 반영된 것`, `부분 반영`, `미반영`을 구분한다.
- 발표 직전에도 빠르게 수용 가능한 수정과, 지금 건드리면 회귀 리스크가 큰 수정을 분리한다.
- FE 단독 작업, 디자이너 협업 작업, 기획/BE 합의가 필요한 작업의 경계를 명확히 한다.

## 2. 현재 상태 종합 판단

### 2-1. 총평

- 화이트 베이스, 오렌지 포인트, 카드 중심 구조, 캐릭터 자산 활용은 현재 코드에도 분명히 반영되어 있다.
- 다만 리뷰에서 가장 크게 지적된 `대시보드 행동 유도 부족`, `회색 텍스트 대비`, `탭바 활성 인지`, `검토 필요 결제 배너 문맥 부족`은 아직 핵심 구조가 그대로 남아 있다.
- 프로모션/추천/상세 계열은 실제 구현이 존재하지만, 그라데이션과 오버레이 스타일이 FE 디자인 기준 문서보다 강하게 들어가 있어 톤 불일치가 남아 있다.
- 문서 기준과 실제 구현 상태가 어긋난 부분이 있다. 특히 `FE/docs/2026-03-19_FE_화면명세서_기반_전체_체크리스트.md`는 일부 화면을 미구현으로 적고 있지만, 현재 코드에는 `SubscriptionConfirm`, `PromotionList/Detail`, `CancelGuide` route가 이미 연결되어 있다. 이 문서 갱신이 선행되지 않으면 작업 우선순위 판단이 계속 흔들린다.

## 3. 리뷰 외 추가로 확인된 FE 디자인 갭

### 3-1. 디자인 시스템 문서와 실제 구현의 추가 차이

- 폰트: `core/ui/theme/Type.kt`가 `PretendardFontFamily`와 실제 `res/font` 리소스(`pretendard_regular/medium/semibold/bold.otf`)를 연결했고, 기존 typography token의 size/lineHeight는 유지했다.
- 문서 정합성: 화면 구현 문서가 현재 코드보다 뒤처져 있다. 실제 구현 상태를 모르고 문서만 보고 작업하면 우선순위가 왜곡된다.
- 디자인 예외 누적: 권한 모달, 약관 진행 패널, 프로모션 카드, 추천 카드에 gradient/tint 예외 스타일이 누적되어 공통 규칙의 일관성이 약해졌다.

### 3-2. 해석

- 이번 라운드는 `리뷰 피드백 반영`이 목적이므로, 폰트 교체나 전체 디자인 시스템 재정의 같은 전면 리팩터링은 후순위가 맞다.
- 대신 전역 토큰, 탭바, 대시보드 배너 문구/위계처럼 `효과가 크고 회귀 리스크가 낮은 것`부터 자르는 편이 합리적이다.

### 4-2. 바로 착수할 세부 작업

#### 1. 문서 상태 동기화

- 대상: `FE/docs/2026-03-19_FE_화면명세서_기반_전체_체크리스트.md`
- 목적:
  - `SCR-002b`, `SCR-005`, `SCR-006`, `SCR-008`의 상태를 현재 구현 기준으로 바로잡는다.
  - 다음 작업자가 placeholder 기준으로 잘못 판단하는 리스크를 제거한다.
- 권장 순서:
  - 라우트 구현 여부 정정
  - 남은 미완 요소만 별도 `보완 필요`로 재분류

#### 2. 보조 텍스트 대비 상향

- 대상:
  - `core/ui/theme/Color.kt`
  - `core/ui/theme/Theme.kt`
  - `DashboardScreen.kt`, `InsightScreen.kt`, `PromotionListScreen.kt`, `PromotionDetailScreen.kt`, `AnalysisLoadingScreen.kt`, `CheckInScreen.kt`
- 제안:
  - `TextSecondary`를 단일 상향하는 대신 `textSubtle`, `textMuted` 2단계로 분리하거나, 우선 1차로 `#666`대까지 올린다.

- 기대 효과:
  - 전체 화면의 스캔 속도가 바로 개선된다.
  - 리뷰에서 반복된 `회색 텍스트가 흐리다`는 항목을 가장 낮은 비용으로 해소한다.

#### 3. 탭바 활성 인디케이터 추가 + 라벨 통일

- 대상:
  - `core/ui/component/navigation/AppBottomTabBar.kt`
  - `DashboardScreen.kt`
  - `InsightScreen.kt`
  - `PromotionListScreen.kt`
  - `MyPageScreen.kt`
- 제안:
  - 선택 탭에 작은 도트 또는 하단 underline을 추가한다.
  - 탭명은 이번 라운드에서 하나로 고정한다. 현재 리뷰와 화면 카피 흐름을 보면 `추천`이 더 자연스럽다.
- 기대 효과:
  - 내비게이션 품질이 즉시 좋아진다.
  - 화면마다 다른 탭 라벨이 보이는 작은 신뢰감 하락을 제거한다.

#### 4. 대시보드 `검토 필요 결제` 배너 1차 개선

- 대상:
  - `feature/dashboard/ui/DashboardScreen.kt`
  - 필요 시 `DashboardContract.kt`
- 현재 상태:
  - `DashboardUiState`에 `pendingReviews` 리스트가 있지만, UI는 `count`만 노출한다.
- 제안:
  - FE 1차: `넷플릭스 외 4건 확인 필요`처럼 첫 항목 기반 preview 문구로 교체한다.
  - 배너 우측 CTA를 유지하되, count headline보다 구체적인 서비스명/가맹점명을 먼저 보여준다.
  - 대시보드 상단 순서를 `행동 유도 배너 -> 총액 -> 리스트`로 재배치하는 lite 버전을 먼저 적용한다.
- 기대 효과:
  - 리뷰의 핵심 문제를 BE 수정 없이 일부 완화할 수 있다.
  - `509건`류의 압박감을 줄이고 바로 눌러야 하는 이유를 설명할 수 있다.

#### 5. 분석 초기 `0원` 숨김

- 대상: `feature/analysis/view/AnalysisLoadingScreen.kt`
- 제안:
  - 진행률 50% 이전 또는 `hasUsageHistory == false`인 동안에는 금액 숫자 대신 `분석 중` 상태로 처리한다.
  - 숫자 영역 레이아웃은 유지해 점프를 막고, copy만 바꾼다.
- 기대 효과:
  - 사용자가 `절약할 게 없네`라고 오해하는 문제를 제거한다.

#### 6. 프로모션/추천/상세 카드 톤다운

- 대상:
  - `feature/promotion/ui/PromotionListScreen.kt`
  - `feature/promotion/ui/PromotionDetailScreen.kt`
  - `feature/dashboard/ui/component/DashboardBottomSheets.kt`
- 제안:
  - 강한 브랜드 gradient를 바로 제거하기보다 1차로 채도/명도/overlay alpha를 낮춘다.
  - 블랙 배지(`Color.Black.copy(alpha = 0.62f)`)는 밝은 배경 대비 기준에 맞게 재조정한다.
  - 카드 본문은 white base 규칙을 유지하고, 컬러는 hero 영역과 절약 badge에만 남긴다.
- 기대 효과:
  - 현재 앱 톤과 가장 부딪히는 영역을 빠르게 정리할 수 있다.

## 5. 지금 수정하면 리스크가 큰 항목

### 5-1. 고위험 항목 표

| 항목 | 리스크가 큰 이유 | 영향 범위 | 선행조건 | 권장 시점 |
| --- | --- | --- | --- | --- |
| 대시보드 정보 구조 전면 개편 | 홈 첫 화면의 정보 위계, 배너, 편집/추가 플로우, Empty 상태까지 동시에 흔든다 | 대시보드 전체, 회귀 QA 범위 큼 | 기획/디자인 합의 | 별도 라운드 |
| 구독료 전월 대비/추이 표시 | 현재 데이터 모델은 `monthlyTotalAmount`만 갖고 있어 비교값이 없다 | FE + BE + 문구 설계 | API/도메인 정의 | 별도 라운드 |

| 구독 상세 바텀시트 -> 풀스크린 전환 | 현재 상세 진입, 딥링크, 편집 연동이 바텀시트 기준으로 구성돼 있다 | 네비게이션, 딥링크, QA 범위 큼 | UX 결정 필요 | 발표 이후 권장 |
| UsageStats 권한 모달 탈출 경로 추가 | 현재 서비스 흐름상 필수 권한이며, 탈출 경로 추가 시 가입 완료 후 어디로 보낼지 정의가 필요하다 | 온보딩 전환, 분석 진입 규칙 | 기획 합의 | 별도 라운드 |
| 오렌지 역할 재정의 + 전체 컬러 체계 정리 | CTA, badge, 탭, progress, banner 전역 영향을 준다 | 전체 앱 | 디자인 시스템 재정의 | 발표 이후 권장 |
| Pretendard 적용 후 잔여 QA | 버튼, 탭바, 프로모션 카드 중심으로 1차 줄바꿈 방지 처리는 반영했지만 전체 화면 QA는 추가로 필요하다 | 전체 앱 | 폰트 적용 후 스모크 테스트 + 화면별 QA | 이번 라운드 반영 |

### 5-2. 해석

- 위 항목들은 `디자인상 맞는 방향`과 `지금 바로 안전하게 넣을 수 있는가`를 분리해서 봐야 한다.
- 특히 대시보드 전면 개편, 전월 대비 데이터, 캐릭터 인사이트는 효과는 크지만 이번 라운드에서 섣불리 들어가면 회귀 위험이 더 크다.
- 이번 라운드에서는 `lite 버전`으로 최대한 효과를 먼저 챙기고, 구조 변경은 별도 태스크로 자르는 편이 맞다.

## 6. 추천 실행 순서
