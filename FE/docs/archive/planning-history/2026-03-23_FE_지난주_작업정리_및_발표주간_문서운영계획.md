## 3. 지난주 작업 내용 요약

지난주 문서 기준으로 정리된 완료 축은 아래와 같다.

- 마이페이지 설정 실연동
- FCM 토큰 등록과 알림함 실연동
- 프로모션 목록/상세 구현
- 구독 관리 실API 전환
- 체크인 응답 실API 전환
- `SCR-002` 분석중 화면 구현

지난주 종료 시점 기준 잔여 리스크는 아래 4개로 정리돼 있었다.

- `223` Naver 버튼 준비중 처리
- `228` 구독 확인 화면 구현
- `229` OTT 6종 노출/정렬/브랜딩 정합성
- `231` 구독 상세 CTA 슬롯 확장과 해지 지원 연결

즉 발표주간에는 "새 기능 확장"보다 "핵심 플로우 연결, QA, 버그 정리, 화면 완성도" 중심으로 문서를 운영하는 편이 맞다.

## 4. 발표주간 문서 운영 축

이번 주 FE 문서는 아래 5개 축으로 관리한다.

### 4-1. QA

- 목적: 핵심 사용자 플로우 종단 테스트와 결함 리스트 관리
- 남길 문서 성격:
  - 테스트 계정/데이터셋 정리
  - 온보딩 → 분석 → 구독확인 E2E
  - 대시보드 → 수동확인 → 구독상세 → 해지지원 E2E
  - 체크인 → 알림함 → 프로모션 E2E
  - 결함 우선순위 분류
- 원칙:
  - 테스트 케이스와 결함 분류표는 루트 유지 후보
  - 실행 로그나 일회성 메모는 `archive/planning-history`

### 4-2. Bugfix

- 목적: 발표 전 `P0/P1` 버그 수정과 회귀 대응
- 남길 문서 성격:
  - 로그인/권한/네비게이션 버그
  - API 실패/재시도/에러 토스트 정책
  - 상태 동기화/새로고침/중복 클릭 이슈
  - 회귀 테스트 결과
- 원칙:
  - 최종 남길 것은 "이슈 목록 + 처리 상태 + 회귀 결과" 문서
  - 개별 수정 메모는 `archive/change-history`

### 4-3. Subscription

- 목적: 구독확인 → 대시보드 → 해지지원 흐름 마감
- 남길 문서 성격:
  - `SCR-002b` 구독 확인
  - OTT 6종 정합성
  - 구독 상세 CTA 확장
  - `SCR-008` 해지 지원 진입 흐름
- 원칙:
  - 화면 흐름 기준 문서는 루트 유지
  - 단일 구현 순서 메모나 프롬프트는 `archive/planning-history`

### 4-4. UX

- 목적: 발표용 화면 폴리싱
- 남길 문서 성격:
  - 문구/라벨/톤앤매너 정리
  - 여백/정렬/버튼 상태/UI 일관성
  - Loading/Empty/Error 상태 통일
  - 최종 디자인 QA 반영
- 원칙:
  - 최종 합의된 기준 문서만 루트 유지
  - 화면별 수정 내역은 `archive/change-history`

### 4-5. Release

- 목적: 실기기 회귀 테스트와 발표 빌드 점검
- 남길 문서 성격:
  - 발표용 계정/환경값/네트워크 조건
  - 실기기 및 에뮬레이터 회귀 테스트
  - 세션 만료/권한 거부/네트워크 끊김 예외 점검
  - 발표용 빌드 체크리스트
- 원칙:
  - 최종 실행 체크리스트는 루트 유지
  - 회차별 실험 로그는 `archive/planning-history`

## 5. 이번 주 문서 운영 규칙

- 루트에는 "오늘 바로 다시 열 문서"만 둔다.
- 테스트 기준 문서, 발표 체크리스트, 최종 QA 기준만 루트 유지 후보로 본다.
- 실행 순서 정리, 프롬프트, 임시 TODO 문서는 작업 종료 직후 `archive/planning-history`로 이동한다.
- 화면 수정 내역은 기능 기준 문서가 아니라면 `archive/change-history`로 이동한다.
- 설계 초안은 현재 구현을 직접 움직이지 않으면 `archive/design-drafts`로 이동한다.

1. QA, Bugfix, Release는 `2026-03-23_FE_발표주간_QA_및_회귀_테스트계획.md` 기준으로 통합 운영한다.
2. Subscription, UX는 `2026-03-23_FE_발표주간_화면_폴리싱_계획.md` 기준으로 통합 운영한다.
3. 결함 리스트와 회귀 결과는 분리하지 말고 같은 QA 축에서 관리한다.
