# FE 초기세팅 작업 정리 (v1.5 기준 보완)

## 1. 작업 목적

`com.ssafy.e106` 패키지 기준 Android FE 프로젝트를 기능 개발 착수 가능한 상태까지 끌어올리는 것을 목표로 했다.

핵심 목표:

- 빌드/의존성 체인 안정화
- 앱 진입점/딥링크/알림 채널/FCM 기본 경로 확보
- 토큰 저장/주입/갱신(Auth) 보일러플레이트 구현
- 네비게이션 골격(App/Auth/Home/Settings) 구현
- 릴리즈 보안 옵션(minify/shrink/proguard) 적용 및 검증

## 2. 최종 상태 요약

### 2-1. 완료

- 프로젝트 베이스: Kotlin DSL, `minSdk 26`, `namespace/applicationId=com.ssafy.e106`
- 빌드 구성: Gradle catalog + app/release 설정 + proguard 규칙 적용
- App 부트스트랩: `@HiltAndroidApp`, `@AndroidEntryPoint`, NotificationChannel 생성, `TokenRepository.initialize()` 연결
- 네비게이션 골격: `AppNavHost`, `AuthNavGraph`, `HomeNavGraph`, `SettingsNavGraph`, 타입세이프 Route
- 토큰/Auth 골격: `TokenDataStore`, `TokenRepository`, `AuthInterceptor`, `TokenAuthenticator`, `AuthRepository`
- API 인터페이스: `AuthApi`, `UserApi` 최소 세트
- FCM 골격: `AekkimFirebaseMessagingService` + Manifest 서비스 등록 + 딥링크 스킴 연결
- 검증: debug/release 빌드 및 핵심 task 수행 성공

### 2-2. 현재 단계 정의

현재는 초기세팅 완료 + 기능 구현 준비 완료 단계다. 소셜 로그인 bootstrap은 v1.5 기준으로 provider-neutral 구조(`ProviderAuthCredential` -> JWT 교환)로 정렬되어 있으며, Google 구현 세부/BE 계약은 `[추가 필요]`, `[협의 필요: FE/BE]`로 분리 관리한다.

- 이미 된 것: 런타임/네트워크/보안/알림/네비 기반 골격
- 아직 남은 것: 각 feature 실제 비즈니스 UI/상태/API 플로우 구현
- 추가 정리 필요: 소셜 로그인 provider 관련 스캐폴드 리네이밍 및 계약 정렬

## 3. 상세 반영 내역

### 3-1. 빌드/의존성/릴리즈 보안

- `FE/gradle/libs.versions.toml`
  - `securityCrypto` 버전/라이브러리 추가
- `FE/app/build.gradle.kts`
  - release에 `isMinifyEnabled = true`, `isShrinkResources = true` 적용
  - `implementation(libs.security.crypto)` 추가
- `FE/app/proguard-rules.pro`
  - Retrofit, kotlinx-serialization, Auth, Firebase 최소 규칙 작성

### 3-2. 앱 진입점/Manifest

- `FE/app/src/main/java/com/ssafy/e106/app/App.kt`
  - NotificationChannel 2종 생성
  - `TokenRepository.initialize()` 앱 시작 시점 호출
  - provider SDK 직접 초기화 제거 (provider-specific bootstrap 분리)
- `FE/app/src/main/java/com/ssafy/e106/app/MainActivity.kt`
  - `AppNavHost(entryIntent = entryIntent)` 적용
  - `onNewIntent()` 구현으로 딥링크 재진입 처리
- `FE/app/src/main/AndroidManifest.xml`
  - `aekkim` 딥링크 intent-filter 추가
  - `AekkimFirebaseMessagingService` 등록
  - 권한: `INTERNET`, `PACKAGE_USAGE_STATS`, `POST_NOTIFICATIONS` 유지

### 3-3. 네비게이션 골격

- `Routes.kt`
  - `AuthRoute`, `HomeRoute` 타입세이프 라우트 정의
  - `CheckIn(subscriptionId, isFullScreen)` 구조 반영
- `AppNavHost.kt`
  - 루트 NavHost + deepLink 처리
- `AuthNavGraph.kt`
  - 온보딩/약관/분석로딩/구독확인 경로 골격
- `HomeNavGraph.kt`
  - 대시보드/프로모션/체크인/해지가이드/수동매핑/마이페이지 경로 골격

### 3-4. 토큰/Auth/네트워크 보일러플레이트

- `TokenDataStore.kt`
  - `EncryptedSharedPreferences` + `MasterKey` 기반 Access/Refresh 저장
- `TokenRepository.kt`
  - 메모리 캐시 + 동기 조회 API
- `AuthInterceptor.kt`
  - 동기 캐시에서 Bearer 헤더 주입
- `TokenAuthenticator.kt`
  - 401 시 refresh 재발급 시도
- `NetworkModule.kt`
  - Json/OkHttp/Retrofit/AuthApi/UserApi Hilt 제공
- `AuthRepository.kt`
  - `exchangeProviderCredential(...)` + refresh 호출 처리
  - provider 인증값(`authorizationCode` 또는 `providerToken`)과 앱 JWT(access/refresh)를 분리해 다룸
  - `exchangeGoogleToken()` → `Result<LoginResponse>` (이전 `Result<TokenPair>` 아님). `LoginResponse`는 `accessToken` + `LoginUser`를 포함하며, refresh와 구분된다. 상세는 `FE/docs/2026-03-10_FE_소셜로그인_관련_변경내용_정리.md` 참조.
- `AuthApi.kt`, `UserApi.kt`
  - 인증/프로필 최소 API 인터페이스 정의
  - provider token과 app access/refresh token을 분리해 명명해야 함

### 3-5. FCM 골격

- `AekkimFirebaseMessagingService.kt`
  - `CHECK_IN`, `CANCEL_REVIEW` 타입 분기
  - 딥링크 Intent 생성
    - `aekkim://checkin/{subscriptionId}?isFullScreen=true`
    - `aekkim://cancel/{subscriptionId}`

## 6. 주의사항 / 리스크

1. `android.overridePathCheck=true`는 Windows 경로 이슈 회피용 실험 옵션
2. `Theme.kt`의 `statusBarColor`는 deprecated warning 존재(동작 영향 없음)
3. `EncryptedSharedPreferences`는 장기적으로 `DataStore + Tink` 전환 검토 필요
4. `google-services.json` 실파일은 보안상 커밋 금지(`.gitignore` 처리)
5. Feature별 실제 화면/상태/도메인 로직은 아직 placeholder 중심
6. 소셜 로그인 교환 endpoint/요청 필드는 임시 placeholder 상태이며 기준 문서상 미정이다. 코드에는 TODO로 남기고 `[협의 필요: FE/BE]`로 관리한다.
7. Android Google 로그인 실제 방식(`Google 로그인 SDK`, `Credential Manager` 포함 여부), 결과값 형식, BE 교환 endpoint는 `[추가 필요]` 또는 `[협의 필요: FE/BE]`다.
