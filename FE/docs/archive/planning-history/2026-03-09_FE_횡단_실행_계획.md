# FE 횡단 실행 계획

> 최초 작성: 2026-03-09  
> 최신 개정: 2026-03-15  
> 목적: 스토리 순서 나열이 아니라, 공통 선행작업과 FE 단독 가능 작업, BE 의존 작업을 분리한 실행계획으로 재구성한다.

## 1. 현재 실행 전제

- 실연동 완료 영역: 인증, 사용자 조회/로그아웃/탈퇴
- 실연동 미완 영역: 구독, 프로모션, 알림
- FE 우선 전략: `실연동 가능한 것 먼저 마감 -> contract-first 준비 -> mock UI 완료`

## 2. P0 공통 선행작업

| 구분 | 작업 | 근거 | 담당 추천 |
| --- | --- | --- | --- |
| 계약 | 공통 체크리스트를 최신 인증/알림 계약으로 고정 | 기존 FE 문서가 구버전 계약을 사용한다. | 공통 |
| 네비게이션 | placeholder route 정리 및 알림함 route 추가 | 현재 route 누락과 placeholder가 많다. | FE |
| 데이터 | mock 모델과 real DTO 분리 | 구독/알림/프로모션 API가 미구현 상태다. | FE |
| QA | 완료 정의와 실패 롤백 기준 통일 | 토글 저장, 체크인 제출, 알림 읽음에 공통 적용 필요 | 공통 |
| BE 선행 | provider 필드, FCM token API, 구독/알림 최소 endpoint 우선순위 확정 | 화면명세와 API/코드가 어긋난다. | BE |

## 3. FE 단독 가능 작업

### 3-1. 즉시 착수

- Story1
  - Naver 준비중 처리
  - Google 로그인 실패/취소 QA 정리
  - 약관/권한 문구 정리
- Story2
  - UsageStats 브리지 화면 정리
  - 분석 로딩 UI 또는 직접 진입 정책 확정
- Story3
  - 구독 확인 화면 목업 구현
  - OTT 6종 fixture 정리
  - 대시보드 상태 화면 보강
- Story5
  - 체크인/해지 가이드/알림함 목업 UI 구현
  - enum/payload 정본 설계 문서 반영

### 3-2. 조건부 착수

- Story4 provider 배지 노출은 API 확정 전까지 임시 placeholder 정책만 정리
- Story5 알림함은 화면 ID 합의 전까지 임시 route로 개발

## 4. BE 의존 작업

| 우선순위 | 작업 | FE 영향 |
| --- | --- | --- |
| P0 | `PATCH /users/me/notification-settings`, `PATCH /users/me/optional-consent` 검증 완료 | Story4 완료 여부 결정 |
| P0 | `POST /api/v1/fcm-tokens` 계약 및 구현 | Story1/5 토큰 등록 완료 여부 결정 |
| P1 | 서비스 목록/요금제/내 구독 목록 최소 API | Story3 mock 제거 시작점 |
| P1 | 체크인 제출/이력 조회 API | Story5 실연동 시작점 |
| P2 | 알림 목록/읽음 처리 API | 알림함 실연동 시작점 |
| P2 | 프로모션 목록/상세 API | 추천 화면 실연동 시작점 |

## 5. 추천 작업 순서

1. Story1 로그인/온보딩 계약 마감
2. Story4 마이페이지 PATCH 저장 완료
3. Story3 구독 확인/대시보드 목업 정합성 완료
4. Story5 FCM 규약/알림함 설계 완료
5. Story2 브리지 화면 정리
6. 구독/알림/프로모션 실API 연결 전환

## 6. 팀 분업 포인트

- FE-A
  - Story1, Story2
- FE-B
  - Story4
- FE-C
  - Story3 목업 정합성, Story5 알림함/체크인 UI
- BE
  - 사용자 PATCH 검증, FCM token API, 구독 최소 API, 알림 최소 API
- 공통
  - enum/payload 계약, 화면 정의 누락분 보정, QA 기준 통일

## 7. 위험 관리

- 구독/알림/프로모션 API 미구현 상태를 완료로 오인하지 않는다.
- provider 필드가 없으므로 마이페이지 하드코딩을 최종 상태로 인정하지 않는다.
- 알림 enum/payload가 정리되기 전까지 FCM 기능 개발을 확장하지 않는다.
- 알림함 화면 정의가 없으면 임시 route를 쓰되 문서에 반드시 표시한다.
