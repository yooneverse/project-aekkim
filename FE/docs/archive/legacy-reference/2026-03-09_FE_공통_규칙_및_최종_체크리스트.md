# FE 공통 규칙 및 최종 체크리스트

> 최초 작성: 2026-03-09  
> 최신 개정: 2026-03-15  
> 이 문서는 기존 초안의 구조만 유지하고, 내용은 2026-03-15 기준 문서와 현재 코드 상태를 반영해 재작성했다.

## 1. 문서 목적

- FE 전 스토리에 공통 적용할 계약, 구현 원칙, 완료 기준을 최신 기준으로 고정한다.
- 기존 계획서가 오래된 API/화면 기준으로 오염되지 않도록 정본 체크리스트 역할을 한다.

## 3. 공통 계약

### 3-1. 인증

- Google 로그인: `POST /api/v1/auth/login/google`
- 요청 body: `googleIdToken`
- Kakao 로그인: `POST /api/v1/auth/login/kakao`
- 요청 body: `kakaoAccessToken`
- 토큰 재발급: `POST /api/v1/auth/refresh`
- 로그아웃: `POST /api/v1/auth/logout`
- 구버전 계약인 `POST /api/v1/auth/login`과 `googleAccessToken`은 사용 금지다.

### 3-2. 사용자 설정

- 내 정보 조회: `GET /api/v1/users/me`
- 알림 설정 변경: `PATCH /api/v1/users/me/notification-settings`
- 선택 동의 변경: `PATCH /api/v1/users/me/optional-consent`
- 회원탈퇴: `DELETE /api/v1/users/me`
- 마이페이지 토글은 로컬 상태만 바꾸면 완료가 아니다. 즉시 저장, 실패 롤백, 재시도까지 포함해야 한다.

### 3-3. 알림/FCM

- 알림 타입 정본: `CHECKIN`, `PROMO`, `CHURN_REVIEW`
- 알림 payload 기준 식별자: `referenceId`
- `CHECKIN`/`CHURN_REVIEW`는 `subscriptionId`, `PROMO`는 `promotionId`를 `referenceId`로 사용한다.
- 기존 FE 셸의 `CHECK_IN`, `CANCEL_REVIEW`, `subscriptionId` 단독 규약은 폐기 대상이다.
- FCM 토큰 등록, 알림 목록, 읽음 처리 API는 MVP 범위다. 다만 현재 BE 구현은 미완이다.

### 3-4. 서비스 범위

- MVP OTT 범위는 6종이다.
- 대상 서비스: Netflix, Tving, Wavve, Coupang Play, Disney+, Watcha
- 수동 구독 추가, 서비스 선택, 목업 fixture 모두 6종 기준으로 맞춘다.

## 4. 공통 구현 원칙

### 4-1. Route와 화면 상태

- placeholder route는 임시 상태로만 유지한다.
- `AuthRoute.AnalysisLoading`, `AuthRoute.SubscriptionConfirm`, `HomeRoute.PromotionList`, `HomeRoute.PromotionDetail`, `HomeRoute.CheckIn`, `HomeRoute.CancelGuide`는 완료로 간주하지 않는다.
- 알림함 route는 신규 추가 대상이다.

### 4-2. mock과 real 분리

- 현재 실연동 가능 영역은 인증/사용자 API다.
- 구독/프로모션/알림은 contract-first + mock 병행 전략을 사용한다.
- mock 모델은 화면 개발용이며 real DTO와 분리된 이름을 유지한다.
- 문서에는 각 태스크별로 `목업 가능`, `BE 확정 필요`, `실연동 가능`을 명시한다.

### 4-3. 상태 관리

- feature 단위 `UiState`, `UiEffect`, `Intent` 패턴을 유지한다.
- 저장형 액션은 `loading`, `success`, `error`, `retry`를 모두 가진다.
- 토글 저장, 체크인 제출, 알림 읽음 처리에는 optimistic update 후 실패 롤백 여부를 문서에 명시한다.

### 4-4. 디자인 시스템

- 공통 컴포넌트 사용 우선순위: `AppButton`, `AppCard`, `AppAgreementRow`, `AppBottomSheet`, `AppBottomTabBar`, `AppLoadingState`, `AppErrorState`, `AppEmptyState`
- 화면별 Loading/Error/Empty 상태는 컴포넌트 명세에 맞춰 별도 정의한다.
- UI 뼈대만 존재하면 완료가 아니다. CTA, 상태값, 예외, 바텀시트/풀스크린 구조까지 맞아야 한다.

## 5. 완료 판정 기준

- 코드, 화면, API 연동, 상태 처리, 예외 처리, 문서 정합성을 모두 충족해야 `완료`다.
- 단순 UI 스켈레톤은 `진행 중`이다.
- 실API가 없어도 목업/더미/로컬 상태로 충분히 검증 가능하면 `FE 단독 가능`이다.
- 최신 화면/디자인/API 기준과 어긋나면 구현되어 있어도 `보완 필요`다.
- 최신 범위에서 빠진 기능은 `삭제` 또는 `보류`로 명확히 표시한다.

## 6. 최종 체크리스트

### 6-1. 계약 체크

- [ ] 인증 엔드포인트와 body 키가 최신 API 문서와 일치한다.
- [ ] 사용자 설정 API와 화면 토글 항목이 일치한다.
- [ ] 알림 enum, payload 키, 딥링크 라우팅이 SRS와 일치한다.
- [ ] OTT 서비스 범위가 6종으로 통일되어 있다.

### 6-2. 화면 체크

- [ ] 화면명세의 진입 경로, CTA, Empty/Loading/Error가 반영되어 있다.
- [ ] 바텀시트/풀스크린/탭바 구조가 화면명세와 일치한다.
- [ ] 알림함처럼 화면명세에 빠진 항목은 `[화면 정의 추가 필요]`로 분리되어 있다.

### 6-3. 구현 체크

- [ ] 실연동 가능 API는 실제 Repository에 연결되어 있다.
- [ ] 실연동 불가 영역은 mock 전략과 제거 시점이 문서에 적혀 있다.
- [ ] 토글/저장/삭제 액션은 실패 시 롤백 또는 재시도 UX를 가진다.
- [ ] 로그아웃/회원탈퇴 후 세션 정리 범위가 명확하다.

### 6-4. QA 체크

- [ ] 성공/실패/취소/권한 거부 시나리오가 문서에 있다.
- [ ] 권한 거부 시 허용/차단 규칙이 정리되어 있다.
- [ ] 목업 데이터와 실데이터 전환 시 DTO/UiModel 매핑 기준이 있다.

## 7. 즉시 수정이 필요한 공통 이슈

- `[문서 충돌]` 기존 FE 문서의 인증 계약은 전면 교체한다.
- `[설계 수정 필요]` 알림 enum/payload 규약을 최신 기준으로 재설계한다.
- `[추가 필요]` 알림함 route와 화면 정의를 횡단 태스크에 포함한다.
- `[BE 확정 필요]` 마이페이지 provider 배지 노출용 필드가 API에 없다.
