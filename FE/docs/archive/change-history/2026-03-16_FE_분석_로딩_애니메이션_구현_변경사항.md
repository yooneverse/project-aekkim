## 1. 개요

구독 분석 진행 중 사용자에게 시각적 피드백을 제공하는 **햄찌 로딩 애니메이션 화면**을 구현했다.  
온보딩 완료 후 `Terms → AnalysisLoading → SubscriptionConfirm` 흐름에서 분석이 진행되는 동안  
햄찌 캐릭터가 좌우로 달리며 혜택을 찾는 연출을 보여준다.

### 핵심 변경 요약

- 스프라이트 시트 기반 프레임 애니메이션 (24fps, 145프레임)
- 사용 기록 권한 분기 처리 (UNKNOWN / DENIED / GRANTED) 3화면
- MVI 아키텍처 기반 3단계 분석 진행 상태 관리
- 절약 금액 실시간 카운트업 애니메이션
- 범용 스프라이트 애니메이션 컴포넌트 (`core/ui/component/animation/`)

## 3. 핵심 구현 상세

### 3-1. 스프라이트 시트 애니메이션 (AnalysisLoadingScreen.kt)

분석 화면의 `AnimatedHamzziSprite`는 단일 스프라이트 시트 PNG에서 프레임을 잘라 Canvas에 렌더링한다.

```
hamzzi_sprite.png 스펙:
- 레이아웃: 12열 × 13행 (마지막 행은 일부만 사용)
- 프레임 크기: 96 × 96 px
- 총 프레임 수: 145
- 재생 속도: 24 fps (프레임당 약 41.6ms)
```

**동작 흐름:**
1. `rememberHamzziSpriteSheet()` — assets에서 `BitmapFactory.decodeStream()`으로 전체 스프라이트 시트를 `ImageBitmap`으로 로드 (`remember` 캐싱)
2. `LaunchedEffect` — `1000/24 ≈ 41ms` 간격으로 프레임 인덱스를 0~144 순환
3. `Canvas.drawImage()` — `srcOffset`과 `srcSize`로 현재 프레임 영역만 잘라서 렌더링

```kotlin
// 프레임 좌표 계산
val column = frame % 12       // 0~11
val row    = frame / 12       // 0~12
srcOffset = IntOffset(x = column * 96, y = row * 96)
srcSize   = IntSize(width = 96, height = 96)
```

### 3-2. 좌우 이동 + 바운스 애니메이션 (HamsterRunwayCard)

`rememberInfiniteTransition`으로 두 개의 연속 애니메이션을 조합한다.

| 애니메이션 | 주기 | 값 범위 | 역할 |
|---|---|---|---|
| `phase` | 3,600ms | 0 → 2 (Restart) | 좌우 왕복 이동. 0→1 정방향, 1→2 역방향 |
| `bounce` | 360ms | 0 → 1 (Restart) | sin 곡선 바운스, 진폭 10dp |

`graphicsLayer`에서 최종 적용:
- `translationX` = 이동 거리 × normalizedPhase (카드 너비 - 햄찌 크기)
- `translationY` = -abs(sin(bounce × π)) × 10dp (위로 튀기)
- `scaleX` = 이동 방향에 따라 -1 또는 1 (좌우 반전)

### 3-3. MVI 아키텍처 연동 (AnalysisViewModel)

```
Intent                   State 변화                     Effect
────────────────────────────────────────────────────────────────────
Entered                → permissionState 확인
  ├─ DENIED            → permissionState=DENIED          (화면 분기)
  └─ GRANTED           → isCollecting=true
                         currentStep=FetchingPayments
                         ──(550ms)──
                         currentStep=MatchingBenefits      savingAmount=2,400
                         completedSteps={FetchingPayments}
                         ──(UsageStats 수집)──
                         currentStep=BuildingReport
                         completedSteps+={MatchingBenefits}
                         savingAmount=(계산값)
                         ──(800ms)──
                         isCollecting=false
                         completedSteps+={BuildingReport}
                         ──(700ms)──
                                                         → NavigateToSubscriptionConfirm

OpenSettingsClicked                                      → OpenUsageAccessSettings
RecheckPermissionOnResume → 권한 재확인 후 위 흐름 반복
ContinueClicked                                          → NavigateToSubscriptionConfirm
```

### 3-4. 절약 금액 추정 로직

```kotlin
val activeItems  = items.count { lastUsed != null || usage30d > 0 || usage7d > 0 }
val recentItems  = items.count { usage7d > 0 }
val candidate    = (activeItems × 3,200) + (recentItems × 900)
val savingAmount = candidate.coerceIn(0, 18_700)
```

`animateIntAsState(tween 700ms)`로 화면에 카운트업 애니메이션 표시.

## 5. 화면별 동작 흐름

### 5-1. 전체 네비게이션

```
Onboarding → Terms → AnalysisLoading → SubscriptionConfirm → Dashboard
                         ↑ 이 화면
```

### 5-2. AnalysisLoading 내부 상태 분기

```
┌─────────────────────────────────┐
│       AnalysisLoadingScreen     │
│                                 │
│  permissionState == UNKNOWN     │──→  AnalysisCheckingState
│   (진입 직후, 권한 확인 중)       │     - 햄찌 달리기 + "출발 준비 중" 메시지
│                                 │
│  permissionState == DENIED      │──→  PermissionRequiredState
│   (사용 기록 접근 거부)           │     - 햄찌 달리기 + 권한 안내
│                                 │     - "설정으로 이동" 버튼
│                                 │     - ON_RESUME 시 자동 재확인
│                                 │
│  permissionState == GRANTED     │──→  AnalysisProgressState
│   (권한 허용됨)                   │     - 햄찌 달리기 + 단계별 진행 카드
│                                 │     - 절약 금액 카운트업
│                                 │     - 프로그레스 바 (0.24→0.62→0.88→1.0)
│                                 │     - 단계별 순환 메시지 (1,050ms 간격)
│                                 │     - 완료 후 자동 이동 or "바로 확인하기"
└─────────────────────────────────┘
```

### 5-3. 분석 3단계

| 단계 | 이름 | UI 표시 | 진행률 |
|---|---|---|---|
| 1 | FetchingPayments | "결제 흔적 확인 완료" | 24% |
| 2 | MatchingBenefits | "할인 혜택 매칭 중" | 62% |
| 3 | BuildingReport | "절약 리포트 생성" | 88% → 100% |

각 단계 카드는 3가지 시각 상태를 가짐:
- **Completed** — 녹색 배경 + 체크 아이콘
- **Active** — 주황색 배경 + 숫자 아이콘
- **Pending** — 회색 배경 + 숫자 아이콘 (반투명)

## 6. 범용 애니메이션 컴포넌트 (core/ui/component/animation/)

분석 화면 외에도 재사용 가능한 애니메이션 컴포넌트를 함께 추가했다.

### SpriteAnimation

assets 폴더의 개별 프레임 이미지를 순서대로 재생하는 범용 Composable.

- `produceState` + `Dispatchers.IO`로 비동기 비트맵 로딩 (메인 스레드 블로킹 방지)
- PNG / WebP / JPG 형식 지원
- 파일명 정렬 순서대로 재생

```kotlin
SpriteAnimation(
    assetFolder = "animations/hamster_run",
    size = 120.dp,
    frameIntervalMs = 80L,
)
```

### HamsterRunAnimation

`SpriteAnimation`의 햄찌 달리기 전용 래퍼. 폴더 경로와 기본 속도가 프리셋되어 있다.

```kotlin
HamsterRunAnimation(size = 120.dp)
```

> `assets/animations/hamster_run/` 폴더에 프레임 이미지(`frame_00.webp`, `frame_01.webp`, ...)를 넣으면 동작한다.

### HamsterImage

Coil + `SvgDecoder`로 정적 SVG를 렌더링하는 Composable.

```kotlin
HamsterImage(size = 200.dp)
```

## 7. 분석 단계 타이밍 개선

### 문제

데이터가 없는 환경(에뮬레이터, OTT 미설치 기기)에서 Step 2(MatchingBenefits)의
`UsageStatsManager` 쿼리가 즉시 리턴되어 전체 로딩이 ~2초 만에 종료됨.
단계별 카드 전환과 햄찌 달리기 애니메이션이 거의 보이지 않는 문제.

### 해결

`AnalysisViewModel`에 **단계별 최소 유지 시간**을 도입.

```kotlin
private companion object {
    const val MIN_STEP_DURATION_MS = 1_500L   // 각 단계 최소 1.5초
    const val POST_COMPLETE_DELAY_MS = 700L   // 완료 후 이동 대기
}
```

Step 2에서는 `measureTimeMillis`로 실제 수집 소요시간을 측정하고,
`max(0, 1500 - 실제소요)` 만큼만 추가 대기하여 불필요한 지연 없이 최소 시간을 보장.

### 타이밍 비교

| 단계 | Before | After |
|---|---|---|
| Step 1 (FetchingPayments) | 550ms 고정 | 1,500ms 최소 보장 |
| Step 2 (MatchingBenefits) | ~0ms (데이터 없을 때) | max(실제 수집, 1,500ms) |
| Step 3 (BuildingReport) | 800ms 고정 | 1,500ms 최소 보장 |
| 완료 → 이동 | 700ms | 700ms |
| **합계 (데이터 없을 때)** | **~2,050ms** | **~5,200ms** |

데이터가 충분한 실기기에서는 Step 2 수집이 1.5초 이상 걸리면
추가 대기 없이 바로 다음 단계로 넘어가므로 체감 지연이 없다.
