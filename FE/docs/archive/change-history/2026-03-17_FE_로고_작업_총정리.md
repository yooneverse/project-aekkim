# FE 로고 작업 총정리

> 작성일: 2026-03-17  
> 범위: 로고 원본 선정, Android 리소스 최적화, 대시보드 적용, 컴파일 검증

## 1. 작업 목적

대시보드 좌상단에 임시 텍스트로 노출되던 브랜드 표식을 실제 애낌(AEKKIM) 로고 이미지로 교체하고, Android 단말 밀도별 리소스를 최적화해 화면 품질과 앱 용량을 함께 맞추는 것이 이번 작업의 목적이었다.

## 2. 최종 결과 요약

- 대시보드 헤더의 `"AE"` 플레이스홀더를 `logo_aekkim.webp` 이미지로 교체했다.
- Android `drawable-mdpi` ~ `drawable-xxxhdpi` 5개 밀도 버킷에 로고 리소스를 배치했다.
- 원본 PNG 966,606B 기준으로 앱 포함 리소스 총량을 약 19.5KB 수준으로 줄였다.
- `DashboardScreen.kt`에서 `Modifier.height(40.dp)` 기준으로 로고 비율이 유지되도록 적용했다.
- `:app:compileDebugKotlin` 기준 컴파일 성공까지 확인했다.

## 3. 작업 범위

### 원본 자산

- 원본 파일: `FE/prototype/화면 캡처 2026-03-03 152858.png`
- 원본 용량: 966,606B
- 원본 해상도: 1920 x 1080

### 앱 반영 대상

- 화면: 대시보드 헤더
- 코드 위치: `app/src/main/java/com/ssafy/e106/feature/dashboard/ui/DashboardScreen.kt`
- 리소스 이름: `logo_aekkim`

## 4. 리소스 최적화 결과

원본 이미지는 모바일 UI에서 직접 사용하기에는 해상도와 용량이 과도했다. 그래서 여백을 정리한 뒤, 40dp 표시 기준에 맞춰 밀도별 사이즈로 리사이징하고 WebP로 변환해 배치했다.

| 버킷 | 해상도 | 파일 크기 |
|---|---:|---:|
| `drawable-mdpi/logo_aekkim.webp` | 40 x 19 | 1,334B |
| `drawable-hdpi/logo_aekkim.webp` | 60 x 28 | 2,202B |
| `drawable-xhdpi/logo_aekkim.webp` | 80 x 38 | 3,316B |
| `drawable-xxhdpi/logo_aekkim.webp` | 120 x 57 | 5,456B |
| `drawable-xxxhdpi/logo_aekkim.webp` | 160 x 76 | 7,646B |
| 합계 | - | 19,954B |

정리하면, 원본 PNG 1장 대신 디바이스 해상도에 맞는 WebP 5종을 두는 구조로 변경했고, 런타임에서는 Android가 적절한 버킷을 자동 선택한다.

## 5. 코드 반영 내용

대시보드 헤더에서는 텍스트형 플레이스홀더 대신 `Image`와 `painterResource`를 사용하도록 변경했다.

현재 반영 상태:

```kotlin
Image(
    painter = painterResource(id = R.drawable.logo_aekkim),
    contentDescription = "애낌 로고",
    modifier = Modifier.height(40.dp),
    contentScale = ContentScale.Fit,
)
```

적용 위치:

- `app/src/main/java/com/ssafy/e106/feature/dashboard/ui/DashboardScreen.kt:264`

구현 포인트:

- `height(40.dp)`로 세로 기준 크기를 고정했다.
- `ContentScale.Fit`으로 원본 비율이 찌그러지지 않도록 유지했다.
- 리소스 이름을 모든 버킷에서 동일하게 맞춰 Android 기본 리소스 선택 규칙을 사용했다.

## 6. 작업 중 이슈와 해결

로고 교체 후 `DashboardScreen.kt`에서 아래 컴파일 오류가 발생했다.

```text
Unresolved reference 'height'
```

원인은 Compose layout 확장 함수 import 누락이었다.  
해결 방식은 `androidx.compose.foundation.layout.height` import를 추가하는 것이었고, 이후 Kotlin 컴파일이 정상 통과했다.

주의할 점:

- `height`는 import가 필요하다.
- `weight`는 이 파일에서 scope 확장으로 해석되므로 별도 import를 추가하면 오히려 잘못된 심볼과 충돌할 수 있다.

## 7. 검증 결과

검증 명령:

```powershell
gradlew.bat :app:compileDebugKotlin
```

검증 결과:

- 컴파일 성공
- 로고 적용 코드와 import 구성이 현재 기준으로 정상
- deprecated warning 몇 건은 남아 있지만 이번 로고 작업과는 무관

## 8. 유지보수 체크리스트

- 로고 교체 시 리소스 이름은 `logo_aekkim`으로 유지하는 편이 코드 변경 범위를 최소화한다.
- 새 로고를 넣을 때는 단일 대형 PNG를 그대로 넣지 말고 현재처럼 밀도별 WebP 리소스로 다시 생성하는 것이 좋다.
- 헤더 높이를 바꾸면 로고의 실제 표시 폭도 함께 달라지므로 `height(dp)` 기준으로 시안 확인이 필요하다.
- 다른 화면에도 브랜드 로고가 들어갈 예정이면 공통 헤더 컴포넌트로 추출하는 것이 중복 감소에 유리하다.
