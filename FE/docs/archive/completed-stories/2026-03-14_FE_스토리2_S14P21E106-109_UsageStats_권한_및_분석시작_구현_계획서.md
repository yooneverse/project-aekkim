# FE 스토리2 - S14P21E106-109 UsageStats 권한 및 분석시작 구현 계획서

> 최초 작성: 2026-03-09  
> 최신 개정: 2026-03-15  
> 상태 요약: `진행 중`

## 1. 문서 목적

- UsageStats 권한 획득과 분석 시작 흐름을 Story1과 Story3 사이의 브리지 스토리로 재정의한다.

## 2. 최신 범위

- 포함
  - UsageStats 권한 안내
  - 설정 이동 및 복귀 판정
  - 권한 거부 재유도
  - 분석 시작/로딩 상태
  - 구독 확인 화면 진입 연결
- 제외
  - 실제 구독 확인 데이터 처리
  - 대시보드 구현

## 3. 선행조건

- Story1에서 로그인과 약관 동의가 끝난 상태여야 한다.
- UsageStats는 필수 권한이다.
- 알림 권한 거부는 서비스 진행 허용, UsageStats 거부는 대시보드 진입 불가 규칙을 따른다.

## 4. 화면 범위

- `SCR-002` 분석 시작/권한 유도
- `SCR-002b` 구독 확인 화면 진입 전 브리지

## 5. 연동 API

- 없음
- 다만 Story3의 구독 감지 결과와 연결될 수 있도록 route/UiState 계약을 먼저 정의한다.

## 6. 데이터 모델

- `UsageStatsPermissionState`: not requested / denied / granted
- `AnalysisStartState`: idle / loading / complete / blocked
- 브리지 출력: Story3로 넘길 감지 결과 placeholder 모델

## 7. 공통 컴포넌트 의존성

- `AuthFlowScaffold`
- `AppCard`
- `AppButton`
- `AppLoadingState`
- `AppErrorState`

## 9. 완료 기준

- UsageStats 권한 허용 전에는 Story3 대시보드로 진입하지 않는다.
- 설정 복귀 후 granted 여부에 따라 다음 화면이 정확히 갈린다.
- 분석 로딩 또는 구독 확인 연결이 플레이스홀더가 아닌 실제 화면 흐름으로 정의된다.

## 10. QA 포인트

- 권한 첫 거부 후 재유도 다이얼로그
- 설정에서 권한 허용 후 자동 진입
- 설정에서 권한 미허용 유지 후 안내 메시지
- 알림 권한 거부와 UsageStats 거부가 다르게 처리되는지 확인
