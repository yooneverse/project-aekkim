# FE 스토리1 - S14P21E106-11A 온보딩 및 로그인 구현 계획서

> 최초 작성: 2026-03-09  
> 최신 개정: 2026-03-15  
> 상태 요약: `진행 중`

## 1. 문서 목적

- 온보딩, 소셜 로그인, 약관 동의, 알림 권한, UsageStats 권한 진입, 온보딩 종료 저장 정책을 최신 계약 기준으로 정리한다.

## 2. 최신 범위

- 포함
  - 온보딩 화면
  - Google 로그인
  - Kakao 로그인
  - Naver 버튼 준비중 처리
  - 약관 동의
  - 알림 권한 요청
  - UsageStats 권한 진입
  - 선택 동의 서버 저장
  - FCM 토큰 등록 트리거
- 제외
  - 실제 구독 확인 화면 구현은 Story3
  - 상세 알림함은 Story5

## 3. 선행조건

- Google/Kakao 로그인 계약은 최신 인증 API 문서를 따른다.
- 선택 동의 저장은 `PATCH /api/v1/users/me/optional-consent` 기준으로 설계한다.
- FCM 토큰 등록 API는 현재 BE 미구현 상태이므로 `[BE 확정 필요]`로 분리한다.

## 4. 화면 범위

- `SCR-001` 온보딩
- 약관 동의/권한 유도 화면
- Story2로 이어지는 UsageStats 권한 브리지

## 5. 연동 API

- `POST /api/v1/auth/login/google`
- `POST /api/v1/auth/login/kakao`
- `PATCH /api/v1/users/me/optional-consent`
- `[BE 확정 필요] POST /api/v1/fcm-tokens`

## 6. 데이터 모델

- 로그인 응답: `accessToken`, `refreshToken`, `tokenType`, `expiresIn`, 사용자 기본 정보, 알림 설정 값
- 온보딩 상태: 약관 동의 여부, 선택 동의 여부, 알림 권한 여부, UsageStats 권한 여부
- FCM 상태: 권한 결과, 토큰 존재 여부, 등록 시도 여부

## 7. 공통 컴포넌트 의존성

## 9. 완료 기준

- Google 로그인은 성공/실패/취소 시나리오가 모두 동작한다.
- Kakao 로그인은 실제 API 연동까지 완료된다.
- Naver 버튼은 준비중 상태로만 동작한다.
- 선택 동의 값이 서버와 로컬 상태에 일관되게 반영된다.
- 알림 권한과 UsageStats 권한의 허용/거부 규칙이 최신 SRS와 일치한다.
