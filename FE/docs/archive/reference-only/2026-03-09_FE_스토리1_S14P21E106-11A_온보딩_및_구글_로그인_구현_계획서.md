## 1. 목표

최초 사용자 진입 흐름을 완성한다. 온보딩 3버튼 노출 -> Google 로그인 -> 약관 동의 -> FCM 권한 요청 -> UsageStats 필수 권한 요청까지의 시퀀스를 정리하고, JWT 기반 자동 로그인 분기를 확정한다.

### 포함 범위

- `SCR-001` 온보딩 화면
- `SCR-001-2` 약관 동의 및 권한 요청 흐름
- Google 로그인 연동 및 결과 처리
- 서버 로그인 API 호출 및 JWT 저장
- FCM 알림 권한 런타임 요청 진입점
- UsageStats 권한 요청 진입점
- 앱 재실행 시 자동 로그인 분기

### 제외 범위

- UsageStats 권한 상세 상태머신/수집기 -> 스토리 2
- `SCR-002` 분석 진행 / `SCR-002b` 구독 확인 -> 스토리 2, 3
- FCM 토큰 등록/수신/체크인 알림 -> 스토리 5

### 역할 분리

| 처리 | 담당 | 비고 |
| --- | --- | --- |
| 소셜 로그인 버튼 노출 | FE | Google/Kakao/Naver 3버튼 모두 노출 |
| 실제 인증 수행 | FE + Google | MVP 실제 인증은 Google만 연결 |
| 로그인 API 호출 | FE | `POST /api/v1/auth/login` |
| JWT 발급/검증 | BE | 최신 인증 API 기준 |
| 약관 동의 저장 | FE/BE | API 미확정 `[확인 필요: FE/BE]` |
| FCM 권한 요청 | FE | Android 런타임 권한 |
| FCM 토큰 등록 | BE | 실제 호출 책임은 스토리 5 |
| UsageStats 권한 요청 | FE | `ACTION_USAGE_ACCESS_SETTINGS` |

## 3. 현재 레포 기준 작업상황

- `feature/auth/ui/OnboardingScreen.kt`, `TermsScreen.kt`, `AuthViewModel.kt`가 이미 존재한다.
- 온보딩에는 Google/Kakao/Naver 3버튼 노출이 이미 반영돼 있다.
- 현재 레포는 Google 실제 연동 경로를 사용하고 있으나, `AuthApi.kt`/`AuthRepository.kt`에는 과거 카카오 관련 DTO/메서드 잔재가 함께 남아 있다.
- `AuthNavGraph`는 `AnalysisLoading`, `SubscriptionConfirm` 이후 화면이 placeholder 상태다.

즉, Story1은 완전 미착수가 아니라 화면/상태 골격은 일부 존재하고, 최신 API/문서 기준으로 용어와 계약을 정렬해야 하는 상태다.

## 4. 화면 / Route 정의

| 화면 ID | Route | 설명 |
| --- | --- | --- |
| `SCR-001` | `AuthRoute.Onboarding` | 온보딩 3버튼 노출, Google-only 실제 인증 |
| `SCR-001-2` | `AuthRoute.Terms` | 약관 동의, FCM 권한, UsageStats 권한 진입 |

### 화면 전환 시퀀스

```text
AppNavHost 진입
├─ 토큰 존재 + 유효 -> HomeRoute.Dashboard
├─ 토큰 존재 + 만료 -> refresh 시도
│  ├─ 성공 -> HomeRoute.Dashboard
│  └─ 실패 -> clearTokens() -> AuthRoute.Onboarding
└─ 토큰 없음 -> AuthRoute.Onboarding

Onboarding
├─ Google 버튼 탭 -> 실제 로그인 수행
│  ├─ 성공 -> POST /api/v1/auth/login -> 토큰 저장 -> Terms 이동
│  ├─ 취소 -> 토스트 후 유지
│  └─ 실패 -> 토스트 후 유지
├─ Kakao 버튼 탭 -> 준비중 토스트
└─ Naver 버튼 탭 -> 준비중 토스트

Terms
├─ 필수 약관 체크
├─ 시작하기 탭
│  ├─ 동의 저장 성공 -> FCM 권한 요청
│  │  ├─ 허용 -> 스토리 5 위임 포인트 기록
│  │  └─ 거부 -> 다음 단계 진행
│  └─ UsageStats 권한 요청
│     ├─ 허용 -> AuthRoute.AnalysisLoading
│     └─ 거부 -> 재유도
└─ 저장 실패 -> 토스트 + 버튼 재활성
```

## 5. UI 상태 정의

```kotlin
data class AuthUiState(
    val currentPage: AuthPage = AuthPage.Onboarding,
    val isGoogleLoading: Boolean = false,
    val consentItems: List<ConsentItem> = defaultConsents(),
    val isAllRequiredConsented: Boolean = false,
    val isTermsSaving: Boolean = false,
    val permissionStep: PermissionStep = PermissionStep.None,
    val showUsageStatsDialog: Boolean = false,
    val showUsageStatsRePrompt: Boolean = false,
)

enum class AuthPage { Onboarding, Terms }
enum class PermissionStep { None, Fcm, UsageStats }
```

### 필수 정책

- 온보딩은 Social 버튼 3개를 모두 노출한다.
- 실제 인증 처리와 provider 노출 수를 혼동하지 않는다.
- FCM 권한은 허용/거부 모두 UsageStats 단계로 진행한다.
- UsageStats는 필수 권한으로 취급한다.
- 화면 스타일은 White Base, Social 버튼 분리, 약관 행 공통 컴포넌트 재사용 원칙을 따른다.

## 6. ViewModel / Repository / API 범위

### ViewModel 책임

`AuthViewModel`

- Google 로그인 상태 및 토스트/네비게이션 effect 관리
- 약관 동의 상태 관리
- FCM 권한 요청/UsageStats 진입 단계 관리
- 자동 로그인 분기 및 세션 복원 실패 처리

### Repository 책임

- `AuthRepository`: 로그인 API, refresh API
- `TokenRepository`: access/refresh 저장 및 clear
- `UserRepository`: 로그인 응답 user 정보 후속 소비 지점에서 사용 가능

### API 목록

| API | 상태 | 비고 |
| --- | --- | --- |
| `POST /api/v1/auth/login` | 확정 | body key: `googleAccessToken` |
| `POST /api/v1/auth/refresh` | 확정 | refresh 시 `TokenPair` 성격 응답 |
| `POST /api/v1/auth/logout` | 확정 | 실제 호출은 Story4 |
| 약관 동의 저장 API | `[확인 필요: FE/BE]` | 계약 미정 |

### 로그인 응답 처리 규칙

- 로그인 응답은 access/refresh/tokenType/expiresIn/user 필드를 포함한 로그인 응답으로 다룬다.
- provider 결과값과 앱 세션 토큰을 혼용하지 않는다.
- 로그인 성공 후 수신한 user 정보는 Story4/Story5 초기값으로 참조 가능하지만, 영속화 전략은 `[확인 필요: FE/BE]`다.
