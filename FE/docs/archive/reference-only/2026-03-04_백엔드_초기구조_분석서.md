## 0) 이번 업데이트 요약 (git pull 반영)

- 백엔드 애플리케이션 코어(`BE/src/main/java`)는 여전히 `global` 중심 초기 부트스트랩 구조를 유지한다.
- API는 현재 3개(`health`, `dev/jwt`, `me`)이며, PRD MVP 비즈니스 API는 아직 미구현 상태다.
- 루트 실행 파일(`docker-compose.yml`) 외에 `INFRA/`에 dev/prod 분리 배포 자산이 추가되어 운영 구조 가시성이 높아졌다.

## 1) 문서 목적

- FE 관점에서 현재 백엔드 구현 범위와 협업 가능 지점을 빠르게 파악한다.
- 인증/응답 규격/실행 환경을 기준으로 FE 선개발(목업 + 실제 연결) 전략을 수립한다.
- PRD 대비 미구현 영역을 명확히 분리해 병렬 개발 시 커뮤니케이션 비용을 줄인다.

## 4) 패키지/레이어별 구현 상태

### 4.1 `global.common`

- `ApiResponse.java`: 성공 응답 공통 포맷
- `BaseEntity.java`: `createdAt`, `updatedAt` 감사 필드 베이스

### 4.2 `global.config`

- `JpaConfig.java`: `@EnableJpaAuditing`
- `JwtConfig.java`: HS256 `JwtEncoder`/`JwtDecoder` 빈 구성
- `RedisConfig.java`: `RedisTemplate<String, Object>` 직렬화 설정
- `SwaggerConfig.java`: OpenAPI 메타 + `bearerAuth` 스키마 선언

### 4.3 `global.controller`

- `HealthCheckController.java`: `GET /api/v1/health`
- `DevJwtController.java`: `POST /api/v1/dev/jwt` (local 프로필 전용)
- `MeController.java`: `GET /api/v1/me` (JWT 인증 확인)

### 4.4 `global.exception`

- `ErrorCode.java`, `ErrorResponse.java`, `BusinessException.java`
- `GlobalExceptionHandler.java`: 비즈니스/검증/기타 예외 응답 통합

### 4.5 `global.security`

- `SecurityConfig.java`: stateless + resource server + 401/403 JSON 핸들링
- `JwtProperties.java`: `jwt.*` 설정 바인딩

## 6) 현재 API 명세 (FE 즉시 연동 가능)

### 6.1 `GET /api/v1/health`

- 파일: `BE/src/main/java/com/ssafy/e106/global/controller/HealthCheckController.java`
- 인증: 없음
- 용도: 백엔드 상태 확인

### 6.2 `POST /api/v1/dev/jwt?sub=dev-user`

- 파일: `BE/src/main/java/com/ssafy/e106/global/controller/DevJwtController.java`
- 인증: 없음
- 제약: `@Profile("local")`에서만 활성
- 용도: FE 로컬 연동용 액세스 토큰 발급

### 6.3 `GET /api/v1/me`

- 파일: `BE/src/main/java/com/ssafy/e106/global/controller/MeController.java`
- 인증: 필요 (`Bearer <token>`)
- 용도: 토큰 서명/만료/클레임 검증 확인

---

## 7) 인증/응답/데이터 계층 포인트

### 7.1 인증 정책

- 파일: `BE/src/main/java/com/ssafy/e106/global/security/SecurityConfig.java`
- `/api/v1/me`만 인증 필요, 나머지 요청 `permitAll`
- 토큰 실패 시 `UNAUTHORIZED`, 권한 부족 시 `FORBIDDEN` JSON 응답

### 7.2 공통 응답 규격

- 성공: `ApiResponse<T>`
- 실패: `ErrorResponse { success, code, message }`

### 7.3 영속 계층 성숙도

- 준비됨: JPA Auditing, BaseEntity
- 미구현: `@Entity`, `Repository`, `Service`, 마이그레이션(Flyway/Liquibase)

## 8) 인프라/배포 구성 (업데이트 반영)

### 8.1 로컬 통합 실행 (루트)

- 파일: `docker-compose.yml`, `.env.example`
- 구성: `mysql` + `redis` + `backend`

### 8.2 dev/prod 분리 배포 자산 (`INFRA/`)

- dev:
  - `INFRA/docker-compose.dev.yml`
  - `INFRA/.env.dev.example`
  - `INFRA/deploy-dev.sh`
- prod:
  - `INFRA/docker-compose.prod.yml`
  - `INFRA/.env.prod.example`
  - `INFRA/deploy-prod.sh` (헬스체크 실패 시 자동 롤백 루틴 포함)

FE 협업 관점:
- API 스펙은 동일하지만 실행 포트/환경파일이 분리되어 QA/스테이징 커뮤니케이션이 쉬워짐

## 9) PRD 대비 갭 (FE 영향)

### 미구현 영역

- 구독 대시보드: 결제내역 연동/가맹점 매핑/구독 감지 API 부재
- 프로모션 추천: 프로모션/번들 모델 + 추천 API 부재
- 해지 검토: 체크인 수집/스케줄러/FCM/해지 가이드 API 부재

### FE 당장 가능한 작업

- 실제 연동: `health`, `dev/jwt`, `me`
- 병렬 개발: 프로토타입 화면 + 상태관리(ViewModel/State/Intent) 선구현
- API 계약 문서화: 예상 DTO/에러코드 표준 선합의

## 10) FE-BE 초기 협업 체크리스트 (권장)

- API base path: `/api/v1` 고정
- 성공/실패 envelope 규격 유지 (`ApiResponse`, `ErrorResponse`)
- 인증 헤더 표준화: `Authorization: Bearer <token>`
- 우선 구현 요청 API:
  1. `GET /api/v1/subscriptions/summary`
  2. `GET /api/v1/subscriptions`
  3. `GET /api/v1/promotions/recommendations`
  4. `POST /api/v1/checkins`
  5. `GET /api/v1/recommendations/cancel`

## 11) 결론

- 현재 백엔드는 "인증/공통 인프라 + 테스트용 엔드포인트" 단계다.
- git pull 이후 기준으로도 FE가 즉시 붙을 수 있는 실 API는 3개이며, 핵심 비즈니스 API는 이후 구현 대상이다.
- 다만 `INFRA/` dev/prod 배포 자산이 정리되어 운영 환경 구분이 명확해졌고, FE 입장에서도 환경별 테스트 전략을 세우기 쉬워졌다.
